# VPN-протоколы: карта и применимость

Этот документ — карта семейства VPN-протоколов с точки зрения **выбора под задачу
оператора** в реальности середины 2026 года. Он отвечает на три вопроса:

1. Какие протоколы существуют, как работают, какие у них fingerprint-сигнатуры.
2. Что блокируется в РФ-2026 и как именно.
3. Какой протокол под какую задачу — VPN для устройств, серверный прокси,
   multi-hop, маскировка.

Документ читают: персона (когда оператор задаёт свободный вопрос про VPN),
скиллы `/setup-vpn-panel`, `/configure-vpn-routing`, `/generate-client-config`
при принятии архитектурных решений.

## 1. Восемь протоколов — техническая карта

### 1.1 OpenVPN

| Параметр | Значение |
|---|---|
| Год создания | 2001 |
| Активная разработка | Да (OpenVPN Inc.) |
| Статус в РФ-2026 | Заблокирован TSPU (массовая блокировка с 2024) |

**Как детектируется DPI:**
- TLS handshake клиента имеет уникальный fingerprint (порядок cipher suites,
  расширения) — это даёт уникальные JA3/JA3S отпечатки.
- P-ACK пакеты одинакового размера на handshake-этапе — сильный сигнатурный признак.
- Академическая работа USENIX Security 2022 «OpenVPN is Open to VPN Fingerprinting»
  (Xue et al.): даже OpenVPN over TLS на порту 443 детектируется по внешнему handshake.

**Порты:** UDP 1194 (по умолчанию), часто переносят на TCP 443 — против DPI это
не помогает.

**Когда выбрать:** корпоративный сегмент с обфускацией (tls-crypt, stunnel),
сеть, где DPI отсутствует, совместимость со старыми клиентами критична.

**Когда не выбирать:** обход блокировок в РФ, где работает TSPU.

### 1.2 WireGuard

| Параметр | Значение |
|---|---|
| Год создания | 2016 |
| Активная разработка | Да (Jason A. Donenfeld) |
| Статус в РФ-2026 | Заблокирован TSPU (с середины 2024) |

**Как детектируется DPI:**
- Жёстко фиксированный 148-байтный handshake initial packet — очень сильная
  сигнатура.
- UDP-протокол без обфускации; TSPU блокирует «неопознанный UDP» с 2025 года.

**Когда выбрать:** только в доверенной сети, где нет DPI (корпоративный VPN
между офисами, домашняя сеть с роутером).

**Когда не выбирать:** обход блокировок РФ. После старта первые 1-3 минуты
соединение работает → TSPU фиксирует 148b handshake → молча дропает UDP-пакеты.

### 1.3 AmneziaWG (AWG)

| Параметр | Значение |
|---|---|
| Год создания | 2024 (Amnezia VPN) |
| Активная разработка | Да |
| Статус в РФ-2026 | AWG 2.0 — частично работает |

Форк WireGuard-Go от команды Amnezia. AWG 1.x заменял фиксированные заголовки
случайными статическими значениями; **AWG 2.0** (2025) использует «mimicry» —
постоянно меняющиеся заголовки и размеры пакетов.

По данным Xeovo Research, AWG 2.0 держится там, где stock WireGuard «мёртв
с середины 2025».

**Когда выбрать:** оператор хочет WireGuard-семейство (низкий overhead, высокая
скорость) с шансом обхода TSPU в РФ. Зрелые клиенты — Amnezia VPN, AmneziaWG
для Windows/Android/iOS.

**Когда не выбирать:** маскировка трафика под HTTPS как первоочередной критерий
(тогда — VLESS+Reality).

### 1.4 Shadowsocks

| Параметр | Значение |
|---|---|
| Год создания | 2012 (clowwindy) |
| Варианты | Original (RC4/AEAD), Shadowsocks-2022 (новый AEAD-протокол) |
| Статус в РФ-2026 | Заблокирован (включая обфускацию simple-obfs/v2ray-plugin) |

**Как детектируется:**
- «Полностью зашифрованный» трафик без TLS-обёртки сам по себе подозрителен —
  это сигнатура (см. net4people/bbs issue #363).
- ML-модели на pattern «короткие control-пакеты + большие data-пакеты»
  определяют SS с точностью 80-95%.

**Хронология блокировок:**
- До 2024 — обфускация обходила.
- Сентябрь 2024 — TSPU начал детектить обфусцированный SS в течение часов.
- 2025 — массовая блокировка.

**Когда выбрать:** в РФ-2026 — почти негде. Может быть полезен как outbound
изнутри уже работающего VPN (для второго уровня обфускации), но не как
первичный inbound.

### 1.5 VMess

| Параметр | Значение |
|---|---|
| Год создания | 2015 (V2Ray) |
| Статус | Legacy — официально вытеснен VLESS |

**Технические недостатки против VLESS:**
- Двойное шифрование (VMess + TLS) → лишняя нагрузка на CPU.
- Зависимость от системного времени (часы клиента и сервера должны быть синхронны).

**Статус в РФ:** заблокирован, детекция ~80% по данным комьюнити.

**Когда выбрать:** только если уже работающая инфраструктура на VMess и переход
не оправдан. Для новых установок — всегда VLESS.

### 1.6 Trojan

| Параметр | Значение |
|---|---|
| Год создания | 2019 |
| Принцип | Маскировка под HTTPS: верный пароль — проксирование, иначе fallback на реальный веб-сервер |

**Как детектируется:**
- Active probing: цензор подключается с заведомо некорректным TLS-handshake,
  анализирует ответ Trojan-сервера. Детекция ~90% по академической работе
  «TrojanProbe» (ScienceDirect, 2024).

**Статус в РФ:** ненадёжен. Август 2025 — массовое active probing TSPU.

**Когда выбрать:** не для нового продакшна в РФ. В странах без active probing —
рабочий вариант (Trojan-Go с расширениями).

### 1.7 VLESS + Reality — современный стандарт

**VLESS** — облегчённый протокол V2Ray/Xray:
- Без собственного шифрования (всё на TLS).
- UUID-based аутентификация без metadata-overhead.
- Поддерживает `flow` (например, `xtls-rprx-vision` для XTLS Vision splice-режима).

**Reality** — отдельный TLS-replacement от XTLS-команды:
- Форк пакета `crypto/tls` из Go 1.19.5 (см. официальный README XTLS/REALITY).
- При TLS handshake клиент получает Server Hello от **реального стороннего сайта**
  (сервер Reality играет роль «зеркала»). Реальный TLS-сертификат настоящего
  сайта попадает клиенту, но шифрование идёт по pre-shared ключу (X25519).
- Цензор при active probing получит реальный сертификат настоящего сайта →
  не сможет идентифицировать прокси-сервер.
- Использует TLS 1.3.

**Требования к `dest`/`serverNames`** (из официальной документации REALITY):
- Целевой сайт поддерживает TLS 1.3 и HTTP/2.
- Не является redirect-only (главный домен может редиректить на `www` — нормально).
- Бонусы: encrypted handshake messages после Server Hello (примеры: `dl.google.com`,
  `www.cloudflare.com`), OCSP Stapling, географическая близость IP к серверу
  прокси, целевой IP не используется массово другими Reality-серверами.
- Не подсанкционный для РФ (для российского сегмента — внешний домен).

**Когда выбрать:**
- Серверный inbound на заграничном VPS (для multi-hop через свой VPS).
- Любой ситуации, где нужна устойчивость к active probing и DPI.

**Когда не выбирать:**
- Серверный inbound на VPS в РФ (для внутреннего РФ-трафика на свой же сервер
  Reality избыточен — TSPU не работает на внутри-РФ маршрутах). VLESS-TCP без
  Reality достаточно.

**Уязвимости при неправильной настройке — см. §6 «Антипаттерны».**

### 1.8 Hysteria 2 и TUIC

**Hysteria 2** (QUIC over UDP):
- Шифрует через TLS с настоящими сертификатами.
- Режим обфускации **Salamander** — оборачивает QUIC-пакеты в случайные UDP-байты,
  убирает QUIC-fingerprint.
- BBR/Brutal congestion control — выше throughput на нестабильных каналах.

**TUIC** (TLS-over-UDP via QUIC) — близкая концепция, активный проект.

**Статус в РФ-2026:**
- РФ блокирует UDP/443 неопознанного трафика и HTTP/3 на нескольких ASN.
- Hysteria без Salamander заблокирован.
- Hysteria с Salamander — частично работает, но «может быть заблокирован вместе
  с другим неопознанным UDP-трафиком» (комьюнити-наблюдения, средний уровень
  доверия).

**Когда выбрать:** оператор знает, что его ISP не блокирует QUIC, и нужна
высокая пропускная способность на нестабильном канале.

**Когда не выбирать:** РФ-2026 как первичный протокол для обхода блокировок —
менее надёжно, чем VLESS+Reality.

## 3. Реальность блокировок РФ-2026

### 3.1 ТСПУ (Технические средства противодействия угрозам)

DPI-боксы, физически установленные на оборудовании каждого крупного провайдера РФ.
Управляются Роскомнадзором. Бюджет 2025-2027: **60 млрд рублей** на «суверенность
сети» (источник: Минфин).

**Возможности (документированные публично):**
- Fingerprint-based detection протоколов (OpenVPN, WireGuard, IKEv2, Shadowsocks).
- ML-классификация по паттернам packet timing / packet size distribution.
- Active probing подозрительных IP.
- TLS-level disruption (срыв соединения на TLS handshake).
- С июня 2025 — throttling foreign CDN: после ~15-20 KB полученных данных
  TCP-пакеты с сервера прекращаются (Hetzner, DigitalOcean, OVH, Cloudflare).

### 3.2 «Белый список» Минфина (с 5 сентября 2025)

Первая версия — 5 сентября 2025, расширенная — 14 ноября 2025. Изначальный состав:
57 сайтов «социально значимых сервисов» (Госуслуги, VK, OK, Mail.ru, Max,
Yandex-сервисы, Ozon/Wildberries/Avito, банки и телеком-операторы).

Применяется при отключении мобильного интернета («дни тишины»). В среднем
~2000 отключений в месяц с мая 2025 — больше, чем во всём остальном мире
за 2024.

### 3.3 Что точно блокируется

| Что | Кто блокирует | Источник |
|---|---|---|
| Anthropic, OpenAI, Gemini API | API-провайдеры (ToS, не DPI) | ToS Anthropic Sep 2025, OpenAI supported countries list |
| Hetzner ASN (AS24940) | TSPU throttling с июня 2025 (16-KB curtain) | Cloudflare блог + комьюнити |
| DigitalOcean (46.101.128.0/17) | TSPU throttling | Комьюнити |
| OVH, Cloudflare | TSPU throttling (16-KB curtain) | Cloudflare блог |
| OpenVPN, WireGuard (stock) | TSPU DPI | РосКомСвобода |
| Shadowsocks, Trojan | TSPU DPI + active probing | HRW отчёт 2025 |
| Tor (включая bridges meek/obfs4) | TSPU | РосКомСвобода |

### 3.4 Что работает (на 2026-05-15)

- **VLESS + Reality** — работает.
- **AmneziaWG 2.0** — работает.
- **Hysteria 2 с Salamander** — частично работает.
- **Российские VPS-провайдеры внутри РФ** — конечно работают. Но: ходить
  из РФ-VPS к Anthropic/OpenAI напрямую блокируется на стороне API
  (геолокация). Решение — multi-hop (см. §4).

## 4. Multi-hop: два равноправных пути

Сценарий, в котором оператор сидит в РФ и хочет ходить в зарубежные API
(Anthropic, OpenAI) или просто в свободный интернет. Прямой путь
«клиент в РФ → API за границей» не работает: API блокирует РФ-IP, либо
ISP блокирует протокол.

Решение — **multi-hop**: трафик проходит через два сервера, один из которых
за границей.

### 4.1 Путь A: outbound через подписку платного VPN-провайдера

```
Клиент → панель 3X-UI на VPS в РФ → outbound через провайдера → интернет
                                       (vless:// из подписки)
```

**Как устроено:**
- Оператор арендует VPS в РФ (минимум — 1 vCPU, 2 ГБ RAM, 15-30 ГБ диск).
- На VPS устанавливается панель 3X-UI с VLESS-TCP inbound (без Reality —
  трафик клиент→панель идёт **внутри РФ**, маскировка не нужна).
- Оператор берёт подписную ссылку у платного VPN-провайдера. Подписная ссылка —
  это HTTP(S)-endpoint, который возвращает base64-список `vless://`/`vmess://`
  ссылок (несколько серверов провайдера).
- Каждая `vless://`-ссылка превращается в **outbound** в панели 3X-UI.
- Если ссылок несколько — собираются в **балансировщик** (стратегия
  `leastPing` или `roundRobin`).
- Маршрутизация: `geoip:ru` + `geosite:category-ru` → direct (прямо в интернет
  из РФ), остальное → outbound через провайдера → за границу.

**Плюсы пути A:**
- Один сервер на стороне оператора (только VPS в РФ).
- Провайдер сам адаптирует свои сервера под новые блокировки РКН — оператор
  только обновляет подписку.
- Стоимость 2026: VPS в РФ ~200-400₽/мес + подписка провайдера ~250-700₽/мес.

**Минусы пути A:**
- Зависимость от провайдера: если провайдер ушёл с рынка или сменил протокол —
  оператор переключается на другого, переподключает подписку.
- Метаданные трафика проходят через провайдера. Доверие нужно.
- Невозможно настроить специфические rule-set на стороне зарубежного хопа.

### 4.2 Путь B: outbound через свой заграничный VPS

```
Клиент → панель 3X-UI на VPS в РФ → outbound через свой VLESS+Reality → интернет
                                       (свой VPS за границей)
```

**Как устроено:**
- Оператор арендует **второй** VPS за границей (Hetzner / Aeza / OVH /
  любой провайдер вне РФ).
- На заграничном VPS устанавливается панель 3X-UI с **VLESS+Reality** inbound
  на 443/TCP (маскировка под легитимный HTTPS — обязательно, потому что
  трафик из РФ к загр.VPS проходит TSPU).
- Оператор берёт `vless://`-ссылку с заграничного VPS и подключает её как
  outbound в основной панели в РФ.
- Маршрутизация — та же, что в пути A.

**Плюсы пути B:**
- Полный контроль над выходным узлом: свой rule-set, свои логи, никакой
  third-party.
- Возможность гибких архитектур (например, разные outbound-ы под разные
  типы трафика, специфические rule-set для нейросетей).
- Независимость от провайдеров.

**Минусы пути B:**
- Два VPS вместо одного (~+800₽-1500₽/мес за загр.VPS).
- Операционная нагрузка: следить за новыми DPI-волнами, ротировать Reality-ключи
  при компрометации, обновлять `serverName` если донор перестал быть подходящим.
- При неправильной настройке Reality (плохой `serverName`, общий домен с панелью —
  см. §6) — палится.

### 4.3 Архитектурный выбор: какой путь когда

**Путь A (подписка) рекомендуется по умолчанию когда:**
- Оператор хочет «настроить и забыть».
- Нет требования к специфическим rule-set на стороне зарубежного узла.
- Бюджет ограничен.
- Есть доверие к выбранному платному провайдеру.

**Путь B (свой загр.VPS) рекомендуется когда:**
- Нужна полная независимость от провайдеров.
- Нужны специфические rule-set на стороне зарубежного узла (например,
  outbound в зависимости от типа трафика).
- Оператор готов к операционной нагрузке.
- Бюджет позволяет два VPS.

**Гибрид (редко):**
- Один загр.VPS как основной outbound + подписка как fallback в балансировщике
  (`fallbackTag` в Xray routing). Это страховка: если свой VPS заблокировался —
  трафик автоматически уходит через подписку.

### 4.4 Конфигурация маршрутизации (типовое правило)

```json
{
  "type": "field",
  "outboundTag": "direct",
  "domain": ["geosite:category-ru", "regexp:.*\\.ru$"],
  "ip": ["geoip:ru"]
}
```

`geoip:ru` — встроенный список IP-диапазонов РФ из `geoip.dat` (поставляется
с Xray-core). `geosite:category-ru` — список российских доменов из `geosite.dat`.
Оба файла лежат рядом с бинарником Xray.

Источники-генераторы: `v2fly/domain-list-community`, `Loyalsoldier/v2ray-rules-dat`,
`chocolate4u/Iran-v2ray-rules`.

### 4.5 Балансировщики (когда несколько outbound)

Если из подписки приходит несколько серверов, или у оператора есть несколько
своих загр.VPS — собираем в `balancers[]` в routing-секции Xray:

- `tag` — имя для ссылки из rules.
- `selector` — массив prefix-match для тегов outbound (например, `["proxy"]`
  отбирает `proxy1`, `proxy2`).
- `fallbackTag` — резервный outbound при недоступности всех в селекторе.
- `strategy.type`:
  - `"random"` — равномерное случайное распределение.
  - `"roundRobin"` — по кругу.
  - `"leastPing"` — выбор с минимальной задержкой (требует `observatory`).
  - `"leastLoad"` — выбор с минимальной нагрузкой (требует `observatory`).

`observatory` — модуль активного мониторинга outbound-ов. Пингует endpoint
(обычно `http://www.google.com/gen_204`) с заданным интервалом, исключает
unhealthy.

## 5. Серверный прокси: отдельный сценарий

Программам на VPS оператора (боты на Python, скрипты на Node, npm install,
pip install, curl, git push) тоже нужен путь в заблокированные API. Они не
могут пользоваться 3X-UI как Hiddify — у них нет UI для импорта подписки.
Решение — **mixed inbound на 127.0.0.1**.

**Принцип:**
- На той же панели 3X-UI создаётся ещё один inbound — `mixed` (SOCKS5 + HTTP
  на одном порту), `listen=127.0.0.1`, `port=1080`, без аутентификации
  (доступ только локальный).
- В `/etc/environment` прописывается:
  ```
  http_proxy=socks5h://127.0.0.1:1080
  https_proxy=socks5h://127.0.0.1:1080
  no_proxy=localhost,127.0.0.1
  ```
- **Буква `h` в `socks5h` критична**: DNS-резолвинг идёт на прокси, не на
  клиенте. Без `h` — DNS-leak в локальный resolver и Anthropic API возвращает
  403 (поведение характерно конкретно для anthropic SDK).

**Подводный камень — self-loop:**
- Go-процессы (включая саму панель 3X-UI и Xray-ядро) уважают `HTTP_PROXY`
  через `httpproxy.FromEnvironment` стандартной библиотеки.
- Если переменные в `/etc/environment` глобальные — x-ui тоже начнёт ходить
  через 127.0.0.1:1080 → через свой же outbound → обратно к себе → петля →
  падение с `INVALIDARGUMENT`.
- **Решение:** systemd drop-in override для `x-ui.service` (и опционально для
  `xray.service`):
  ```
  systemctl edit x-ui
  ```
  Содержание override:
  ```
  [Service]
  Environment="HTTP_PROXY="
  Environment="HTTPS_PROXY="
  Environment="NO_PROXY=*"
  ```
- Это очищает прокси-переменные только для конкретных юнитов, оставляя
  глобальные `/etc/environment` для пользовательских сессий.

**Подробности — `3x-ui-panel.md` (известные подводные камни) и сам скилл
`/setup-server-proxy`.**

## 6. Антипаттерны

### 6.1 Голый WireGuard / OpenVPN на зарубежный VPS из РФ

Заблокирован TSPU. Первые 1-3 минуты соединение работает, потом ISP-устройство
фиксирует сигнатуру протокола и молча дропает все последующие пакеты.
Перезапуск даёт ту же картину.

**Альтернативы:** AmneziaWG 2.0, VLESS+Reality.

### 6.2 Бесплатные VPN-сервисы

Независимые исследования (CSIRO / ICSI / UC Berkeley / UNSW; Ikram et al.):
- 67% бесплатных Android-VPN содержат сторонние tracking-библиотеки.
- 38% содержат malware/malvertising.
- 88% утекают user data.
- 71% передают personal user data третьим сторонам.

Бесплатный VPN — это «вы платите трафиком и метаданными». **Никогда не
подключать через бесплатный VPN к API с токенами** (Anthropic, OpenAI,
GitHub) — это compromise токенов.

### 6.3 Reality с `serverName` того же домена, что и панель

**Уязвимость:**
1. Цензор замечает TLS-handshake к `proxy.example.com`.
2. Сам подключается к `proxy.example.com:443` с тем же SNI.
3. Получает сертификат панели вместо «настоящего» внешнего сайта.
4. Несовпадение → пометка как circumvention traffic.

**Правильно:** `serverNames` в Reality указывают на **независимый внешний
домен** (`www.cloudflare.com`, `dl.google.com`, `www.microsoft.com` —
частые проверенные примеры), не имеющий отношения к серверу проксирования.

### 6.4 Хардкод порта 443 без обоснования

**Проблема:** все массовые VPN-конфиги используют 443. Цензор смотрит на 443
пристально. TSPU делает targeted DPI на UDP/443 и пристальный DPI на TCP/443.

**Альтернативы:**
- Reality можно поставить на любой TCP-порт (8443, 10443). Главное — порт
  открыт у клиента. Мобильные ISP часто разрешают только 80/443.
- Для multi-user `home`-сценария 443 удобнее (универсальная доступность
  с любых сетей, корпоративные firewalls).

**Решение мастера:** баланс между «защита от DPI» (нестандартный порт)
и «универсальная доступность» (443). Это архитектурное решение, а не догма.

### 6.5 Reality на сервере в РФ (внутренний трафик)

Серверный VLESS+Reality оправдан только когда трафик к серверу пересекает
DPI-границу (клиент в РФ → сервер за границей). Внутри РФ TSPU не работает
на маршрутах между клиентом и сервером в одном регионе. Reality на
внутреннем сервере — лишняя криптография и лишняя сложность настройки
без выгоды.

**Правильно:** на VPS в РФ — **VLESS-TCP без Reality** (TLS опционально,
для маскировки панели — но это про панель, не про сам VLESS).

### 6.6 Закрытые «только-приложение» VPN-сервисы

NordVPN, ExpressVPN, Surfshark и подобные. Они не дают `vless://`-конфиг,
работают только через своё приложение. **Не годятся для outbound** в свою
панель — нечего разбирать.

**Альтернативы для outbound:** провайдеры, отдающие подписку (Blanc, Bebra,
v2RaY-провайдеры из China-сегмента, профильные VPN с публичным конфигом).

### 6.7 Сервер для VPN на компрометированном ASN

Hetzner (AS24940), DigitalOcean (46.101.128.0/17), Cloudflare, OVH — все
throttled из РФ с июня 2025 («16-KB curtain»: после ~16 KB полученных
данных TCP-пакеты с сервера перестают приходить).

**Если планируется загр.VPS как outbound для multi-hop:** проверять ASN
провайдера. Менее известные провайдеры (Aeza, ProstoVPS, мелкие VPS-хостинги
в Финляндии/Эстонии/Израиле) пока не на радаре throttling. Точный список
актуальных ASN — комьюнити-форумы (net4people/bbs, NTC.party), не статика.

## 7. Decision tree: какой протокол под какую задачу

```
Задача: серверный inbound на VPS оператора в РФ
  → VLESS-TCP без Reality (трафик внутри РФ, маскировка не нужна)
  → TLS опционально для самой панели (acme через 3X-UI или certbot)

Задача: серверный inbound на VPS оператора за границей (для multi-hop)
  → VLESS + Reality, flow = xtls-rprx-vision
  → serverName = dl.google.com / www.cloudflare.com / www.microsoft.com
  → порт 443/TCP (или нестандартный 8443/10443 при компрометации 443)

Задача: outbound в multi-hop с подписки провайдера
  → разобрать vless:// (или весь subscription URL)
  → создать outbound в панели с теми же параметрами
  → если несколько серверов в подписке — собрать в balancer (leastPing)

Задача: outbound в multi-hop через свой загр.VPS
  → подключить vless://-ссылку со своего загр.VPS как outbound
  → routing: geoip:ru → direct, остальное → этот outbound
  → опционально: fallback на подписку через balancer

Задача: серверный прокси для программ на VPS
  → mixed inbound на 127.0.0.1:1080 (на той же панели 3X-UI)
  → routing: geoip:ru → direct, остальное → upstream outbound
  → /etc/environment с socks5h:// (буква h обязательна)
  → systemd override для x-ui (защита от self-loop)

Задача: гибкая маршрутизация на клиенте
  → sing-box JSON с rule_set (geosite + geoip)
  → процесс-based маршрутизация (process_name на desktop, package_name
    на Android, недоступно на iOS)
  → TUN-mode на desktop (auto_redirect=true на Linux, strict_route=true
    на Windows)

Задача: клиентское приложение
  → iOS — Hiddify Proxy & VPN или Karing (sing-box внутри)
  → Android — Hiddify, NekoBox, SFA, или v2rayNG (xray)
  → desktop — Hiddify (универсал) или sing-box CLI как systemd-сервис
```
