## §3 Эталонная архитектура курса

См. визуальную схему в `_diagrams/vpn-architecture-student.md`.

Кратко:

```
[Устройство ученика, Hiddify, TUN-режим включён]
        │
        ├─ 🟢 РФ-сайт (sber.ru, vk.com, *.ru) → напрямую через домашний интернет
        ├─ 🚫 Реклама/трекеры → block (geosite:category-ads-all)
        └─ 🔵 Foreign → твой РФ-VPS (3X-UI, VLESS-TCP inbound)
                                          ↓
                                  outbound: подписка VPN-провайдера
                                          ↓
                                  свободный интернет
```

**Главные принципы:**

1. **Маршрутизация на клиенте (Hiddify), не на сервере.** Это даёт минимум hops для РФ-сайтов, минимум latency, минимум приватности утечек на свой VPS.
2. **TUN-режим обязателен.** Без него VSCode/Telegram desktop/Spotify идут мимо VPN.
3. **РФ-сервер — просто канал.** На нём ничего сложного, всё уже сделал клиент.
4. **Подписка VPN-провайдера = outbound на РФ-сервере.** Один URL, провайдер сам адаптируется к блокировкам.

## §4 Протокол подключения клиент → РФ-сервер

**Default: VLESS-TCP без Reality.**

Аргументы:
- Внутри РФ маршрутов ТСПУ не палит — маскировка избыточна сейчас
- Простота для ученика-новичка (одна кнопка в 3X-UI)
- Поддерживается всеми клиентами (Hiddify, FoxRay, v2rayN)
- Минимум overhead, максимум скорость

**Когда переключаться на Reality:** когда в `_live/timeline.md` появится запись
«РКН начал мониторинг исходящего VPS-трафика». Прогноз — Q3-Q4 2026, тогда обновим
эту памятку.

**Альтернативы НЕ default (для специальных кейсов):**
- VLESS+TLS+WebSocket+CDN — если VPS-IP заблокировал какой-то конкретный провайдер
- VMess — legacy, не использовать
- Trojan — если нужен порт 443 с реальным TLS-сертификатом
- AmneziaWG — если ученик хочет нативную iOS WG-интеграцию (но не работает с гибкой маршрутизацией Hiddify так же хорошо)

## §5 TUN-режим: что это, зачем, как включить

### 5.1 Простая аналогия

**Представь, что у тебя в офисе есть охранник на входной двери.** Он проверяет каждого
входящего и решает «этот идёт в кабинет А, этот в кабинет Б, этот не пройдёт».

- **Без охранника (без TUN):** все ходят как хотят. Кто-то через главный вход, кто-то
  через чёрный ход, кто-то через окно. Браузер слушается «иди через главный вход»
  (системный прокси), а VSCode не слушается и лезет в окно (напрямую в интернет).
- **С охранником (с TUN):** все входы и окна закрыты. Единственный путь наружу — через
  охранника. Браузер, VSCode, Telegram, любая программа — идут только через него. А он
  уже решает куда дальше.

**TUN — это и есть охранник на уровне операционной системы.**

### 5.2 Технически

TUN (от слова tunnel) — виртуальный сетевой адаптер. Когда Hiddify создаёт TUN-интерфейс
с правами уровня системы, **операционная система направляет туда ВЕСЬ исходящий трафик**.
Программа просто пишет в свой сокет — а ядро ОС перехватывает и кладёт в TUN. Hiddify
забирает оттуда пакеты, прогоняет через свою маршрутизацию, и отправляет дальше.

Программа об этом ничего не знает. Никаких настроек в VSCode, Telegram, curl, git, npm,
pip — ничего трогать не надо.

### 5.3 В чём отличие от SOCKS5-режима (старого)

| Параметр | SOCKS5-режим (старый) | TUN-режим (новый стандарт) |
|---|---|---|
| Как работает | Hiddify слушает на 127.0.0.1:1080, программы должны сами знать про прокси | Hiddify создаёт виртуальный адаптер, ОС перенаправляет ВСЁ |
| Браузер | ✅ работает (знает про системный прокси) | ✅ работает |
| VSCode, Cursor | ⚠️ только если вручную настроишь HTTPS_PROXY | ✅ работает |
| Telegram desktop | ❌ не работает | ✅ работает |
| curl, git, npm, pip | ⚠️ только с HTTPS_PROXY env | ✅ работает |
| Spotify, Discord | ❌ не работает | ✅ работает |
| Системный ping/dns | ❌ не работает | ✅ работает |

**Default в курсе — только TUN.** SOCKS5-режим оставлен только для специфичных
кейсов на сервере (для бота).

### 5.4 Включение по платформам

**iPhone (Hiddify):**
1. Открыть Hiddify
2. Settings → Mode → выбрать **Tun Mode** (вместо System Proxy)
3. При первом включении iOS попросит разрешение на VPN — Allow
4. Готово. Тумблер ON и весь трафик через Hiddify.

**macOS (Hiddify Desktop):**
1. Открыть Hiddify
2. Settings → Tun Mode → Enable
3. Установит system extension → попросит разрешения в System Settings → Privacy & Security → Allow
4. Может попросить перезагрузку (один раз)
5. Готово.

**Windows (Hiddify Desktop):**
1. Установить Hiddify Desktop
2. При первом запуске установит WinTun driver автоматически
3. Settings → Tun Mode → Enable
4. Запустить от имени администратора (иначе TUN не поднимется)
5. Готово.

**Android (Hiddify):**
1. Открыть Hiddify
2. Settings → Mode → Tun Mode
3. Android попросит разрешение «Allow VPN connection» — Yes
4. Готово.

### 5.5 Проверка что TUN работает

После включения — в браузере / терминале / любой программе:

```bash
# До включения Hiddify:
curl ifconfig.me
# Возвращает твой реальный российский IP

# Включить Hiddify в TUN-режиме, повторить:
curl ifconfig.me
# Возвращает IP твоего РФ-сервера (или загр.провайдера, если правило прошло)
```

Если IP меняется → TUN работает, ВСЕ программы будут идти через VPN.

### 5.6 Типичные проблемы TUN

| Проблема | Причина | Решение |
|---|---|---|
| Tun не включается на Windows | Запустил не от админа | Правый клик → Run as administrator |
| Tun не включается на macOS | Не дал разрешение на system extension | System Settings → Privacy & Security → Allow + перезагрузка |
| Tun включён, но интернет не работает | DNS не настроен в Hiddify | Settings → DNS → 1.1.1.1 (DoH) + Yandex DNS для .ru |
| Tun включён, но РФ-сайты медленные | Не настроены direct-правила | См. §6 правила маршрутизации |
| iPhone: «не могу одновременно включить рабочий WG и Hiddify» | iOS позволяет один VPN | Использовать multi-outbound внутри одного Hiddify (см. §8) |

## §6 Правила маршрутизации — 5 уровней защиты

Цель: **полная изоляция РФ-трафика на direct** без false-positive (foreign-сайт случайно ушёл напрямую) и false-negative (РФ-сайт ушёл через VPN и заблокировался).

В Hiddify правила задаются через **Routing Rules** (Settings → Routing). Можно
импортировать готовый JSON в **Advanced Settings → Custom Configuration**.

### 6.1 Уровень 1: geoip:ru + geoip:private

```yaml
- ip: geoip:ru → direct
- ip: geoip:private → direct  (КРИТИЧНО: локальная сеть 192.168.x, 10.x)
```

**Что ловит:** все IP-адреса в российских подсетях, плюс локальная сеть (роутер, NAS).

**Что не ловит:** российские сайты на иностранном CDN (Cloudflare, Akamai), их IP не российский.

### 6.2 Уровень 2: geosite:category-ru

```yaml
- domain: geosite:category-ru → direct
```

**Что ловит:** список российских доменов из community-репозитория
[v2fly/domain-list-community](https://github.com/v2fly/domain-list-community/blob/master/data/category-ru).

**Что не ловит:** свежие домены, добавленные после последнего обновления geosite.dat.

### 6.3 Уровень 3: regex по TLD

```yaml
- domain: regexp:.+\.ru$ → direct
- domain: regexp:.+\.su$ → direct
- domain: regexp:.+\.рф$ → direct
- domain: regexp:.+\.xn--p1ai$ → direct  # .рф в punycode
```

**Что ловит:** свежие домены в национальных зонах, даже не попавшие в geosite.

**Что не ловит:** российские сервисы на .com / .io / .app.

### 6.4 Уровень 4: явный список российских доменов на не-российских TLD

### 6.5 Уровень 5: блок рекламы и трекеров

```yaml
- domain: geosite:category-ads-all → block
```

**Что ловит:** глобальный список рекламы и трекеров (Google Ads, DoubleClick,
Facebook Pixel, российские рекламные сети).

**Бонус:** ускоряет загрузку сайтов и экономит трафик.

### 6.6 Default правило

```yaml
- (всё что не подошло выше) → proxy (через РФ-сервер)
```

### 6.7 Порядок правил критичен

**В sing-box правила выполняются сверху вниз. Первое совпавшее применяется.** Поэтому
порядок:

1. `geoip:private → direct` (локальная сеть, безусловно)
2. `geosite:category-ads-all → block` (реклама — раньше всего)
3. `geoip:ru → direct`
4. `geosite:category-ru → direct`
5. Regex по .ru/.su/.рф → direct
6. Явный список российских .com/.io доменов → direct
7. Default → proxy

### 6.8 DNS-маршрутизация (защита от утечек)

В Hiddify (Settings → DNS):

```yaml
# Российские домены резолвятся через Yandex DNS (быстро, не утекает в Google)
- domain: regexp:.+\.(ru|su|рф)$ → 77.88.8.8
- domain: geosite:category-ru → 77.88.8.8

# Всё остальное через Cloudflare DoH (зашифровано, провайдер не видит DNS)
- (default) → https://1.1.1.1/dns-query
```

**Зачем DoH:** обычный DNS-запрос идёт открытым текстом, провайдер видит «человек
резолвит chatgpt.com». DoH (DNS over HTTPS) — шифрует DNS-запрос, провайдер видит
просто «человек ходит на 1.1.1.1».

## §7 Серверная сторона — что настраивать на РФ-VPS

**Минимум.** Сервер должен только:

1. **Принимать** клиентов на VLESS-TCP inbound (uuid каждого ученика)
2. **Отправлять** весь их трафик в один outbound — подписку VPN-провайдера

**В 3X-UI:**

### 7.1 Inbound (для подключения клиентов)

```
- Protocol: VLESS
- Network: TCP (без Reality, без TLS)
- Port: любой (например 443 для маскировки)
- Listen IP: 0.0.0.0
- Clients: создать по одному для каждого устройства/ученика
- Sniffing: enabled (для маршрутизации по доменам, если будет)
```

### 7.2 Outbound (подписка провайдера)

В 3X-UI: **Xray Settings → Outbounds → Add Subscription**:

```
- URL подписки: https://провайдер.com/sub/токен
- Update interval: 1440 (раз в сутки автоматически)
- Strategy: leastPing (выбирает самый быстрый сервер автоматически)
```

3X-UI сам подтянет все серверы провайдера и будет ротировать по latency.

### 7.3 Routing rules на сервере

```yaml
- inbound: vless-tcp → outbound: subscription-balancer
- default → block  (если что-то не подошло — не отправляем)
```

**Это всё.** Никакой сложной маршрутизации на сервере — она вся на клиенте.

### 7.4 Бот ученика на сервере (уже настроено в уроке 5.09)

Бот ходит через SOCKS5 на 127.0.0.1:1080 → попадает в тот же routing → outbound через
подписку. Никаких изменений.

## §8 Подписочная ссылка для клиента

Чтобы Hiddify подключился к РФ-серверу и получил все правила маршрутизации — нужна
**одна подписочная ссылка**.

### 8.1 Генерация в 3X-UI

В 3X-UI: **Inbounds → выбрать клиента → Subscribe → Copy Link**

Получится URL вида:
```
https://твой-сервер.ru:2096/sub/abc123def456
```

### 8.2 Что должна содержать подписка

3X-UI по умолчанию даёт **base64-список vless:// ссылок** (только серверы). Для **гибкой
маршрутизации** нужно вместо этого отдать **sing-box JSON** с правилами.

**Способ 1: 3X-UI subscription с настроенным generator.**

В 3X-UI: **Panel Settings → Subscription → Custom JSON template**. Указать шаблон
sing-box JSON с правилами из §6. Тогда подписка отдаст готовый конфиг.

**Способ 2: скилл `/generate-client-config`.**

Агент сгенерирует JSON под конкретного клиента, выложит файл на сервер, отдаст ссылку:
```
https://твой-сервер.ru:2096/configs/student-name.json
```

Hiddify импортирует — получит и сервер, и правила, и DNS, и TUN-настройки сразу.

### 8.3 Импорт в Hiddify

**На iPhone/iPad:**
1. Открыть ссылку подписки в Safari — Hiddify подхватит
2. Или: открыть Hiddify → Add Profile → From URL → вставить ссылку

**На macOS/Windows/Android:**
1. То же — открыть Hiddify → Add Profile → From URL
2. Или: скопировать ссылку в буфер обмена → Hiddify сам предложит импорт

### 8.4 Обновление подписки

Hiddify автоматически обновляет подписку раз в сутки. Если на сервере изменились
правила — клиент подтянет автоматически. Никаких ручных переимпортов.

## §9 Опциональное расширение А: отдельный outbound для AI

См. схему в `_diagrams/vpn-architecture-student.md` (вторая mermaid-схема).

**Когда:** если у ученика несколько AI-аккаунтов и боится «слипания по shared IP», или если AI начали его банить (хотя обычно причина не в IP).

**Что добавить:**

1. **Второй outbound на сервере** — свой ЛИЧНЫЙ загр.VPS (надо сначала арендовать и настроить, см. расширение Б).
2. **Правило в RouterConfig сервера:**
   ```yaml
   - domain:
       - geosite:openai
       - domain:anthropic.com
       - domain:claude.ai
       - domain:perplexity.ai
     outbound: my-personal-foreign-vps

   - default → outbound: subscription-balancer
   ```
3. **На клиенте ничего не меняется.** Hiddify по-прежнему шлёт всё foreign на РФ-сервер, сервер сам решает где AI а где остальное.

**Стоимость:** +800-1500₽/мес за свой загр.VPS.

## §10 Опциональное расширение Б: вообще без подписки провайдера

Если ученик хочет **полную независимость** от платных VPN-провайдеров и согласен
сам админить (с агентом-сисадмином).

См. третью схему в `_diagrams/vpn-architecture-student.md`.

**Что меняется:**

1. **Вместо подписки** в outbound 3X-UI на РФ-сервере прописывается **свой загр.VPS**.
2. **Свой загр.VPS** настраивается так же как РФ-сервер: 3X-UI + VLESS+Reality (с donor=github.com или microsoft.com — Reality здесь ОБЯЗАТЕЛЕН, потому что трансграничный трафик).
3. **Платишь только за два VPS** (РФ + заграница) — суммарно ~1100-1800₽/мес.

### 10.1 Важно объяснить ученику

> 🚨 **Самостоятельный админ:** Свой загр.VPS значит ты сам отвечаешь за:
>
> - Обновления панели 3X-UI на обоих серверах (ежемесячно)
> - Мониторинг «грязности» IP загр.VPS (если попал в blocklist какого-то сервиса — менять)
> - Ротация ключей Reality раз в несколько месяцев (профилактика)
> - Реагирование на изменения блокировок РКН (когда фронт меняется)
> - Бэкапы конфигов панелей
>
> VPN-провайдер всё это делает **за тебя** автоматически. Свой сервер = больше
> контроля, но больше работы. Агент-сисадмин помогает с каждой операцией, но решения
> принимаешь ты.

### 10.2 Когда выбирать

- Готов разбираться и брать ответственность
- Хочешь полную приватность от провайдеров
- Долгосрочно (краткосрочно дороже — провайдер 500-700₽ vs второй VPS 800-1500₽)

### 10.3 Когда НЕ выбирать

- Новичок — лучше начать с подписки провайдера
- Не хочешь думать про администрирование
- Хочешь чтобы поломки решал саппорт

### 10.4 Где брать загр.VPS

**Хорошие провайдеры (не в РФ-санкциях, относительно «чистые» ASN):**
- Aeza (зарубежные локации)
- Timeweb Cloud (EU)
- vdsina (EU)
- DigitalOcean
- VPSDime

**Избегать:**
- Hetzner — IP часто в blocklist западных сервисов (но дёшево, можно если IP чистый)
- Contabo — подтверждённо грязный IP-пул для Anthropic/CF (issue #36201)

## §11 FAQ типичных проблем

### 11.1 "У меня всё через VPN, но Госуслуги/Сбер не открываются"

**Причина:** платформа блокирует тебя как VPN-пользователя (с 15.04.2026 это норма).

**Решение:** настроить direct-правила по §6. После настройки `geosite:category-ru → direct` Госуслуги/Сбер пойдут напрямую через домашний интернет, увидят твой реальный РФ-IP, не заблокируют.

### 11.2 "VPN включён, но VSCode не идёт через него"

**Причина:** работает в SOCKS5-режиме, VSCode не знает про прокси.

**Решение:** переключить Hiddify в TUN-режим. См. §5.4 по платформам.

### 11.3 "На iPhone не могу одновременно включить рабочий WireGuard и Hiddify"

**Причина:** iOS позволяет только один активный VPN-профиль.

**Решение:** внутри Hiddify настроить два outbound — один для основной маршрутизации, второй для рабочего VPN. Переключаться через rule по домену.

### 11.4 "Claude/ChatGPT всё равно банит мой аккаунт"

**Причина:** баны AI обычно НЕ по IP, а по поведению (смена платёжного метода, Apple Pay, частые перелогины, TLS-fingerprint скриптов).

**Решение:**
- НЕ менять способ оплаты чаще раза в месяц
- НЕ перелогиниваться с разных устройств в течение часа
- НЕ использовать Apple Pay для подписки (известный триггер для Claude)
- Скрипты на Node.js — это известный fingerprint, использовать cloudscraper или claude-code напрямую
- Если думаешь что причина в IP — см. §11.5

### 11.5 "Я уверен что причина в IP — мой VPS грязный"

**Проверка:**
```bash
# На VPS, не через VPN:
curl -I https://claude.ai 2>&1 | head -5
curl -I https://chatgpt.com 2>&1 | head -5

# Если ответ 200 OK / 405 Method Not Allowed → IP чистый
# Если ответ 403 + "cf-mitigated: challenge" → IP грязный
```

**Если грязный:**
- Сменить VPS-провайдера (избегать Hetzner/Contabo)
- Или: использовать опциональное расширение А (отдельный outbound для AI на чистом VPS)

### 11.6 "Я в роуминге за границей, VPN не нужен — но Госуслуги меня видят как зарубежного и блокируют"

**Причина:** ты физически за границей, твой IP — иностранный, Госуслуги считают тебя «не россиянином».

**Решение:** включить Hiddify, но **инвертировать** default-правило:
- РФ-сайты → через свой РФ-VPS (видят российский IP, не блокируют)
- Foreign → direct (ты уже за границей, VPN не нужен)

Это второй профиль Hiddify, можно держать рядом с основным.

### 11.7 "Skype/Zoom/Discord звонки рвутся через VPN"

**Причина:** real-time UDP-трафик плохо переносит лишние hops и нестабильность TCP-туннеля.

**Решение:** добавить эти домены в direct-правила:
```
- domain:discord.com → direct
- domain:zoom.us → direct
- domain:skype.com → direct
```

Это работает если ты не пытаешься через VPN звонить из РФ в заблокированные сервисы. Если пытаешься — нужен Hysteria2 outbound (см. `transports.md` §8).

### 11.8 "Hiddify ест батарею iPhone"

**Причина:** TUN-режим держит постоянное соединение.

**Решение:** это нормально, оверхед ~5-10% батареи в день. Если критично — можно
переключаться в System Proxy режим когда не нужен полный охват.

## §12 Чек-лист для агента при диагностике

Когда ученик жалуется «не работает» — пройти по чек-листу:

```
☐ 1. Какая программа не работает? (браузер / VSCode / бот)
☐ 2. На каком устройстве? (iPhone / Mac / PC / Android)
☐ 3. На каком сайте/сервисе? (РФ / foreign / AI)

☐ 4. Hiddify включён? (тумблер ON)
☐ 5. Hiddify в TUN-режиме или SOCKS5? (надо TUN)
☐ 6. Подписка обновлена? (Hiddify auto-update раз в сутки)
☐ 7. Профиль активен в Hiddify? (если их несколько)

☐ 8. curl ifconfig.me показывает IP сервера?
☐ 9. Прямой curl https://сайт.com работает с сервера?
☐ 10. Routing rules в Hiddify настроены? (§6)
☐ 11. DNS настроен правильно? (§6.8)

☐ 12. На сервере: 3X-UI работает? (curl https://сервер:2096/login)
☐ 13. На сервере: outbound с подписки работает? (логи в 3X-UI)
☐ 14. На сервере: маршрутизация settings → правило подключения inbound→outbound?
☐ 15. У провайдера-подписки серверы живые? (тест отдельным клиентом)
```
