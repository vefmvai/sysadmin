# Сосуществование веб-сайтов и VPN на одном сервере (кто слушает 443)

Этот документ — про **самый частый архитектурный конфликт на личном сервере**: на
машине уже работает VPN, и тут оператору нужен ещё и обычный сайт с HTTPS. Оба хотят
порт 443. Документ объясняет, кто на 443 имеет право слушать, почему Reality и
nginx-сайт не делят один порт, и как развязать конфликт по-человечески, не сломав
живым клиентам VPN.

Это **эталон уровня «весь сервер»**, а не только VPN-кусок: остальные networking-файлы
описывают VPN так, будто на сервере больше ничего нет. Здесь — про то, как VPN уживается
с сайтами, мониторингом, ботами на одной 443.

Связан с `_reference/transports.md` (устройство XHTTP/HTTPUpgrade), `_reference/fronting-strategies.md`
§5 (Reality + fallback), `_reference/routing-server-3xui.md` (маршрутизация внутри),
`_reference/3x-ui-panel.md` (раскладка панели). Скиллы-потребители: `/configure-vpn-routing`,
`/setup-vpn-panel`, `/finalize-vpn-routing`, `/deploy-service`.

## §1 Железный закон: один TCP-порт — один слушатель

На один TCP-порт (`443`) может биндиться **ровно один процесс**. Это закон ОС, не
вопрос настройки. Если на 443 уже сидит Xray — nginx туда не встанет, и наоборот.
`Address already in use` — это про него.

Отсюда всё остальное. Когда «и сайт, и VPN хотят 443» — кто-то один должен либо
**пустить другого через себя**, либо **уехать на другой порт**. Третьего нет.

**NEVER** молча сажать nginx на 443, где уже работает VPN, — это рвёт связь живым
клиентам. Это операция красной зоны (необратима для текущих подключений), не правка
конфига. См. §6.

## §2 Кто умеет жить ЗА nginx, а кто — нет

nginx (или Caddy/traefik) на 443 — это **HTTP/TLS-швейцар**: он сам терминирует TLS,
смотрит на домен/SNI и путь, и раздаёт трафик дальше. Чтобы швейцар мог завернуть
трафик куда-то внутрь, этот трафик должен приходить к нему как **обычный HTTPS**.

Разделение по этому признаку — главное в документе:

### 2.1 Живут за nginx (едут внутри обычного HTTPS)

- **VLESS+XHTTP** (он же SplitHTTP) — VPN-поток замаскирован под обычные HTTP-запросы.
  nginx видит HTTPS, по `location <секретный-путь>` заворачивает на локальный Xray
  (`security: none`, TLS терминирует nginx). Это «нативный» способ спрятать VPN внутри
  сайта.
- **VLESS+HTTPUpgrade** — то же семейство, апгрейд HTTP→туннель. Тоже проксируется nginx'ом.
- **VMess/Trojan/VLESS поверх WebSocket** — классика, тоже HTTP-совместимы.
- **Любые обычные сайты, боты, панели, мониторинг** — это и есть их родная среда.

### 2.2 НЕ живут за nginx (сами держат TLS-рукопожатие)

- **VLESS+Reality** — Reality делает **свой особый TLS-handshake**, имитируя чужой
  популярный сайт (`dest`/`serverName`). Это **не** обычный HTTPS — nginx такой
  handshake не понимает и завернуть не может. Reality обязан слушать порт **сам, напрямую**.
- **VLESS+Vision поверх «голого» TLS** — тот же случай, TLS свой.
- **Hysteria2 / TUIC** — это вообще **UDP/QUIC**, не TCP. nginx (HTTP-уровень) их не
  видит в принципе. Живут на своём UDP-порту мимо любого TCP-швейцара.
- **WireGuard / OpenVPN-UDP** — тоже UDP, мимо. (Плюс для обхода РФ они не годятся —
  TSPU детектирует, см. `_reference/vpn-protocols.md` и рефлекс 3.8.1.)

**Аналогия для оператора:** nginx-швейцар пускает только посетителей «в костюме HTTPS»
и читает у них на бейдже, куда идти. XHTTP надевает этот костюм. **Reality приходит в
собственном маскировочном костюме** — швейцар его не распознаёт, Reality ломится в
отдельную дверь сам. **Hysteria/WG приезжают вообще по другой дороге (UDP)** — мимо двери.

### 2.3 3X-UI — это не один процесс на одном порту, а массив инбаундов

Частая ошибка новичка: думать про 3X-UI как про «один VPN-сервер, который надо
поставить либо за nginx, либо параллельно». Это не так.

**3X-UI устроена иначе.** Это веб-панель (Go) + ядро Xray-core. У ядра в `config.json`
есть массив `inbounds` — каждый элемент это **отдельный приёмник со своим протоколом,
портом и интерфейсом**. Ядро слушает все эти порты одновременно. Панель — удобная
обёртка для редактирования массива (UI/API вместо ручной правки JSON).

Это значит: **одна 3X-UI = много инбаундов сразу, в любых комбинациях «за nginx /
параллельно / внутренние».** Не «выбор архитектуры», а норма.

У каждого инбаунда два независимых выбора:

**Где слушать (интерфейс):**
- `127.0.0.1:порт` — **только локально**. Снаружи не достучаться; работает, только
  если кто-то на этом же сервере (обычно nginx) пробросит сюда трафик. **За nginx.**
- `0.0.0.0:порт` — **наружу на любом интерфейсе**. Клиенты из интернета ходят сюда
  напрямую. **Параллельно nginx.**

**Какой протокол (определяет, имеет ли смысл первый выбор):**
- `XHTTP / HTTPUpgrade / WS` с `security: none` — едет внутри обычного HTTPS, TLS
  делает nginx снаружи. **Только за nginx** (`127.0.0.1`), иначе бессмысленно.
- `Reality / Vision` — держит свой TLS-handshake сам. **Только параллельно nginx**
  (`0.0.0.0:свой-порт`).
- `mixed / SOCKS / HTTP-прокси` — обычно `127.0.0.1`, для программ самого сервера
  (curl, pip, контейнеры). **Внутренний**, наружу не торчит.
- `Hysteria2 / TUIC` — UDP, всегда мимо nginx (другая дорога).

Эти два выбора перемножаются. Поэтому **в одной 3X-UI нормально жить одновременно**:
XHTTP-инбаунду на `127.0.0.1:10443` (за nginx, основной VPN для большинства),
Reality-инбаунду на `0.0.0.0:8443` (параллельно nginx, прямой вход для тех, кому
XHTTP капризит — см. §5), и mixed-инбаунду на `127.0.0.1:1080` (внутренний прокси
для программ сервера).

Это не «или-или», это **семейство дверей, управляемое из одной панели**. Эталон ниже
(раскладки A/B/C) описывает именно такие комбинации.

### 2.4 Порт 80 — тоже nginx, и зачем

nginx у тебя слушает **оба порта одним процессом: 80 (HTTP) и 443 (HTTPS)**. На 80 он
такой же привратник, просто его работа там сильно проще.

На 80 живут **ровно две вещи**:

1. **ACME-челлендж для Let's Encrypt.** Когда acme.sh (или certbot) выпускает/продлевает
   сертификат через **HTTP-01**, Let's Encrypt стучится на `http://<домен>/.well-known/acme-challenge/<токен>` —
   **по 80, без шифрования**. nginx отдаёт файл челленджа из ACME-папки. Если на 80
   никого нет — сертификаты не продлятся, через 90 дней сайты сломаются. Это
   обязательная инфраструктурная роль 80 порта.
2. **Редирект `http://` → `https://`.** Пользователь набрал `example.com` без `https://` —
   браузер по умолчанию идёт на 80. nginx отвечает `301 Moved Permanently → https://<тот-же-домен><тот-же-путь>`,
   и браузер переходит на 443. Без этого половина пользователей видела бы «сайт
   недоступен».

Никаких сайтов и проксирований на 80 быть не должно — вся реальная работа на 443.
80 это «парадный вход для тех, кто пришёл без https, и для робота Let's Encrypt».

**NEVER** закрывать порт 80 в UFW, даже если «сайт только по https» — это сломает
автопродление сертификатов через HTTP-01. (Альтернатива — DNS-01 challenge через
API DNS-провайдера, но это отдельная история и не дефолт.)

### 2.5 Полная картина портов

Сводная таблица «кто на каком порту в эталонном сервере»:

| Порт | Кто слушает | Что делает |
|---|---|---|
| **80/TCP** | nginx | ACME HTTP-01 + редирект `http://` → `https://` |
| **443/TCP** | nginx | привратник: TLS, по домену/пути раздаёт сайты и XHTTP-VPN внутри |
| **8443/TCP** (опц.) | Xray Reality напрямую | VPN-инбаунд параллельно nginx |
| **9443/UDP** (опц.) | Hysteria2 напрямую | VPN-инбаунд по UDP, мимо nginx |
| **127.0.0.1:разные** | сайты-апстримы, mixed-прокси Xray, боты | внутренние, наружу не торчат |

Снаружи открыты только три «двери» — 80, 443 и опциональные VPN-порты. Всё остальное
живёт на `127.0.0.1` и достижимо только через nginx-привратника. Это и есть «nginx как
главный привратник на сервере».

## §3 Три эталонные раскладки 443 (и когда какая)

### Раскладка A — nginx-швейцар, VPN внутри через XHTTP (богатая, дефолт для «много сайтов + VPN»)

```
:443  nginx ── по домену/пути ──┬── site1.example.com        → сайт
                                ├── site2.example.com        → сайт/бот/панель
                                └── site1.example.com/<секрет> → Xray XHTTP (127.0.0.1:10443, security:none)
:80   nginx  ── HTTP-01 ACME + redirect → 443
```

- Один порт 443, один швейцар, сколько угодно сайтов **и** VPN мирно вместе.
- VPN снаружи неотличим от обычного сайта-маскировки (XHTTP внутри HTTPS).
- Цена: VPN-транспорт обязан быть XHTTP/HTTPUpgrade, **не** Reality.
- Это архитектура «как у зрелого сервера, где сайтов много». Подтверждённый рабочий
  паттерн (боевой `news.vefmvtech.ru/<путь>` → Xray XHTTP на `127.0.0.1:10443`).

### Раскладка B — Reality на 443, сайт на отдельном порту (минимальная, нулевой риск)

```
:443   Xray VLESS+Reality (0.0.0.0)   → VPN, маскировка под чужой сайт
:8443  nginx                          → сайт example.com:8443 (порт в адресе)
:80    nginx                          → ACME + сайт
```

- VPN **не трогаем вообще** — нулевой риск для живых клиентов.
- Цена: у сайта в адресе торчит `:8443`. Для калькулятора/утилиты — неважно.
- Когда брать: VPN важнее, сайт второстепенен, не хочется переезда.

### Раскладка C — Reality уезжает на 8443, сайт получает чистый 443 (когда нужен Reality И чистый URL)

```
:443   nginx       → сайт example.com (чистый URL, без порта)
:8443  Xray Reality → VPN (тот же протокол, тот же конфиг, меняется ТОЛЬКО порт)
:80    nginx       → ACME + redirect
```

- Сайт получает чистый URL на 443.
- VPN остаётся **честным Reality** — XHTTP не нужен. Меняется только порт: клиентам
  дописать `:8443` в ссылку, конфиг не переделывается.
- nginx рулит **только сайтами**, VPN живёт рядом мимо швейцара.
- Когда брать: оператор хочет именно Reality (а не XHTTP) **и** чистый адрес сайта.
- **Это НЕ «и Reality, и nginx-швейцар на одном 443»** — такого не бывает (§1). Это
  «развести по портам»: 443 сайту, 8443 VPN.

### Раскладка D — SNI-роутинг (Xray fallback / nginx ssl_preread) — продвинутая, осторожно

Xray на 443 принимает TLS, и по **SNI** (имени домена в запросе) раздаёт: на свой сайт
по одному имени, на VPN по другому. Достигается через Xray `fallbacks` по `name`/`alpn`
или nginx `stream` + `ssl_preread`.

- Плюс: и VPN-Reality, и чистый URL сайта на одном 443.
- Минус: правка **живого** Xray (sqlite + рестарт ядра), ошибка роняет VPN. Сложнее
  отлаживать, выше риск. Брать только при реальной необходимости и с бэкапом.

## §4 Decision tree: что выбрать

```
Нужен ли сайту чистый URL без :порта?
├── НЕТ (порт в адресе ок)
│     → Раскладка B. Reality на 443, сайт на :8443. Нулевой риск. ДЕФОЛТ для «не трогать VPN».
│
└── ДА (чистый URL обязателен)
      Какой транспорт у VPN / готов ли менять?
      ├── VPN уже XHTTP/HTTPUpgrade ИЛИ готовы перейти на него
      │     → Раскладка A. nginx-швейцар на 443, VPN внутри по секретному пути.
      │       Богато, но XHTTP-за-nginx бывает капризен в sing-box (см. §5).
      │
      ├── Нужен именно Reality, XHTTP НЕ нужен
      │     → Раскладка C. Сайт на 443 за nginx, Reality уезжает на :8443.
      │       Клиентам дописать :8443. ПРОСТО и без смены транспорта. ← частый правильный выбор
      │
      └── Нужны и Reality, и чистый URL, и без второго порта в ссылке
            → Раскладка D (SNI-роутинг). Продвинуто, правка живого Xray, только с бэкапом.
```

**Дефолтная рекомендация для вайбкодера-новичка:** если важно «не сломать рабочий VPN» —
Раскладка B. Если важен чистый URL и VPN — это Reality — Раскладка C (развести по портам).
Раскладку D предлагать только осознанно желающему чистый URL без второго порта.

## §5 Известный подводный камень: XHTTP-за-nginx → PROTOCOL_ERROR

Боевой урок (selectel, 2026-05-21): XHTTP-инбаунд за nginx может давать
**`PROTOCOL_ERROR` в sing-box-клиентах** при неаккуратной настройке транспорта.
Лечение:

- точная настройка XHTTP: `host` = домен сайта-маскировки, `path` = секретный путь,
  `mode: auto`, `security: none` на инбаунде (TLS делает nginx);
- если не лечится для конкретных клиентов — **держать запасной Reality-инбаунд на
  отдельном порту** (например 8443) для проблемных клиентов. Это легитимное «и-и»:
  XHTTP за nginx для большинства + прямой Reality на 8443 для капризных.

Не воспринимать первую такую ошибку как «схема не работает» — это типовой нюанс XHTTP,
а не тупик. Источник нюанса — взаимодействие версий sing-box (нижняя планка 1.11.x,
см. рефлекс 3.8.3) с XHTTP-транспортом.

## §6 Переезд VPN с 443 — это красная зона, не правка конфига

Любая раскладка, где VPN **съезжает** с 443 (C, или переход Reality→XHTTP в A) — это
**необратимая для текущих подключений операция**. Обязательный порядок:

1. **Бэкап ДО всего**: `/etc/x-ui/x-ui.db` + текущий `config.json` (+ nginx-конфиги).
   Без бэкапа — не начинать. (См. C.3, §6.3 персоны — type-to-confirm.)
2. Вести как **красную зону**: ASSESS → PROPOSE (с планом отката) → EXECUTE → VERIFY.
3. **Предупредить клиентов заранее**: будет короткий обрыв VPN; при смене порта (C) —
   новый порт в ссылке; при смене транспорта Reality→XHTTP (A) — **новые клиентские
   конфиги целиком**.
4. Сломалось на полпути — **откат к бэкапу**, а не «дочинить руками» (§6.3 персоны).

Разница в цене для клиента по раскладкам:
- **C (Reality→:8443):** дописать `:8443` в существующую ссылку. Конфиг тот же. Дёшево.
- **A (Reality→XHTTP):** транспорт сменился → **полностью новые ссылки/конфиги**. Дороже.

Это и есть причина, по которой C часто предпочтительнее A, когда VPN — уже работающий
Reality: меньше боли клиентам.

## §7 Как объяснять это оператору (обёртка §4.3 персоны)

Тема сложная, оператор — вайбкодер. **NEVER** вываливать три-четыре раскладки как
экзамен с терминами «SNI-роутинг», «observatory», «sqlite + рестарт ядра». По §4.3:

1. **Контекст простыми словами:** «443 — единственная дверь сервера для HTTPS. Сейчас
   через неё ходит твой VPN. Сайту нужна та же дверь — но дверь одна».
2. **Мини-урок:** «VPN бывает двух видов входа: один (XHTTP) умеет прятаться внутри
   обычного сайта за швейцаром-nginx; другой (Reality) держит дверь сам и швейцара
   рядом не терпит».
3. **Варианты честно:** B (сайт на :8443, VPN не трогаем — безопасно), C (сайт на 443,
   VPN-Reality переезжает на :8443 — клиентам дописать порт), A/D — продвинутые.
4. **Рекомендация конкретная:** обычно B (если порт в адресе ок) или C (если нужен
   чистый URL и VPN — это Reality). Не «зависит от ситуации».
5. **Разрешение довериться:** «не хочешь разбираться — сделаю B, это безопасно и обратимо».
6. **Открытая дверь:** «хочешь чистый URL — скажу, как переехать на C без боли клиентам».
