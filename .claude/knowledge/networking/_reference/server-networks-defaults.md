# Серверные сети по умолчанию (Docker, сегментация, firewall)

Этот документ — про **правильную раскладку сетей на сервере с самого старта**. Не VPN, а
обычные docker-сети между контейнерами: какие, сколько, что в них живёт, как они соединены
с хостом и интернетом, какие порты выставляются наружу, кому можно `network_mode: host`,
зачем `internal: true`, как UFW дружит с Docker.

Это **эталон уровня «весь сервер целиком»** — соседний с
`_reference/web-and-vpn-coexistence.md` (там разобрана раскладка **портов на хосте**: 80/443
nginx, опц. 8443 Reality, опц. 9443 UDP). Здесь разбираются **сети между контейнерами и
изоляция хоста от контейнеров и наоборот** — слой ниже.

Связан с `_reference/web-and-vpn-coexistence.md` (§2.5 «полная картина портов»),
`_reference/3x-ui-panel.md` (3X-UI как массив инбаундов, частный случай этой темы для VPN),
а также со скиллами `/deploy-service`, `/bootstrap-new-server`, `/inventory-scan`,
`/install-monitoring-stack`, `/cleanup-existing-server`. Источники достоверности — Docker
docs (см. фронтматтер) + боевой ADR-0012 стека-носителя (4-сеть-сегментация, 5 недель
эксплуатации, 3 выстраданных урока в lessons-learned).

## §1 Железный закон + базовые понятия

### 1.1 Что такое docker-сеть простыми словами

Когда оператор запускает контейнер, ему нужно решить: **как этот контейнер общается с
остальным миром**. С другими контейнерами на том же хосте, с самим хостом, с внешним
интернетом. Docker для этого даёт **сети** — это виртуальные коммутаторы на хосте, через
которые контейнеры обмениваются пакетами.

Аналогия: представь сервер как многоквартирный дом. Каждый контейнер — отдельная квартира.
Docker-сеть — это **общий коридор**, куда квартира открывает свою дверь. В одном коридоре
квартиры слышат друг друга по именам («сходи к `postgres`, попроси воды»); в разных
коридорах — нет, разве что специально пробьют дверь.

### 1.2 default bridge vs user-defined bridge

Docker поставляется со встроенной сетью `bridge` (она же `docker0`). Если контейнер
запустить без явного указания сети — он попадёт туда. **Это плохая дверь.** На default
bridge контейнеры **не видят друг друга по имени** (DNS не работает), общаются только по
IP. Имена и DNS появляются только в **user-defined bridge** — сети, которую создаёт сам
оператор (`docker network create <name>` или объявление `networks:` в docker-compose).

**NEVER оставлять контейнер на default bridge.** Любой production-сервис обязан жить в
user-defined сети. Если в `docker ps` видишь контейнер без явной сети — это либо
техдолг, либо забытый тест.

### 1.3 Три способа «контейнер ↔ внешний мир»: expose, ports, network_mode

Это самая частая путаница в compose. Три ключевых слова значат **разные вещи**:

| Слово | Что делает | Когда брать |
|---|---|---|
| `expose: [<PORT>]` | Метаданные — никаких портов не открывает. Это «декларация: внутри контейнера слушается такой-то порт». Используется healthcheck'ами и инструментами документации. | Почти никогда руками — `EXPOSE` уже есть в Dockerfile официальных образов. Не путать с `ports:`. |
| `ports: ["<HOST>:<CONT>"]` | **Реальная публикация порта на хост.** Docker делает DNAT: пакеты с порта хоста идут в контейнер. Без префикса хоста — на `0.0.0.0` (все интерфейсы, видно из интернета). С префиксом `127.0.0.1:` — только локально на хосте. | Когда нужен **доступ с хоста или извне**: nginx-апстрим через 127.0.0.1, psql с хоста, SSH-туннель. |
| `network_mode: host` | Контейнер **делит сетевой namespace с хостом**. Никакой изоляции — все его порты сразу видны хосту, никаких docker-сетей он не использует. | Почти никогда. Только при доказанной технической необходимости (см. §6), записанной в ADR. |

И один ключ для сети:

| Ключ сети | Что делает |
|---|---|
| `internal: true` в `networks: <name>:` | Запрещает контейнерам в этой сети **выходить наружу через NAT** (egress в интернет). Дополнительно — блокирует DNAT для publish-портов (§5.1, важный нюанс). |

### 1.4 Железный закон

На один TCP-порт хоста — один слушатель (как в `web-and-vpn-coexistence.md` §1). Дальше
вопрос: **что должно слушать порт хоста?** Ответ: см. §2.

## §2 Что выставлять наружу

Правило одной строкой: **наружу торчит только nginx**. Всё остальное либо публикуется на
`127.0.0.1:port` (доступ только с хоста), либо вообще без `ports:` (доступ только из своей
docker-сети).

### 2.1 Полная картина

| Кто | Где слушает | Зачем |
|---|---|---|
| nginx на хосте | `0.0.0.0:80` + `0.0.0.0:443` | парадные двери — сайты, ACME, VPN-XHTTP внутри (см. `web-and-vpn-coexistence.md`) |
| опц. Reality / Hysteria2 | `0.0.0.0:8443` TCP / `0.0.0.0:9443` UDP | прямые VPN-инбаунды (см. `web-and-vpn-coexistence.md` §2.5) |
| все прочие сервисы | `127.0.0.1:<PORT>` | nginx ходит к ним через 127.0.0.1:<PORT>, psql с хоста, SSH-туннели |
| внутренние сервисы | без `ports:` | nginx ходит к ним **по имени сервиса** через общую docker-сеть |
| БД, кэши | без `ports:` (или 127.0.0.1 только если нужен psql с хоста) | только из своей `data`-сети |

**NEVER `ports: ["5432:5432"]` для БД.** Это публикация на `0.0.0.0:5432` — postgres
становится доступен из интернета. Правильно: либо `["127.0.0.1:5432:5432"]` (psql с хоста),
либо вообще без `ports:` (только из контейнеров в сети `data`).

### 2.2 Когда nginx ходит «по имени», а когда «через 127.0.0.1»

Два варианта подключить nginx к контейнеру-апстриму:

- **Через имя сервиса** — если nginx и апстрим **в одной docker-сети**:
  ```nginx
  proxy_pass http://my-app:3000;   # my-app — service key в compose
  ```
  Это работает за счёт встроенного Docker-DNS (`127.0.0.11:53` внутри user-defined сетей).
  Контейнер `my-app` НЕ нуждается в `ports:` — он достижим только из своей сети, что
  безопасно.

- **Через `127.0.0.1:port`** — если nginx **на хосте** (не в контейнере), а апстрим — в
  контейнере с `ports: ["127.0.0.1:<PORT>:<APP_PORT>"]`:
  ```nginx
  proxy_pass http://127.0.0.1:3000;
  ```
  Docker делает DNAT с хостового 127.0.0.1:3000 на контейнер. Снаружи это не видно
  (только loopback).

**NEVER** `proxy_pass http://127.0.0.1:port` из контейнера nginx, который сидит **не на**
`network_mode: host` — это локалхост самого контейнера, а не хоста, апстрима там нет.

## §3 Эталонная сегментация — 4 сети по ролям

Здесь — паттерн, проверенный 5 недель эксплуатации на стеке-носителе агента
(ADR-0012-docker-network-segmentation проекта-носителя). Имена сетей в твоём стеке могут
быть другими — важны **роли** и **свойства**, не имена.

### 3.1 Четыре роли

| Роль | Internal? | Что в ней | Свойство |
|---|---|---|---|
| **data** (хранилища) | `internal: true` | postgres, redis, mongo и их клиенты | без egress в интернет — защита БД от утечек даже при компрометации приложения |
| **services** (приложения) | нет | сайты, API, боты, воркеры, nginx-host-апстримы | обычная bridge-сеть, может ходить наружу |
| **proxy-corridor** (опц.) | нет, **фикс. subnet+gateway** | контейнеры, которым нужен обход блокировок через VPN/прокси на хосте | gateway хоста хардкодится в `.env` контейнеров: `HTTP_PROXY=http://172.18.0.1:1081` |
| **monitoring** | нет | Uptime-Kuma, Dozzle, Dockge, Diun, дашборды | имеет право подключаться к другим сетям для observability; другие сети monitoring не видят |

В стеке-носителе они называются `data` / `services` / `xray` / `monitoring`. Выбирай свои
имена — но смысл сохраняй.

### 3.2 Почему четыре, а не одна плоская сеть

Если все контейнеры в одной общей сети — компрометация любого контейнера даёт атакующему
видимость БД, прокси-маршрутов, мониторинга. Сегментация снижает blast radius: упал nginx —
БД в `data` всё ещё не видна снаружи.

Дополнительно: **разные политики egress**. БД в `data internal: true` физически не может
утащить дамп наружу через интернет, даже если в контейнере выполнили `curl evil.com/dump`.
Это defense-in-depth, не паранойя.

### 3.3 Паттерн «БД в двух сетях» (важно)

Сценарий: postgres сидит в `data internal: true` (защищён), но оператору нужен `psql` с
хоста для админских задач. Очевидное решение — добавить `ports: ["127.0.0.1:5432:5432"]`.

**Это не сработает.** `internal: true` блокирует не только egress, но и **DNAT-правила
Docker для publish-портов**. Контейнер недостижим даже через 127.0.0.1:5432.

Решение паттерном: **postgres подключён к двум сетям одновременно** — `data` (для
defense-in-depth и обычных клиентов) + `services` (для DNAT, чтобы работал
`127.0.0.1:5432`). Это не костыль и не техдолг — это документированное исключение в ADR
стека-носителя. См. §5.1 «подводные камни» для механики.

Аналогичный паттерн для любого контейнера в `data`, которому нужна публикация порта на
хост.

## §4 Decision tree «новый сервис → как подключать»

Вайбкодеру не нужно держать в голове 4 роли и 3 ключевых слова compose. Дерево решений:

```
Что это за сервис?

├── БД или кэш (postgres / redis / mongo / etcd / ...)
│     networks: [data]
│     БЕЗ ports: (только из контейнеров своей сети)
│     Нужен доступ с хоста для psql/redis-cli?
│       → добавить вторую сеть [services] для DNAT (см. §3.3)
│       → ports: ["127.0.0.1:5432:5432"]
│
├── Прокси-клиент (контейнер, которому нужен обход блокировок РФ)
│     networks: [services, proxy-corridor]
│     В .env: HTTP_PROXY=http://<gateway-proxy-corridor>:1081 и аналогичные
│     БЕЗ ports: (если nginx в сети services)
│
├── Монитор (Uptime-Kuma / Dozzle / Dockge / Diun / собственный дашборд)
│     networks: [monitoring]
│     Если мониторит БД — добавить [data]
│     ports: ["127.0.0.1:<PORT>:<APP_PORT>"] для проброса в nginx
│     /var/run/docker.sock — только :ro и понимая риски (§8)
│
└── Обычный веб-сервис (сайт, API, бот, воркер)
      networks: [services]
      Если читает БД — добавить [data]
      Если нужен обход блокировок — добавить [proxy-corridor]
      БЕЗ ports: если nginx в той же сети — апстрим по имени сервиса
      ports: ["127.0.0.1:<PORT>:<APP_PORT>"] — если нужен доступ с хоста / nginx-host
```

**Антипаттерны, которые tree отбрасывает по построению:**

- `networks: [default]` или вообще без `networks:` → default bridge, нет DNS (§1.2).
- `ports: ["5432:5432"]` без префикса `127.0.0.1:` → публикация на `0.0.0.0` (§2.1).
- `network_mode: host` для обычного веб-сервиса → потеря изоляции (§6).
- Монтирование `/var/run/docker.sock` без `:ro` → root-эквивалент (§8).

## §5 Подводные камни (3 выстраданных урока)

Эти грабли пришли из боевой эксплуатации стека-носителя — реальные, не «лучших практик с
интернета».

### 5.1 `internal: true` ломает publish-порты

**Симптом:** контейнер в сети с `internal: true` имеет `ports: ["127.0.0.1:<PORT>:..."]`,
но `psql -h 127.0.0.1 -p <PORT>` не подключается. С хоста порт не отвечает.

**Причина:** `internal: true` запрещает все исходящие правила NAT в `iptables` для этой
сети. Docker создаёт правила DNAT (хостовой порт → IP контейнера) в том же chain, который
блокирует `internal`. Результат — DNAT не работает, контейнер недостижим с хоста, хотя
порт «вроде бы опубликован» в compose.

**Лечение:** подключить контейнер ко **второй сети без internal** (паттерн §3.3). DNAT
сработает через неё, а защитный периметр `data` сохранится для контейнер-к-контейнеру
обращений.

Перекрёстная ссылка: `.claude/skills/cleanup-existing-server/references/typical-grabli.md`
§10 — там этот случай тоже разобран, но в контексте чистки legacy-сетей.

### 5.2 `external: true` сети не удаляются при `docker compose down`

**Симптом:** в compose объявлено `networks: <name>: external: true`. После `docker compose
down` контейнеры остановлены, но сеть `<name>` осталась. Кажется, что compose работает
криво.

**Причина:** `external: true` означает «эта сеть управляется не этим compose-файлом, она
существует сама по себе» (создана `docker network create` или другим compose). Поэтому
`compose down` её не трогает — это by design, иначе разные стеки на одной сети валили бы
друг друга.

**Лечение:** если действительно нужно удалить — отдельным шагом `docker network rm
<name>` после того, как **все** контейнеры из неё отключены или удалены. Запросто
становится Red Zone (см. trust-zones), если контейнеры активны: их связь оборвётся.

### 5.3 Service key в compose автоматически становится DNS-алиасом

**Симптом:** переименовал контейнер через `container_name: my-app-v2`, а в nginx остался
`proxy_pass http://my-app:3000` — и nginx по-прежнему ходит к старому имени. Или наоборот:
переименовал service key с `app` на `webapp`, а compose не подтягивается, потому что в
других compose-файлах ссылаются на сеть, ожидающую сервис `app`.

**Причина:** в user-defined сети Docker автоматически создаёт DNS-алиас **из service key
compose** (это ключ в `services:`, не `container_name`). `my-app:3000` резолвится потому,
что в каком-то compose-файле есть `services: my-app: ...`, подключённый к той же сети.
`container_name` — отдельная штука, на DNS не влияет.

**Лечение:** при переименовании сервиса синхронно править: (а) service key в compose
(`services:` верхнего уровня), (б) все `proxy_pass http://<name>:...` в nginx,
(в) `depends_on:` в других сервисах, (г) `HTTP_PROXY`/`UPSTREAM_HOST` в .env-файлах,
указывающих на этот сервис по имени. `container_name` тоже желательно — но это
косметика для `docker ps`, не для DNS.

## §6 `network_mode: host` — только с обоснованием в ADR

`network_mode: host` снимает сетевую изоляцию контейнера: он делит namespace с хостом.
Все его порты сразу слушаются хостом, никаких docker-сетей он не видит, UFW работает
напрямую с этими портами, observability сетей по нему не работает.

**Цена:**
- Изоляция исчезает — контейнер видит все хостовые порты, может биндиться на любые.
- Конфликт портов: два host-network контейнера, желающие порт 8080, не запустятся.
- Сеть не видна в `docker network inspect`, в диаграммах inventory.

**Когда брать (примеры легитимных кейсов):**
- Контейнер использует UDP-протокол с большим диапазоном эфемерных портов (FRP-туннели,
  игровые серверы).
- Контейнер раздаёт мультикаст/broadcast (mDNS-discovery, SSDP).
- Доказанный баг bridge-сети для конкретного протокола (например, UDP 45876 у Beszel-hub —
  bridge режет, host работает; см. `.claude/skills/install-monitoring-stack/SKILL.md:245`).

**NEVER** добавлять `network_mode: host` молча, без записи **зачем** в ADR проекта-носителя.
Каждый случай — отдельная запись. Документированные исключения стека-носителя
(`beszel`, `beszel-agent`, `jarvis-frps`) — это не разрешение копировать паттерн, это
архив причин для будущего ревью.

## §7 UFW + Docker (DOCKER-USER chain)

Главный сюрприз для новичка: **Docker обходит UFW**. Когда контейнер запущен с `ports:`,
Docker вставляет свои `iptables`-правила в chain `DOCKER`, который применяется **до**
UFW-правил. Результат: `ufw deny incoming` на хосте не блокирует доступ к контейнерному
порту извне.

Три уровня решения по возрастанию радикальности:

### 7.1 Дефолт: биндить на `127.0.0.1`

`ports: ["127.0.0.1:<HOST_PORT>:<CONT_PORT>"]` — Docker создаёт DNAT только для loopback,
из интернета не достучаться. UFW для этого порта не нужен — он и так не виден снаружи.
**Это работает для 95% случаев** и не требует никакого UFW-маневра. Все примеры в §2 и §4
рекомендуют именно это.

### 7.2 Если порт обязан слушать наружу: DOCKER-USER chain

Когда правда нужно `ports: ["0.0.0.0:<PORT>:..."]` (например, nginx-host, который у тебя
не на host network), UFW делает по-другому:

- В `/etc/ufw/after.rules` добавляется блок правил в chain `DOCKER-USER` (его Docker
  смотрит **перед** своими правилами).
- Правила фильтруют source-IP / sub-сети / порты и решают, пускать ли пакет до Docker'а.

Детали и образцы — `.claude/skills/bootstrap-new-server/references/ubuntu-vs-debian-quirks.md`,
секция «UFW vs Docker». Здесь не дублирую.

### 7.3 Радикально: отключить Docker iptables

`/etc/docker/daemon.json`: `{"iptables": false}`. Docker не пишет правила; всё на UFW. Но
тогда **контейнеры теряют сеть до интернета**, пока ты руками не пропишешь NAT в UFW. Это
для редких случаев (kubernetes-кластер с CNI, специфичная сеть). Дефолт — не брать.

**Дефолт-выбор:** §7.1 (`127.0.0.1:port`) для всех контейнеров, кроме явных публичных
демонов. Тогда UFW нужен только для хостовых сервисов (sshd, опц. VPN-инбаунды) и не
конфликтует с Docker.

## §8 `/var/run/docker.sock` — root-эквивалент

Сокет `/var/run/docker.sock` — это API демона Docker. Кто может в него писать, тот
**эквивалентен root на хосте**. Тривиальный exploit: запустить контейнер с
`--privileged -v /:/host` и chroot — за 2 секунды выйти из контейнера на хост в режиме root.

Поэтому правила работы с сокетом:

- **NEVER** монтировать `/var/run/docker.sock` без явной необходимости. Большинству
  контейнеров он не нужен.
- Если нужно (Dozzle, Dockge, watchtower, Portainer — им docker.sock нужен по
  определению): монтировать **только** `:ro` (`/var/run/docker.sock:/var/run/docker.sock:ro`).
  Это не идеальная защита (некоторые API-эндпоинты доступны read-only и достаточно опасны),
  но снимает 80% риска.
- **NEVER** выставлять контейнер с docker.sock наружу без аутентификации. Dozzle —
  через simple-auth, Dockge — через basic-auth nginx, Portainer — admin password. Без
  auth публичный Dozzle = root-shell на хосте.
- Альтернатива минимальных прав — **socket-proxy** (`tecnativa/docker-socket-proxy`):
  отдельный контейнер, который выставляет наружу только нужные API-методы. Это лишнее
  звено, но снимает риск целиком. Дефолтом не делаю — добавляет сложность; включаю в
  ADR-обсуждение, когда контейнер с docker.sock публичен.

## §9 Как объяснять оператору-новичку (обёртка §4.3 персоны)

Когда вайбкодер просит «помоги развернуть новый сервис», агенту не нужно вываливать всю
матрицу выше. Обёртка из 6 шагов (по §4.3 персоны):

1. **Контекст простыми словами:** «Сервер — как многоквартирный дом. Контейнеры — квартиры.
   Docker-сеть — общий коридор, куда квартиры открывают двери. Чтобы никто не перепутал, кто
   с кем общается, мы делаем несколько отдельных коридоров: для хранилищ, для приложений,
   опционально для прокси, опционально для мониторинга».
2. **Мини-урок:** «У сервиса три вопроса: в каком коридоре он живёт (`networks:`), какие
   двери открывает наружу (`ports:`), и пускаем ли его в интернет (`internal: true` запрещает)».
3. **Варианты честно:** перечислить раскладки A/B/C из §4 dec.tree для конкретного сервиса,
   с плюсами и минусами.
4. **Рекомендация конкретная:** «Для postgres + API я рекомендую так — БД в `data` для
   защиты, в `services` для psql с хоста; API в `services`; ports у API нет — nginx ходит к
   нему по имени сервиса в сети `services`».
5. **Разрешение довериться:** «не хочешь разбираться глубже — сделаю по рекомендации,
   обратимо».
6. **Открытая дверь:** «если захочешь понять, **почему** БД в двух сетях — спроси, расскажу
   про `internal: true` и DNAT».

NEVER вываливать `internal/DNAT/iptables/socket-proxy` сразу. Это для второго круга, когда
оператор сам захочет разобраться.
