# Evals для /sysadmin-init

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «настрой агента»
- «первый запуск»
- «init agent»
- «/sysadmin-init»
- «хочу как у Василия»
- «перенастрой конфиг»
- «поменять язык агента»
- «переключить менеджер паролей»
- «обнови sysadmin-config.json»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «познакомь с агентом» — это `/sysadmin-meet` (объяснение), не настройка конфига
- «настрой сервер» — это `/bootstrap-new-server`, не агент → редирект
- «настрой бэкапы» — это `/setup-backups`, не агент
- «расскажи что умеет агент» — это объяснение, ответ персоны или `/sysadmin-meet`
- «удали sysadmin-config.json» — обратная операция, скилл этого не делает → ответ персоны: «удалить вручную, потом /sysadmin-init создаст заново»
- «настрой второй сервер» — добавление сервера в существующий конфиг — это `/sysadmin-init --reconfigure`, активировать в этом режиме

## Edge-cases (опционально)

- «у меня уже есть конфиг, что-то с ним не так» — пограничная: либо `--reconfigure` (перенастройка), либо ad-hoc починка JSON → уточнить «что именно сломалось — структура файла или поля?»
- «миграция конфига после обновления агента» — пограничная: если изменился JSON Schema → активировать `--reconfigure` для прохода по новым полям
- «нужен конфиг без интервью, скопируй example» — поддерживается, но потом всё равно нужно подставить реальные значения → ответ персоны: «копирую example, но без `/sysadmin-init` 4 скилла не заработают — пройди вопросы»
