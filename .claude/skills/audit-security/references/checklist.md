# Чек-лист security-аудита: обоснование каждого пункта

Документация для понимания «зачем». Каждый пункт чек-листа в `security-audit.sh`
имеет конкретный риск, который он закрывает.

Связанные документы:
- `knowledge/security-hardening.md` — базовое укрепление Ubuntu-сервера.
- `decisions/0010-local-images-latest-allowed.md` — почему локальные сборки могут
  быть `:latest`.

---

## Host scope

### 1. UFW активен + default deny incoming

**Риск без проверки:** на сервере публичный IP, любой порт по умолчанию доступен
из интернета. Если приложение случайно слушает на 0.0.0.0 (а не 127.0.0.1) — оно
открыто для атаки без firewall'а.

**Что должно быть:**
```bash
ufw status verbose
# Status: active
# Default: deny (incoming), allow (outgoing), disabled (routed)
```

**Почему `deny incoming` важнее всего:** на публичных VPS постоянно идёт сканирование
портов (SSH brute force, Redis 6379, MongoDB 27017). Default deny — защитный слой,
даже если приложение случайно публикуется наружу.

### 2. UFW allow только 22/80/443

**Риск:** если открыты дополнительные порты «для удобства» (8080, 5432, 6379) —
прямой target для атаки.

**Исключения, которые допустимы:**
- Кастомный SSH-порт (например, 2222 вместо 22) — задокументировать в `inventory/server.md`.
- Provider-level firewall на VPN/SSH — может открывать дополнительные порты для
  оператора, но не для всех.

### 3. SSH PasswordAuthentication no

**Риск:** SSH с паролями уязвим для brute-force. Даже с fail2ban, слабый пароль
ломается за часы перебором словарных вариантов.

**Почему ключи безопаснее:** ed25519 ключ — это эффективно 256-бит entropy,
brute-force невозможен в обозримое время. Пароли же — обычно < 64 бит entropy.

### 4. SSH PermitRootLogin не yes

**Риск:** root-логин = полный root доступ при компрометации. С `prohibit-password`
оператор должен SSH как обычный пользователь, потом sudo — двухэтапная защита.

**Допустимые значения:**
- `no` — root не может логиниться через SSH вообще.
- `prohibit-password` — root может только по ключу, без пароля.

`yes` — небезопасно.

### 5. fail2ban active + sshd jail

**Риск:** даже с ключами SSH, brute-force forçает запись в логи и тратит ресурсы
(fork SSH-сессии). fail2ban банит IP после N неудачных попыток.

**Что важно проверить:**
- `systemctl is-active fail2ban` → active
- `fail2ban-client status sshd` → отвечает (jail настроен, не просто демон работает)

### 6. unattended-upgrades без auto-reboot

**Риск без unattended-upgrades:** security-патчи приходят с задержкой («руки
не дошли»), уязвимости открыты дольше необходимого.

**Риск с auto-reboot включённым:** сервер ребутает себя в случайный момент
(обычно 2-3 ночи), это сюрприз для оператора, особенно если идут долгие задачи.

**Решение:** unattended-upgrades включён, но `Automatic-Reboot "false"`.
Оператор сам решает когда ребутнуть после kernel update.

### 7. Открытые внешние порты только 22/80/443

**Риск:** контейнер случайно публикует порт через `ports: - "5432:5432"` —
PostgreSQL открыт миру. UFW его не блокирует (Docker обходит UFW через iptables,
см. `knowledge/security-hardening.md`).

**Что проверять:** `ss -tlnp` и фильтр на 0.0.0.0 (а не 127.0.0.1). Все нестандартные
порты должны слушать на 127.0.0.1.

## Docker scope

### 1. daemon.json без insecure-registries

**Риск:** `insecure-registries` отключает TLS-валидацию registry. Это допустимо
для внутреннего registry без TLS, но опасно для публичного — MITM может
подменить образ.

**Допустимое исключение:** внутренний Docker registry в той же приватной сети,
без публичного выхода. Документировать в `inventory/server.md`.

### 2. .env permissions mode 0600

**Риск:** `.env` обычно содержит DB passwords, API tokens. Mode 644 = читаемо
любым пользователем сервера. Если есть второй пользователь (deploy-bot,
sandbox), он читает все секреты.

**Что должно быть:** `chmod 600 .env` — только владелец может читать.

### 3. Внутренние UI на 127.0.0.1

**Риск:** Uptime Kuma, Beszel, Dozzle, Dockge содержат интерфейсы для управления
сервером. Если они открыты на 0.0.0.0 без auth — кто угодно из интернета может
управлять контейнерами.

**Что должно быть:** `127.0.0.1:3001:3001` (bind только на localhost). Доступ
снаружи — через nginx с Basic Auth или WebAuthn.

### 4. Образы pinned

**Риск без pin:** `:latest` тег обновляется silent, при `docker compose up -d`
может прийти новая версия с критичной бага или backdoor (supply chain attack).

**Допустимое исключение:** локальные сборки (контейнеры, собираемые из исходников
проекта в `build:` блоке compose) — `:latest` эквивалентен текущему git-состоянию
исходников. Принцип формализован в ADR 0010 проекта-носителя или эквивалентном
решении.

## Git scope

### 1. gitleaks scan (working tree + history)

**Риск:** оператор случайно закоммитил `.env`, ключ, пароль. Даже если потом
удалил — в git history живёт навсегда (пока не сделан `git filter-repo`).

**Что важно:** скан истории (`--log-opts='--all'`) находит утечки в любом
коммите, не только текущем.

**Что делать при FAIL:** ротировать утёкшие секреты (через `rotate-secrets`),
`git filter-repo` для удаления из истории, force-push в ремоут (если есть).
Force-push в публичный репо — отдельный риск, рассматривать в Yellow Zone.

### 2. .gitignore содержит security patterns

**Риск:** если `.env`, `*.key`, `*.pem`, `secrets/` не в .gitignore — оператор
случайно `git add .` зацепит их.

**Что должно быть:** базовые patterns + специфичные для проекта (например,
`*.tfstate` для Terraform, `node_modules/` стандартно).

## TLS scope

### Сертификат не истекает в ближайшие 14 дней

**Риск:** просроченный сертификат → браузер ругается → пользователи не заходят.

**Что должно быть:** acme.sh обновляет за 30 дней до истечения автоматически
(cron). Аудит проверяет, что cron работает.

**Пороги:**
- < 14 дней: FAIL — что-то сломалось в acme.sh, срочно расследовать.
- 14-30 дней: WARN — приближается срок, но cron должен сработать. Проверь, не
  упал ли cron при последнем обновлении.
- > 30 дней: PASS — нормально.

## Что НЕ проверяется (явные out-of-scope)

- **Application-level security** (SQL injection, XSS, CSRF в самом приложении) —
  это ответственность разработчика приложения, не security-аудита инфры.
- **Network-level monitoring** (Wireshark, IDS) — не входит в базовый аудит,
  отдельный продукт.
- **Compliance-frameworks** (PCI-DSS, GDPR audit trails) — отдельная тема, не
  закрывается этим скиллом.
- **Backup-проверка** — отдельный скилл `setup-backups` + check-phase2.sh.

## Связь с security-hardening.md

Чек-лист — это **аудит** того, что описано в `knowledge/security-hardening.md`
как **рекомендации**. Если security-hardening.md обновляется (новая практика
добавляется), этот чек-лист тоже должен обновиться.
