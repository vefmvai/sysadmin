# Evals для /audit-security

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «проверь безопасность сервера»
- «сделай аудит безопасности»
- «security audit»
- «проверь хардеринг»
- «audit hardening»
- «compliance check»
- «проверь не утекли ли секреты в git»
- «прошло три месяца с последней проверки — пройдись по чек-листу»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «исправь дыры в безопасности» — это операция, audit-security read-only → ответ персоны: сначала /audit-security для отчёта, потом обсуждаем что чинить вручную
- «проверь как там сервер» — общая диагностика → `/health-check`
- «настрой fail2ban» — настройка с нуля → часть `/bootstrap-new-server`
- «защити сервер от взлома» — слишком широкая формулировка → уточняющий вопрос: «у тебя свежий VPS (bootstrap) или работающий (audit + что починим)?»
- «найди уязвимости через nmap-сканер снаружи» — penetration test, не security audit → ответ персоны: отдельный жанр, у скилла другой scope
- «сделай ротацию секретов» — это `/rotate-secrets`

## Edge-cases (опционально)

- «перед публикацией нового сервиса наружу — проверь меня» — context подходит для `/audit-security` (предрелизный аудит), но если оператор только что закончил deploy — сначала уточнить «прогнать только новый сервис или весь сервер?»
- «у меня incident, кто-то ломится по SSH» — пограничная: это `/audit-security` (проверить состояние) ИЛИ это уже инцидент-расследование (вне набора скиллов). Уточнить «уже взломали или подозреваешь попытки?»
- «проверь .env-файлы» — это часть audit-security, но узкая → активировать, но в режиме focused (только check на права 0600 и поиск секретов в git-истории)
