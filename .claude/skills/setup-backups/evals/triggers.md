# Evals для /setup-backups

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «нужны бэкапы»
- «настрой backup»
- «restic с retention»
- «offsite на S3»
- «бэкапы на Backblaze»
- «бэкапы на Яндекс.Диск через WebDAV»
- «без бэкапов нельзя ничего менять»
- «настрой бэкап БД с шифрованием»
- «алерт если бэкап старше 36 часов»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «восстанови данные из бэкапа» — это restore, обратная операция (Red Zone), не setup → ответ персоны: «есть в скилле прогон restore, но полный restore — отдельный runbook»
- «бэкап uploads и медиафайлов» — это другой шаблон (rsync/btrbk), не покрывается restic-стеком setup-backups → ответ персоны: «у меня скилл про БД-бэкапы, по uploads — ad-hoc подскажу подход»
- «проверь что бэкап свежий» — это `/health-check`, не setup
- «удали старые бэкапы» — это retention (часть setup), но если без настроенного retention — ответ персоны: «настрой retention через setup-backups, удалять руками опасно — можешь снести цепочку инкрементов»
- «поставь мониторинг» — это `/install-monitoring-stack`, не backups (хотя алерт о свежести бэкапа интегрируется в монитор)
- «бэкап на другой сервер» — частный случай, скилл умеет с SFTP, но фраза узкая → активировать с пояснением «лучше offsite в облако, второй сервер тоже может сгореть»

## Edge-cases (опционально)

- «настрой бэкапы, но хранилища нет» — скилл требует настроенное хранилище → ответ персоны: «сначала зарегистрируй S3/B2/WebDAV, дай credentials, потом запускаем setup»
- «бэкап без шифрования» — restic шифрует всегда, отключить нельзя → ответ персоны с объяснением почему это правильно
- «retention другой — 30+12+0» — поддерживается через параметры → активировать с кастомным retention
