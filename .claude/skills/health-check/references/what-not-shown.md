# Что НЕ показывает health-check (и мониторинг-стек)

Документ нужен, чтобы оператор не путал «здесь зелено» и «всё в порядке». У каждого инструмента
есть слепые зоны — ниже разбор этих зон по слоям.

## 1. Beszel

**Что показывает:** CPU / RAM / Disk хоста и каждого контейнера, network throughput, uptime.

**Что НЕ показывает:**

- **Application-level метрики.** Если у бизнес-приложения 1000 пользователей в очереди — Beszel
  показывает «CPU 30% RAM 40%», и всё выглядит нормально. Очередь видно только в логах
  приложения или через application-level метрики (prometheus exporters внутри сервиса —
  это уровень не v1, а v1.x как REQ APP-OBS-01).
- **Custom queries:** «сколько активных HTTP-сессий», «сколько rows вставили за минуту» —
  такого Beszel не умеет, это Prometheus/Grafana.
- **Business KPI:** конверсии, доход, ошибки оплаты — это слой выше (Plausible, internal
  dashboards конкретных продуктов).
- **Health истории всплесков короче 1 минуты.** Beszel опрашивает раз в минуту — flapping
  быстрее минуты теряется.

## 2. Uptime Kuma

**Что показывает:** HTTP-status доменов и TCP-доступность портов, TLS expiry, uptime SLA по периоду.

**Что НЕ показывает:**

- **Latency distribution.** Kuma показывает median response time, но не процентили (p95/p99).
  Если 5% запросов медленные — Kuma это не выделит.
- **Error rate breakdown.** «Сейчас 200 OK» ≠ «200 OK на всех endpoints». Kuma бьёт по одной
  URL — если /api/checkout сломан, а / в порядке, Kuma этого не увидит без отдельного монитора.
- **Внутренние ошибки сервиса.** Контейнер может отвечать 200 на healthcheck и при этом
  падать на реальных запросах с пятисотками — это видно только в логах.
- **Cascade failures.** Сервис healthy, но БД, к которой он ходит — тормозит. Kuma не видит
  цепочку зависимостей, только конечный endpoint.

## 3. Diun

**Что показывает:** доступны ли новые версии образов в registry (Docker Hub, GHCR, etc).

**Что НЕ показывает:**

- **Changelog.** «Версия 7.5 → 7.6» — но что изменилось? Diun не покажет, надо смотреть
  GitHub releases / RELEASE.md образа руками.
- **Breaking changes.** Diun ничего не знает о том, что 7.6 несовместима с твоим конфигом.
  Только тестовый прогон скажет.
- **Security advisories.** Diun не агрегирует CVE — для этого нужен Trivy / Snyk / Grype
  (это область скилла `audit-security`).

## 4. Сам этот скилл (health-check)

**Что показывает:** моментальный срез по основным сигналам.

**Что НЕ показывает:**

- **Долгосрочные тренды.** «Disk растёт по 1 ГБ в день» — это видит только Beszel за неделю.
  Health-check скажет «Disk 60%», но не «через 14 дней будет 90%».
- **Внутреннее состояние БД.** «Postgres healthy» = TCP-порт открыт + healthcheck pg_isready
  возвращает 0. Но репликация может отставать на сутки, autovacuum может сломаться, дисковый
  fragmentation расти — это видно только специализированными инструментами (pg_stat_*).
- **Drift конфигов.** docker-compose.yml на сервере и в репо могут расходиться (если правили
  напрямую). Health-check `MODE=full` показывает drift по списку контейнеров, но не по
  содержимому файлов. Полный drift — это `inventory-scan` + `git status` на сервере.
- **Состояние GitHub-репо и pipeline'ов.** Сломан ли deploy-key, актуален ли origin/main —
  это область деплоя, см. `deploy-service`.
- **Безопасность.** Открытые порты, weak passwords, CVE в образах — это `audit-security`.
- **Свежесть конкретных бэкапов БД.** Restic latest есть — но какие именно БД попали?
  Это `setup-backups` knowledge, не health-check.

## 5. Когда «всё зелёное», а на самом деле плохо

Реальные кейсы из практики:

1. **Бэкап «свежий», но битый.** Restic snapshot создан вчера, но restore-test не прогонялся
   полгода → внутри tar мусор. Health-check этого не видит. Защита: ежемесячный DR-runbook
   с реальным restore (RB-10).
2. **Сервис healthy, но сломан логически.** Healthcheck `curl /healthz` возвращает 200, но
   `/api/checkout` падает 500 — пишут пользователи в саппорт. Защита: бизнес-критичные
   endpoint'ы добавить в Kuma как отдельные мониторы.
3. **Сертификат «зелёный 13 дней», но обновляется не он.** acme.sh обновляет сертификат
   `example.com`, но nginx ссылается на старый путь после переустановки. Health-check
   увидит пометку «13 дней», но не подскажет, что после `--renew` нужен `nginx -s reload`.
   Защита: cert-reload-smart.sh + ручная сверка после каждого certbot-обновления.
4. **`docker ps` показывает 23/23 running, но один из них — забытый testcontainer.**
   Drift inventory ↔ реальность ловит это в `MODE=full`, но только если services.md
   обновлён. Защита: периодический inventory-scan.

## Куда идти за «слепыми» данными

- Application metrics → exporters внутри приложений + Prometheus (v1.x: APP-OBS-01)
- Долгосрочные тренды → Beszel UI, секция history (последние 90 дней)
- Безопасность → скилл `audit-security`
- Drift inventory ↔ реальность → скилл `inventory-scan`
- Состояние бэкапов с детализацией по БД → скилл `setup-backups` + restic stats
- Состояние deploy-pipeline → скилл `deploy-service` + `git status` на сервере
