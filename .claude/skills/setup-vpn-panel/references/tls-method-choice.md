# Выбор метода выпуска TLS для панели 3X-UI

Четыре способа выпуска Let's Encrypt сертификата, поддерживаемых скриптом
`03-configure-tls.sh`. Выбор зависит от конфигурации сервера, наличия nginx
и страны оператора.

**Дефолтный выбор скилла — `acme-webroot`.** Это безопасный путь для самого
частого сценария «сервер вырастет в многосервисный»: nginx ничего не моргает,
никаких секретов сторонних провайдеров на сервер не кладётся, продление
работает само. `acme-standalone` остаётся fallback'ом для одноразового VPS,
где **гарантированно** не будет ни nginx, ни сайтов.

## acme-webroot (РЕКОМЕНДАЦИЯ ПО УМОЛЧАНИЮ)

**Когда выбирать:**
- На сервере есть или планируется nginx (сайты, боты, n8n, мониторинг).
- Не хочется моргания nginx каждые 60 дней при продлении.
- Не хочется класть на сервер Global API Key Cloudflare (см. ниже).
- Сценарий «личный многосервисный сервер», который растёт.

**Как работает:**
1. На сервере уже стоит и запущен nginx (это требование метода).
2. Создаём папку, например `/var/www/letsencrypt/`.
3. В дефолтный server-блок nginx добавляется один `location`:
   ```nginx
   location ^~ /.well-known/acme-challenge/ {
       root /var/www/letsencrypt;
   }
   ```
   nginx перезагружается (`nginx -t && systemctl reload nginx`).
4. `acme.sh --issue -d $DOMAIN -w /var/www/letsencrypt --reloadcmd
   "systemctl restart x-ui"` — выпускает cert. acme.sh кладёт файл-челлендж
   в webroot, Let's Encrypt стучится по `http://$DOMAIN/.well-known/acme-challenge/<token>`,
   nginx отдаёт файл, остальные сайты продолжают работать без перерыва.
5. Привязываем cert+key к панели через прямую правку SQLite (как везде).
6. Порт 80 в UFW **остаётся открыт** (это нужно и для renew, и для редиректа
   `http://` → `https://` на сайтах). См. `web-and-vpn-coexistence.md` §2.4.

**Плюсы:**
- **Nginx не моргает ни при выпуске, ни при продлении.** Все сайты продолжают работать.
- **Никаких секретов сторонних провайдеров на сервере** (Cloudflare API Key не нужен).
- Renew (каждые 60 дней) полностью автоматический, без касания критичных сервисов.
- Универсально подходит и для РФ-операторов (см. acme-cloudflare ниже про подвох).

**Минусы:**
- Требует, чтобы nginx уже стоял и был запущен на момент выпуска.
- Один разовый шаг настройки: добавить location-блок в nginx-конфиг.
- Нужен deploy-hook (`--reloadcmd`), чтобы 3X-UI подхватил продлённый cert.

## acme-standalone (fallback для чистого VPS-без-сайтов)

**Когда выбирать:**
- Сервер **гарантированно** одноразовый или семейный VPS под VPS-only сценарий.
- nginx не установлен и не планируется.
- Доступ к Cloudflare API не настроен **и** webroot не подходит.

**Как работает:**
1. Открываем порт 80 в UFW (временно, на время выпуска).
2. Останавливаем `nginx`/`apache2` если они занимают 80 (если есть).
3. Acme.sh выпускает сертификат через HTTP-01 challenge, сам бинд на 80.
4. Привязываем cert+key к панели через прямую правку SQLite.
5. Восстанавливаем остановленные сервисы.
6. Закрываем 80 в UFW **только если на сервере нет сайтов** (renew acme
   делает временное открытие сам).

**Плюсы:**
- Не требует Cloudflare API.
- Не требует уже настроенного nginx.

**Минусы:**
- **Моргает nginx целиком (80 + 443) на 30-60 секунд** при выпуске и при каждом
  renew раз в 60 дней — для сервера с сайтами это бомба замедленного действия.
- Если на сервере планируется nginx с сайтами — `acme-webroot` лучше выбрать
  **сразу**, а не «потом переедем».

## acme-cloudflare (узкий случай, осторожно)

> ⚠️ **Для российских операторов и для доменов, обслуживающих российскую аудиторию,
> метод НЕ рекомендуется.** Cloudflare в России блокируется и замедляется —
> делегирование DNS на Cloudflare ради одного сертификата ухудшает доступность
> домена для целевой аудитории. Дефолт для РФ-контекста — `acme-webroot`.
>
> Дополнительно: Cloudflare Global API Key даёт **полный** доступ к аккаунту
> Cloudflare (не только к одному домену — ко всем доменам, биллингу,
> настройкам). Хранение его на сервере расширяет blast radius при компрометации
> сервера. Если выбрали этот метод — ключ только в менеджере паролей, а не в
> файлах на диске.

**Когда выбирать:**
- Сервер **гарантированно** обслуживает только нерусскоязычную аудиторию (или
  это инфраструктурный домен без публичного сайта).
- Домен **уже** делегирован на Cloudflare по другим причинам, и переезд DNS
  нежелателен.
- nginx **отсутствует** и его установка нежелательна (а значит, webroot
  неприменим).
- Оператор осознанно принимает риск Global API Key на сервере.

**Как работает:**
1. Передаём `CF_Email` + `CF_Key` через ENV.
2. Acme.sh выпускает сертификат через DNS-01 challenge (создаёт временную
   TXT-запись в Cloudflare).
3. Привязываем к панели.
4. Renew автоматический, без касания порта 80.

**Плюсы:**
- Не трогает порт 80 вообще.
- Работает без установленного nginx.

**Минусы:**
- **В РФ-контексте: Cloudflare блокируется/замедляется → ухудшение доступности
  домена.**
- Cloudflare Global API Key даёт полный доступ к аккаунту Cloudflare.
- Привязка к Cloudflare как DNS-провайдеру.
- `acme.sh` поддерживает только Global Key для встроенного `dns_cf`, не
  scoped Token с правами `Zone.DNS.Edit`.

## certbot-webroot (альтернатива acme-webroot для тех, кто на certbot)

**Когда выбирать:**
- На сервере уже стоит и запущен nginx.
- Сервер уже использует certbot для других сайтов — не хочется ставить
  ещё и acme.sh ради одного сертификата.
- Хочется единого инструмента для всех сертификатов на сервере.

**Как работает:**
1. Устанавливаем certbot (apt/dnf/yum), если ещё нет.
2. Создаём папку `$WEBROOT_PATH` (по умолчанию `/var/www/letsencrypt/`).
3. В дефолтный server-блок nginx добавляется `location` для ACME-челленджа
   (тот же, что у acme-webroot):
   ```nginx
   location ^~ /.well-known/acme-challenge/ {
       root /var/www/letsencrypt;
   }
   ```
   `nginx -t && systemctl reload nginx`.
4. `certbot certonly --webroot -w $WEBROOT_PATH -d $DOMAIN
   --deploy-hook "systemctl restart x-ui"` — выпускает cert без остановки nginx.
5. Копируем cert+key из `/etc/letsencrypt/live/$DOMAIN/` в
   `/root/cert/$DOMAIN/` (унифицированный путь для скилла) — либо делаем
   симлинк, либо настраиваем хук, который копирует при renew.
6. Привязываем к панели через SQLite (как везде).
7. Renew автоматический через `certbot.timer`, без касания nginx.

**Плюсы:**
- Знакомый инструмент для тех, кто уже на certbot.
- **Nginx не моргает** ни при выпуске, ни при продлении (как и acme-webroot).
- Renew стандартный (`certbot.timer`).
- Никаких сторонних API-ключей (Cloudflare и т.п.) на сервере не нужно.

**Минусы:**
- Требует уже работающий nginx (как и acme-webroot).
- Apt-пакет certbot отстаёт от мейнлайна на полгода-год — для базовых
  сценариев несущественно.
- Унификация путей `/root/cert/$DOMAIN/` требует доп. шага (симлинк или
  deploy-hook с копированием) — в отличие от acme.sh, который сам кладёт
  туда.

## certbot-standalone (fallback для VPS-без-nginx, кто на certbot)

**Когда выбирать:**
- Сервер чистый, nginx нет и не планируется.
- Оператор уже на certbot для других задач и не хочет ставить acme.sh.

**Как работает:**
1. Устанавливаем certbot.
2. Открываем 80 в UFW (временно), останавливаем nginx/apache если они есть.
3. `certbot certonly --standalone -d $DOMAIN`.
4. Копируем cert+key в `/root/cert/$DOMAIN/`.
5. Привязываем к панели через SQLite.
6. Создаём cron / systemd-timer для `certbot renew` (если ещё не настроен).

**Плюсы:**
- Знакомый инструмент.
- Работает без nginx.

**Минусы:**
- **Моргает 80 при выпуске и каждом renew** — та же бомба замедленного
  действия, что у acme-standalone, если на сервере появятся сайты.
- Apt-пакет отстаёт от мейнлайна.
- Pre/post hook на остановку/старт других сервисов на 80 — ручная работа.

## Сравнительная таблица

| Параметр | acme-webroot | certbot-webroot | acme-standalone | certbot-standalone | acme-cloudflare |
|---|---|---|---|---|---|
| Инструмент | acme.sh | certbot | acme.sh | certbot | acme.sh |
| Требует CF API | ❌ | ❌ | ❌ | ❌ | ✅ Global Key |
| Требует уже работающий nginx | ✅ | ✅ | ❌ | ❌ | ❌ |
| Моргает nginx при выпуске | ❌ | ❌ | ✅ (целиком) | ✅ | ❌ |
| Моргает nginx при renew | ❌ | ❌ | ✅ | ✅ | ❌ |
| Renew без касания критичных сервисов | ✅ | ✅ | ❌ | ❌ | ✅ |
| Безопасно для РФ-операторов | ✅ | ✅ | ✅ | ✅ | ❌ (CF блок.) |
| Wildcard-сертификаты | ❌ | ❌ | ❌ | ❌ | ✅ |
| Дефолт скилла | ✅ | — | fallback | — | — |

**Когда выбирать какой:**

- **acme-webroot** (дефолт) — сервер с nginx (или будет), не моргать.
- **certbot-webroot** — то же самое, но оператор уже на certbot для других задач.
- **acme-standalone** — VPS под VPN-only, nginx не будет.
- **certbot-standalone** — то же, но оператор уже на certbot.
- **acme-cloudflare** — только если домен **уже** на Cloudflare и аудитория
  **не** российская. Для РФ-контекста — отдельный осознанный выбор оператора.

## Что НЕ работает в текущей версии 3X-UI

- DNS-01 challenge без Cloudflare через встроенный acme в 3X-UI —
  только Cloudflare встроен. Yandex DNS, Route53, hosting.ru —
  не поддерживаются встроенным механизмом. Решение: использовать
  `acme.sh` напрямую (вне 3X-UI) с любым DNS-провайдером, потом
  привязать cert через SQLite.

## Известные проблемы

- **Issue #4390** (см. `3x-ui-panel.md` §7.1) — встроенный `ssl_cert_issue_main`
  в `x-ui.sh` зацикливается при автоматизации с закрытым stdin. Наш скилл
  работает **снаружи** этого механизма (acme.sh напрямую → SQLite UPDATE) —
  не попадает в этот баг.

- **Сертификат на IP вместо домена** — возможен через `--ip-cert` опцию
  в 3X-UI (6-дневные shortlived от Let's Encrypt). Этот скилл его не
  поддерживает, потому что 6-дневный cert требует автоматического renew
  каждые ~4 дня — это отдельный паттерн, не подходит для простой панели.
