# Выбор метода выпуска TLS для панели 3X-UI

Три способа выпуска Let's Encrypt сертификата, поддерживаемых скриптом
`03-configure-tls.sh`. Выбор зависит от конфигурации сервера и доступа
к Cloudflare API.

## acme-standalone (рекомендация для чистого сервера)

**Когда выбирать:**
- Сервер чистый или не имеет других сервисов на порту 80.
- Доступ к Cloudflare API не настроен.
- Сценарий «семейный VPN на отдельном VPS».

**Как работает:**
1. Открываем порт 80 в UFW (временно, на время выпуска).
2. Останавливаем `nginx`/`apache2` если они занимают 80.
3. Acme.sh выпускает сертификат через HTTP-01 challenge.
4. Привязываем cert+key к панели через прямую правку SQLite.
5. Восстанавливаем остановленные сервисы.
6. Закрываем 80 в UFW (renew acme делает временное открытие).

**Плюсы:**
- Не требует Cloudflare API.
- Стандартный путь, рекомендуется большинством туториалов.

**Минусы:**
- Краткое окно недоступности 80 на 30-60 секунд (если что-то использовало).
- Renew (каждые 60 дней) тоже нужен порт 80.

## acme-cloudflare (для серверов с другими сервисами на 80)

**Когда выбирать:**
- На сервере уже работает nginx с реальными сайтами (нельзя останавливать).
- Домен делегирован на Cloudflare.
- Готов положить Cloudflare API Key (Global, не Token).

**Как работает:**
1. Передаём `CF_Email` + `CF_Key` через ENV.
2. Acme.sh выпускает сертификат через DNS-01 challenge (создаёт временную
   TXT-запись в Cloudflare).
3. Привязываем к панели.
4. Renew автоматический, без касания порта 80.

**Плюсы:**
- Не трогает порт 80 вообще.
- Удобно для серверов с действующими сайтами.

**Минусы:**
- Cloudflare Global API Key даёт **полный** доступ к аккаунту Cloudflare.
  Хранить только в менеджере паролей. По возможности — использовать Token
  с минимальными правами (Zone.DNS.Edit для нужной зоны), но `acme.sh` пока
  поддерживает только Global Key для встроенного `dns_cf`.
- Привязка к Cloudflare как DNS-провайдеру.

## certbot (для тех, кто не любит acme.sh)

**Когда выбирать:**
- Сервер уже использует certbot для других сайтов.
- Хочется единого инструмента вместо двух (certbot + acme.sh).
- Не нужны wildcard-сертификаты.

**Как работает:**
1. Устанавливаем certbot (apt/dnf/yum).
2. Открываем 80, останавливаем nginx.
3. `certbot certonly --standalone -d $DOMAIN`.
4. Копируем cert+key в `/root/cert/$DOMAIN/` (унифицированный путь).
5. Привязываем к панели через SQLite.
6. Создаём cron / systemd-timer для `certbot renew` (если ещё не настроен).

**Плюсы:**
- Знакомый инструмент.
- Renew стандартный (`certbot.timer`).

**Минусы:**
- Apt-пакет certbot отстаёт от мейнлайна на полгода-год.
- Renew требует останова nginx (если используется standalone) — нужен
  pre/post hook.

## Сравнительная таблица

| Параметр | acme-standalone | acme-cloudflare | certbot |
|---|---|---|---|
| Требует CF API | ❌ | ✅ Global Key | ❌ |
| Останавливает 80 при выпуске | ✅ | ❌ | ✅ |
| Renew без касания 80 | ❌ | ✅ | ❌ |
| Wildcard-сертификаты | ❌ | ✅ | ✅ (с DNS-плагином) |
| Дефолт скилла | ✅ | — | — |

## Что НЕ работает в текущей версии 3X-UI

- DNS-01 challenge без Cloudflare через встроенный acme в 3X-UI —
  только Cloudflare встроен. Yandex DNS, Route53, hosting.ru —
  не поддерживаются встроенным механизмом. Решение: использовать
  `acme.sh` напрямую (вне 3X-UI) с любым DNS-провайдером, потом
  привязать cert через SQLite.

## Известные проблемы

- **Issue #4390** (см. `3x-ui-panel.md` §7.1) — встроенный `ssl_cert_issue_main`
  в `x-ui.sh` зацикливается при автоматизации с закрытым stdin. Наш скилл
  работает **снаружи** этого механизма (acme.sh напрямую → SQLite UPDATE) —
  не попадает в этот баг.

- **Сертификат на IP вместо домена** — возможен через `--ip-cert` опцию
  в 3X-UI (6-дневные shortlived от Let's Encrypt). Этот скилл его не
  поддерживает, потому что 6-дневный cert требует автоматического renew
  каждые ~4 дня — это отдельный паттерн, не подходит для простой панели.
