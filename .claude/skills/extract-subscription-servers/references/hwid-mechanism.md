# HWID-привязка подписки (как устроена и как с ней работать)

## Зачем провайдеры это делают

«Под ключ» провайдеры (Panterra, NurVPN и другие на панели **Remnawave**)
защищают список своих серверов **привязкой к устройству**. Идея: подписка
куплена на N устройств — и сервера видят только те N. Это мешает «расшарить
одну подписку на весь подъезд».

Технически устройство опознаётся по **HWID** (hardware ID — «отпечаток
устройства»): случайная hex-строка, которую фирменное приложение (Happ)
генерирует один раз и шлёт в каждом запросе заголовком `X-Hwid`.

## Что приходит, если HWID «не тот»

Сервер на запрос с незнакомым/чужим HWID (или вообще без него) отдаёт
**заглушку** вместо реального списка:

```
vless://00000000-0000-0000-0000-000000000000@0.0.0.0:1?...#App%20not%20supported
```

Подписка при этом **живая** — в личном кабинете виден расход трафика и срок.
Список просто спрятан за замком.

## Заголовки состояния (читает inspect-subscription.sh)

Сервер прямо сообщает в HTTP-заголовках ответа, что происходит:

| Заголовок | Состояние |
|---|---|
| `x-hwid-active: true` | механизм HWID включён вообще |
| `x-hwid-limit: true` | лимит устройств включён |
| `x-hwid-not-supported: true` | мы прислали НЕ-Happ клиент (или без Happ-заголовков) → провайдер вернул «App not supported» |
| `x-hwid-max-devices-reached: true` | Happ-заголовки есть, но наш HWID НЕ в списке устройств И свободных слотов нет |

> **`not_supported` ≠ `max_devices_reached`** — это РАЗНЫЕ диагнозы:
> - `not_supported` — лечится правильным набором заголовков Happ (это делает
>   `parse-subscription.sh` автоматически при заданном `HWID`).
> - `max_devices_reached` — лечится только **освобождением слота** в приложении/
>   боте провайдера (curl этого сделать не может — это действие пользователя в
>   личном кабинете).

## Заголовки расхода и срока

```
subscription-userinfo: upload=5368709120; download=10737418240; total=107374182400; expire=1782283875
```

- `upload`/`download` — байты (расход = upload + download, в ГБ).
- `total` — лимит трафика в байтах (`0` = безлимит).
- `expire` — Unix timestamp окончания подписки (`0` = бессрочно).

Плюс `profile-title: base64:...` (название) и `announce: base64:...` (сообщение
провайдера — там часто инструкция или ссылка на бот поддержки).

## Как добыть рабочий HWID — два пути (resolve-hwid.sh)

### Путь A — HWID из установленного Happ (ПРИОРИТЕТНЫЙ)

Если у оператора на маке стоит приложение **Happ** и подписка в нём УЖЕ
работает — берём его HWID из plist:

```bash
plutil -p "$HOME/Library/Group Containers/group.su.ffg.happ.plus/Library/Preferences/group.su.ffg.happ.plus.plist" | grep -i hwid
```

Это устройство уже зарегистрировано → **слот НЕ тратим**, переиспользуем
существующую регистрацию. Это путь, который сработал в боевом прогоне — он
отдал полный конфиг.

### Путь B — стабильный сгенерированный HWID

Если Happ не установлен — генерируем свой стабильный HWID и сохраняем рядом
с подпиской в infra (`.hwid-<provider>`). При первом запросе с ним провайдер
(если есть свободный слот) привяжет его к «устройству-серверу». «Стабильный» =
перезапуски агента переиспользуют тот же HWID и НЕ жгут новые слоты.

`resolve-hwid.sh` сам выбирает: сначала A (Happ), иначе B. `MODE=happ` /
`MODE=generate` форсируют путь.

## Освобождение слота — это действие ПОЛЬЗОВАТЕЛЯ, не curl

Если все слоты заняты (`max_devices_reached`) — добавить новое устройство
нельзя, пока не освободишь старое. **Отдельного HTTP-эндпоинта регистрации/
удаления НЕТ** (проверено: `/register`, `/device` → 502). Слоты управляются
ТОЛЬКО через приложение или Telegram-бот провайдера:

> Приложение провайдера (или его бот) → «Мои устройства» / «Управление
> устройствами» → удалить одно ненужное.

Поэтому в скилле этот шаг — просьба к пользователю сделать это руками, а не
попытка автоматизировать.

## Гео-блокировка поверх HWID

Некоторые (NurVPN) дополнительно отдают `502`/заглушку на **не-РФ IP**. Если
HWID правильный, а ответ всё равно пустой — делать запрос с РФ-хоста.

## Failed attempts (зафиксированные грабли)

- **Трактовать `max_devices_reached` как «тупик, только через бота»** —
  НЕВЕРНО. Рабочий путь — HWID оператора из установленного Happ (путь A):
  устройство уже в списке, заглушки нет, сервера приходят. Бот нужен только
  если слот реально надо освободить.
- **Декодировать Xray-JSON как base64** — даёт мусор и ложный «0 серверов».
  Тело Xray-подписки — это JSON, а не base64. `parse-subscription.sh` сначала
  проверяет `jq empty`, и только не-JSON пробует декодировать.
- **Искать эндпоинт регистрации HWID** (`/register`, `/device`) — его нет
  (502). Регистрация = первый запрос с новым HWID при свободном слоте.
