# Реестр секретов

**Менеджер:** <vault-name> (Keychain / pass / KeePassXC / Bitwarden — заменить на конкретный)
**Принцип:** имена секретов — указатели в `.env.example`, реальные значения только в менеджере и на runtime сервера.

Этот файл — индекс «где что лежит», без самих значений. Безопасно коммитится в git.

## Формат записи

| Имя              | Тип            | Где хранится                  | Последняя ротация | Кому доступно |
| ---------------- | -------------- | ----------------------------- | ----------------- | ------------- |
| `<name>`         | `<type>`       | `<vault>://<path>`            | `YYYY-MM-DD`      | оператор / команда |

**Типы:**
- DB password — пароли к БД.
- API token — токены к внешним API (Telegram, OpenAI, Stripe, ...).
- TLS cert — сертификаты (если хранятся отдельно от acme).
- SSH key — приватные ключи (если есть в менеджере).
- Basic auth — htpasswd-пароли для nginx.
- Webhook secret — секреты вебхуков.
- OAuth secret — client_id/client_secret для OAuth-приложений.

## Ротация — расписание

- **DB passwords:** ежеквартально (или сразу при подозрении на утечку).
- **API tokens:** при подозрении на утечку или раз в 6 месяцев. Telegram-боты — ежегодно (через @BotFather → revoke).
- **SSH ключи:** ежегодно. При смене оператора в команде — немедленно.
- **TLS certificates:** автоматически через acme.sh / certbot (90 дней).
- **OAuth secrets:** при подозрении на утечку или при ротации производителя (Google, GitHub, ...).

## Принцип использования в `.env.example`

В репозитории коммитим `.env.example` со ссылками на vault:

```
# .env.example — коммитим в git
DB_PASSWORD=<vault>://infra/db/postgres/main
TELEGRAM_BOT_TOKEN=<vault>://infra/telegram/bot-main
NGINX_HTPASSWD=<vault>://infra/nginx/htpasswd
```

На сервере существует реальный `.env` с подставленными значениями — он в `.gitignore`.

При деплое — отдельный скрипт читает значения из менеджера и подставляет в runtime-окружение (или генерирует `.env` на сервере перед `docker compose up`).

## Что НЕ записываем в этот файл

- Сами значения секретов (никогда!).
- Master-пароль от менеджера.
- Полные приватные ключи.
- Recovery codes 2FA.

## Что обновляем

- При добавлении нового секрета — новая строка в таблице + коммит.
- При ротации — обновить дату «Последняя ротация».
- При удалении сервиса — пометить запись как `[DEPRECATED YYYY-MM-DD]` или удалить (если точно не нужно).
