# Evals для /setup-secrets-vault

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «настрой менеджер паролей»
- «куда хранить секреты»
- «не хочу секреты в репе»
- «.env по правилам»
- «secrets vault»
- «password manager»
- «настрой Keychain под мои нужды»
- «KeePassXC для проекта»
- «pass / Bitwarden — что выбрать»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «ротировать пароль БД» — это `/rotate-secrets` (обновление существующего), не первичная настройка vault
- «секрет утёк, что делать» — комбинация: сначала `/rotate-secrets` (новый секрет), потом ревизия — но не setup-secrets-vault
- «настрой 1Password в команду» — корпоративный менеджер, не покрывается скиллом (он про индивидуальные) → ответ персоны: «1Password Teams — другой жанр, помогу ad-hoc»
- «поставь LastPass» — миграция между менеджерами, не setup → ответ персоны: «это миграция, помогу ad-hoc»
- «удали все секреты из vault» — обратная операция, опасна → ответ персоны с предупреждением и явным отказом активировать что-то без понимания зачем
- «настрой агента» — это `/sysadmin-init` (агент в целом), не vault → редирект

## Edge-cases (опционально)

- «у меня уже есть Keychain — добавь туда секреты для проекта» — пограничная: vault уже настроен, нужно только индекс в `inventory/access.md` → активировать в режиме «vault уже есть, делаю только конвенции и шаблон индекса»
- «секреты в .env прямо в репе, давай это исправим» — типичная ситуация (особенно после миграции с другого проекта) → активировать setup-secrets-vault + параллельно очистка git-истории (отдельный шаг)
- «нужен менеджер паролей и для команды, и для проекта» — пограничная: индивидуальный + корпоративный — это два решения → ответ персоны с уточнением «для тебя одного — выбираем из Keychain/pass/KeePassXC/Bitwarden. Команде — нужен 1Password/Bitwarden Teams, отдельная задача»
