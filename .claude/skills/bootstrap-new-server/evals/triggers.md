# Evals для /bootstrap-new-server

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «настрой новый сервер»
- «купил VPS, нужно настроить с нуля»
- «свежий ubuntu, что делать дальше»
- «bootstrap server»
- «подними чистый сервер»
- «получил VPS у Hetzner, дальше что»
- «хочу как у тебя на проде, только мне»
- «поставь SSH-ключи, файрвол, докер на свежий сервер»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «бардак на сервере, наведи порядок» — это работающий сервер, не свежий → `/cleanup-existing-server`
- «у меня есть сервер от предыдущего админа, разберись» — наследие, не bootstrap → `/cleanup-existing-server` (+ возможно `/inventory-scan` сначала)
- «обнови мой сервер» — обновление уже настроенного, не первичная настройка → `/deploy-service` (если про сервис) или ad-hoc `apt upgrade`
- «защити сервер» — слишком абстрактно → уточнить: свежий (bootstrap) или работающий (audit-security)
- «настрой docker» — узкая задача, входит в bootstrap, но если сервер уже настроен — ad-hoc установка docker без полного bootstrap
- «переезжаю на новый сервер» — это `/migrate-server-to-server`, не bootstrap (хотя целевой сервер сначала bootstrap)

## Edge-cases (опционально)

- «у меня свежий сервер, но я уже руками поставил docker» — пограничная: bootstrap не идемпотентный полностью, могут быть конфликты → уточнить: «откатить твои изменения и пройти весь bootstrap или дополнить точечно?»
- «настрой второй сервер по такому же шаблону» — это bootstrap, но с привязкой к существующему inventory → активировать, но проверить что новый сервер уже в `sysadmin-config.json` (если нет — добавить через `/sysadmin-init --reconfigure` сначала)
- «переезжаю — настрой новый сервер для миграции» — здесь сначала /bootstrap-new-server, потом /migrate-server-to-server → можно активировать bootstrap, но в финале явно отправить на /migrate
