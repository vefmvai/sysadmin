# socks5:// vs socks5h:// — почему буква `h` критична

Один символ в URI прокси меняет поведение библиотеки. Этот документ — почему
скилл `/setup-server-proxy` всегда пишет `socks5h://` в `/etc/environment`.

## TL;DR

- **`socks5://`** — клиент сначала резолвит домен в IP **локально**, потом
  отправляет уже IP на прокси.
- **`socks5h://`** — клиент отдаёт прокси **сам домен** (как строку), прокси
  резолвит и идёт по IP.

Буква `h` = `hostname`.

## Почему это важно для сценария «обход блокировок»

### Проблема DNS-leak

При `socks5://` (без `h`):

```
[Программа на сервере]
   ↓ resolve api.anthropic.com через локальный DNS (8.8.8.8 / провайдер)
   ↓ получает реальный IP — например, 160.79.104.10
   ↓ открывает SOCKS5-соединение на прокси с этим IP
[Прокси] → пересылает к 160.79.104.10
```

**Минусы:**
1. **DNS-leak**: DNS-провайдер видит, что ты резолвишь `api.anthropic.com`.
   Если DNS — российский (Yandex DNS, локальный provider) — этот резолв
   попадает в логи.
2. **Блокировки по DNS**: если провайдер блокирует `api.anthropic.com` на
   уровне DNS (DPI-фильтр на 53 порту), резолв вернёт `NXDOMAIN` или
   неправильный IP. Прокся получит мусор.
3. **Геолокация на ответе API**: для `socks5://` IP исходит из локальной
   сети сервера, но **домен ресолвен локально**. Если CDN отдаёт разные IP
   разным регионам — программа получит «русский» IP CDN и потом будет
   ходить туда через прокси.

При `socks5h://` (с `h`):

```
[Программа на сервере]
   ↓ отдаёт прокси строку "api.anthropic.com"
[Прокси] → resolve через свой DNS-резолвер (например, DNS upstream-сервера)
         → получает зарубежный IP (без блокировок)
         → пересылает соединение
```

**Плюсы:**
1. **Никакого DNS-leak** — локальный DNS вообще не вовлечён.
2. **Резолв через upstream**: прокси резолвит через свой DNS, который
   обычно за пределами РФ.
3. **Правильная геолокация**: CDN видит запрос с IP прокси, отдаёт
   правильный «зарубежный» IP.

## Конкретный кейс: Anthropic API возвращает 403 без `h`

**Симптом:** на сервере в РФ ставишь прокси через 3X-UI, прописываешь
`https_proxy=socks5://127.0.0.1:1080` (без `h`), пытаешься обратиться к
`api.anthropic.com`:

```bash
$ curl -I https://api.anthropic.com
HTTP/2 403
```

**Причина:** anthropic SDK резолвит `api.anthropic.com` через локальный
DNS, получает IP CDN (Cloudflare/AWS). Этот IP в whitelist Anthropic для
"геолокации RU" → API возвращает 403 «not available in your region».

**Решение:** одна буква:
```
https_proxy=socks5h://127.0.0.1:1080
```

Теперь резолв идёт через прокси-узел (за границей) → получает «зарубежный»
IP → Anthropic думает «это пользователь из Германии» → 200/404.

## Какие библиотеки поддерживают `socks5h://`

| Клиент / библиотека | Поддерживает `socks5h://` |
|---|---|
| `curl` 7.x+ | ✅ |
| Python `requests` | ✅ через PySocks |
| Python `urllib3` | ✅ через PySocks |
| Python `httpx` | ✅ |
| Python `aiohttp` | ✅ через `aiohttp-socks` |
| `git` | ⚠️ только через `http.proxy` config, не env |
| `wget` | ❌ — только HTTP-прокси |
| Node.js `https-proxy-agent` | ❌ — это HTTP-прокси |
| Node.js `socks-proxy-agent` | ✅ |
| Go stdlib `net/http` | ❌ — только HTTP/HTTPS-прокси, для SOCKS5 нужен `golang.org/x/net/proxy` |
| Go (через 3X-UI) | ✅ потому что Go-прокси клиент в xray использует свой стек |

## А если программа не поддерживает SOCKS5?

Решение — поднять локальный HTTP-прокси (Privoxy) поверх SOCKS5:

```
[Программа] → HTTP://127.0.0.1:8118 (privoxy)
                ↓
[Privoxy] → SOCKS5H://127.0.0.1:1080 (xray)
                ↓
[Xray] → upstream VPN
```

Конфиг Privoxy:
```
# /etc/privoxy/config
forward-socks5t / 127.0.0.1:1080 .
listen-address 127.0.0.1:8118
```

После этого можно использовать `http://127.0.0.1:8118` как прокси даже для
программ без SOCKS5-поддержки (apt, старые версии npm и т.д.).

Этот скилл **не настраивает Privoxy** — это отдельная задача, не входит
в скоуп. Достаточно socks5h для основных программ.

## Откуда буква `h`?

Историческое: SOCKS5 спецификация (RFC 1928) сразу поддерживала и IP,
и hostname в полях запроса. Но многие реализации клиентов (Python urllib2,
старые curl) **передавали только IP**, делая локальный DNS-резолв.

Когда стало понятно, что это даёт DNS-leak, был добавлен модификатор `h`
в URI scheme. Это **расширение URI-схемы curl** (документировано в curl
docs), стало де-факто стандартом для всех Python-HTTP-библиотек.
