# Evals для /cleanup-existing-server

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «бардак на сервере, наведи порядок»
- «контейнеры с дикими именами, переименуй»
- «приведи в порядок без простоя»
- «cleanup server»
- «достался сервер от предыдущего админа»
- «дрифт между sites-available и sites-enabled, разберись»
- «.env файлы с правами 0644 — поправь»
- «мёртвые nginx-конфиги остались, удали»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «настрой свежий VPS» — это `/bootstrap-new-server`, не cleanup
- «удали БД test-app» — это red zone, drop-volume — НЕ для cleanup (cleanup yellow) → ad-hoc red-zone процедура с type-to-confirm
- «удали все старые volume» — потенциально деструктивно, выходит за scope cleanup → ответ персоны: cleanup сюда не лезет, нужно явное решение по каждому volume
- «обнови все контейнеры до последней версии» — это `/deploy-service` (или ad-hoc), не cleanup
- «проверь безопасность сервера» — это `/audit-security`, не cleanup (хотя cleanup может попутно поднять security-замечания)

## Edge-cases (опционально)

- «у меня контейнеры с непонятными именами, что с ними делать» — пограничная: cleanup умеет переименовать через aliases, но если оператор не уверен что эти контейнеры вообще нужны — сначала `/inventory-scan` для понимания, потом cleanup
- «новый сервер для меня, но я там уже что-то наставил» — пограничная между bootstrap и cleanup → уточнить: «если меньше дня назад и не критично — откатить и сделать bootstrap. Если уже работает прод — cleanup»
- «помоги навести порядок в nginx» — узкая задача, входит в cleanup, активировать в режиме focused (только nginx-категория)
