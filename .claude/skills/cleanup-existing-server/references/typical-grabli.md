# Cleanup — типичные грабли

Памятка для скилла `cleanup-existing-server`. Каталог типовых паттернов
поломок при наведении порядка на «обжитой» сервер: симптом → причина →
лечение. Когда натыкаешься на похожий симптом — сначала смотри сюда.

---

## 1. container_name vs service key — двухстадийность недостаточна без правки service key

**Симптом:** переименовали `container_name`, alias добавили, но в DNS Docker
старое имя всё ещё резолвится.

**Причина:** Docker Compose автоматически создаёт network alias = service key
compose-проекта (не container_name). Если service key не менять, старое имя
останется в DNS.

**Лечение:** Stage 2 переименования включает правку И container_name, И service
key. Если только container_name — DNS не очистится.

**Типичный кейс:** переименование `<old-app>` → `<new-app>` без правки service key —
старое имя продолжает отвечать в DNS Docker даже после `docker compose up -d`.
Лечение — править service key одной правкой compose-файла, не двумя проходами.

---

## 2. Compose без `image:` — переименование триггерит пересборку

**Симптом:** при правке compose-файла, где есть только `build:` без явного
`image:`, выполнение `docker compose up -d --no-deps <new-name>` запускает
**полную** пересборку Docker-образа из Dockerfile (несколько минут простоя).

**Причина:** auto-name образа формируется как `<project>-<service-key>`. Когда
service key меняется, Docker Compose думает, что образа не существует, и
запускает build.

**Лечение:** **до** переименования добавить в compose-файл:

```yaml
services:
  <new-service-key>:
    image: <project>-<old-service-key>:latest  # реальный existing image
    build: ...
```

Тогда Docker Compose увидит готовый image и не будет билдить.

**Реальная цена ошибки:** одно переименование service key в compose-файле без
явного `image:` приводило к простою порядка 8 минут — Docker Compose
запускал full rebuild Dockerfile-образа.

## 3. mem_limit без замеров → OOM kill

**Симптом:** добавили `mem_limit: 512m` в compose, контейнер стал
рестартоваться (`Status: Restarting (137)`).

**Причина:** exit code 137 = SIGKILL от OOM killer. Лимит был меньше реального
потребления.

**Лечение:** замеры **до** установки лимита, минимум 24 часа на разное время
суток. Лимит = `peak_observed × 1.5`. Если есть Beszel — берём оттуда.

**Реальный кейс:** postgres-контейнер получил `mem_limit: 768m` при
наблюдаемом пике `1100m` — рестартовался по OOM. Подняли до `1024m`.
В ADR 0009 проекта-носителя (или эквивалентном решении) фиксируется
исключение из правила «суммарный лимит ≤ 3.2 ГБ при 4 ГБ хоста» — БД
имеет приоритет.

## 4. Drift sites-available vs sites-enabled

**Симптом:** правишь конфиг в `sites-available`, делаешь `nginx -s reload`,
но изменения не применяются.

**Причина:** в `sites-enabled` лежит **регулярный файл** (копия), а не симлинк.
nginx читает копию.

**Лечение:**

```bash
[ -L /etc/nginx/sites-enabled/$NAME ] || {
  mv /etc/nginx/sites-enabled/$NAME /tmp/$NAME.bak
  ln -s /etc/nginx/sites-available/$NAME /etc/nginx/sites-enabled/$NAME
  nginx -t && nginx -s reload
}
```

**Типичный кейс:** `sites-enabled/<example.com>.conf` оказался регулярным
файлом со старыми путями certbot/acme — поправили `sites-available/...`,
а nginx продолжал отвечать со старого конфига в `sites-enabled/...`.

---

## 5. Удалённый nginx server-блок без `nginx -t`

**Симптом:** удалил мёртвый server-блок, сделал `nginx -s reload`, **весь**
nginx упал.

**Причина:** забытый `;` или непарная `}` в соседнем server-блоке. nginx -s
reload требует валидный конфиг **целиком**.

**Лечение:** `nginx -t` ВСЕГДА перед reload. Без исключений.

```bash
mv /etc/nginx/sites-available/$DEAD /tmp/$DEAD.dead-$(date +%s)
nginx -t || {
  echo "ОТКАТ"
  mv /tmp/$DEAD.dead-* /etc/nginx/sites-available/$DEAD
  exit 1
}
nginx -s reload
```

## 6. .env с mode 0644 — секреты читаются всеми пользователями

**Симптом:** `find /opt -name '.env' -perm 644` нашёл несколько файлов.

**Причина:** при копировании `.env.example` → `.env` забыли `chmod 600`.

**Лечение:** backup + chmod:

```bash
for f in $(find /opt -name '.env' -perm 644); do
  cp -p "$f" "$f.bak.$(date +%s)"
  chmod 600 "$f"
done
```

**Типичный кейс:** на «обжитой» инфре регулярно находятся 3-5 `.env` файлов
с mode `0644` — недосмотр при копировании `.env.example` → `.env`. После
`chmod 600` обычно ничего не ломается: процессы работают из-под того же
user, что владеет файлом, mode `0600` их устраивает.

**Граничный случай:** сервис запущен из-под другого UID — после chmod 600
не сможет прочитать. Проверить владельца до chmod:

```bash
stat -c '%U:%G' /opt/<service>/.env
docker inspect <container> --format '{{.Config.User}}'
```

Если UID не совпадает — отдельная задача (chown или права 0640 + group).

## 7. Один service key в compose — два контейнера в одной сети

**Симптом:** `docker network inspect` показывает 2 контейнера с одинаковым
alias.

**Причина:** compose-проект пересоздавал контейнер (`up -d --force-recreate`),
старый не удалился (например, потому что в `paused` state) — оба остались.

**Лечение:**

```bash
docker compose down --remove-orphans
docker compose up -d
```

**Типичный кейс:** после смены service key в compose остаётся orphan-контейнер
со старым именем — флаг `--remove-orphans` для `docker compose down`/`up` обязателен.

## 8. find -mtime +N округляет вниз

**Симптом:** `find /tmp -mtime +1 -delete` пропускает файлы, созданные «вчера»,
если им меньше 48 часов.

**Причина:** `-mtime +N` означает «строго старше N полных дней». Файл возрастом
30 часов имеет mtime = 1 полный день, не «больше 1».

**Лечение:** для очистки по дате — sortировка по имени (если файлы датированы)
или `-mtime +0` (всё что старше дня) с пониманием, что это текущий день не
тронет.

**Типичный кейс:** ожидаешь удаления safety-tarball'ов с возрастом ≥1 день —
они остаются. Возрасту 36 часов соответствует `-mtime 1` (один полный день),
а `-mtime +1` отбирает только то, что старше 2 полных дней.

## 9. Переименование БД-контейнера — особый случай

**Симптом:** все клиенты БД одновременно потеряли соединение на ~30 секунд.

**Причина:** в БД-контейнер ходят 5+ клиентов через alias. Single-stage rename
пересоздаёт контейнер — все клиенты видят connection refused.

**Лечение:** обязательный two-stage паттерн (ADR 0011 проекта-носителя
или эквивалент):
- Stage 1: добавить новый alias **без** перезапуска
- Stage 2: переименовать container_name + service key, **оставить старый alias
  переходным окном**
- Stage 3: убрать старый alias только когда все клиенты переключатся

Окно простоя сокращается с 30 секунд до 10 секунд (только Stage 2 рестарт).

**Реальный замер:** переименование БД-контейнера `<old-pg-name>` → `postgres`
через two-stage у проекта-носителя прошло с простоем ~12 секунд для 5 клиентов.
До этого single-stage клал тех же клиентов на ~30 секунд.

## 10. internal: true ломает publish-port

**Симптом:** контейнер в сети `internal: true` имеет `ports: [127.0.0.1:5432:5432]`,
но порт не отвечает на хосте.

**Причина:** `internal: true` запрещает любой egress через NAT, включая DNAT
для publish-портов (это design Docker, не баг).

**Лечение:** подключить контейнер к **двум** сетям — `internal: true` (для
изоляции egress) + не-internal (для publish). Паттерн зафиксирован в
ADR 0012 проекта-носителя (или эквивалентном решении): БД-контейнер в
`data` (internal) + `services` (non-internal). Для cleanup это значит:
если кандидат на cleanup использует `internal: true` и publish-port — не
пытайся «упростить», это рабочая конфигурация.

## 11. Brief 6 пунктов — не «формальность», а защита

**Симптом:** оператор устал, согласовал брифинг быстро, в выполнении выяснилось,
что в пункте «откат» команды не верны.

**Причина:** брифинг должен быть **проверяемым**. Если команды отката не
тестировались хотя бы в голове — это не брифинг, это вид деятельности.

**Лечение:** перед предъявлением брифинга — самопроверка:
- Откат запустится прямо сейчас? Файл, на который ссылаемся, существует?
- Страховка действительно есть? `restic snapshots --latest 1` в последние сутки?
- Проверка работает? Команда возвращает что-то осмысленное?

Если на любой вопрос «не уверен» — НЕ предъявлять брифинг, сначала уточнить.
