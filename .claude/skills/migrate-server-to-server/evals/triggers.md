# Evals для /migrate-server-to-server

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «переезд на новый сервер»
- «migrate server»
- «сменить VPS»
- «переместить инфру»
- «у меня новый сервер у Hetzner, перенеси с старого Selectel»
- «сменить хостинг»
- «апгрейд железа — переехать на сервер побольше»
- «нулевой даунтайм — переезжаю с одного сервера на другой»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «настрой свежий VPS» — это `/bootstrap-new-server`, не migrate
- «навести порядок на сервере» — это `/cleanup-existing-server`, не миграция
- «обнови сервис» — это `/deploy-service`, не миграция
- «бэкап-restore» — backup-restore это одна из стратегий migrate, но узкая фраза → ответ персоны с уточнением: «это часть переезда или standalone — у тебя сломался сервер и нужно восстановить из бэкапа?»
- «скопировать БД на другой сервер» — частный случай, не полный migrate → ответ персоны: либо migrate с фокусом на БД, либо ad-hoc через `pg_dump` если речь только об одной БД
- «копия сервера для теста» — это про staging, не миграция продакшена → ответ персоны с уточнением

## Edge-cases (опционально)

- «новый сервер уже подготовлен, переноси» — пограничная: нужно подтверждение что target прошёл `/bootstrap-new-server` → активировать, но в pre-check проверить готовность target
- «миграция, но БД оставить на старом сервере» — частичная миграция → активировать, но с явным предупреждением что это редкий сценарий и придётся вручную решить вопрос сетевого доступа между серверами
- «переезжаю с k8s на docker-compose» — миграция стека, не сервер-к-серверу → отказ скилла, ответ персоны: «это другой жанр, не покрывается migrate-server-to-server»
