# Evals для /deploy-service

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «задеплой новый сервис»
- «накати обновление»
- «обнови контейнер до последней версии»
- «deploy»
- «выкатить версию 1.2.3»
- «выкати релиз»
- «откати последний деплой»
- «rollback»
- «git push, потом обнови на сервере»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «удали контейнер test-bot» — это red zone (необратимое удаление), не deploy → ad-hoc red-zone процедура
- «обнови систему» — это про `apt upgrade`, не про сервис → ad-hoc Yellow Zone
- «настрой docker-compose с нуля» — это первичная настройка, входит в `/bootstrap-new-server`, не deploy
- «исправь бага в моём коде» — это код приложения, не инфра → ответ персоны: «продукт — твой, я только инфра»
- «выкати без проверок, прод горит» — попытка обойти pre-check → отказ персоны (см. конституцию: «безопасность важнее скорости»)
- «деплой в обход git» — нарушение паттерна push-to-pull → отказ + редирект на аварийный runbook

## Edge-cases (опционально)

- «обнови мониторинг-стек» — пограничная: мониторинг ставится через `/install-monitoring-stack`, но обновление существующего — это deploy → активировать deploy в режиме update
- «накати фикс прямо сейчас» — пограничная: если есть git push и стандартный compose — deploy. Если хак мимо процедуры — отказ
- «выкатил руками вчера, теперь хочу как положено» — нужен бутстрап git+compose, а не сразу deploy → ответ персоны: «сначала зафиксировать существующее состояние в репе, потом deploy с MODE=update»
