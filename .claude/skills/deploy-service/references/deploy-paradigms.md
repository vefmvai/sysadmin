# Парадигмы деплоя — почему push-to-pull, а не rsync или CI/CD

Документ для того, кто видит SKILL.md впервые и хочет понять «почему не CI/CD как у всех».
Краткий разбор трёх альтернатив и причин выбора. Полное обоснование — в ADR 0015
проекта-носителя (`decisions/0015-iac-deploy-model.md`) или эквивалентном решении.

## 1. rsync + ручные правки на сервере (антипаттерн)

**Как это выглядит:**
```bash
# на маке
rsync -avz services/ "$SERVER":/opt/services/
ssh "$SERVER" "cd /opt/services/my-app && docker compose up -d"
```

**Что плохо:**

- **Drift между сервером и репо.** Через неделю после первого rsync кто-то правит compose
  напрямую на сервере (быстро же!). На маке репо отстаёт. Через месяц никто не помнит,
  где «правда».
- **DR-runbook теряет смысл.** Если сервер упал — что восстанавливать? Репо устаревший,
  сервер мёртв. Источника истины нет.
- **`rsync --delete` опасен.** Один забытый `--delete` убивает локальные файлы на сервере
  (логи, .env, snapshots).
- **Никакой истории «когда и что катили».** Только git хранит хронологию изменений.

**Типичный сценарий за полгода эксплуатации:** несколько prod-сервисов
используют rsync-deploy → постепенно копятся правки прямо на сервере → drift в файлах
compose / nginx / .env → неделя на обратную синхронизацию репо ↔ сервер.

## 2. Cron-автодеплой (отвергнут в ADR 0015 проекта-носителя; типично отвергается в любом соло-проекте)

**Как это выглядит:**
```bash
# crontab на сервере
*/5 * * * * cd /opt/infra && git pull && docker compose up -d
```

**Что плохо:**

- **Автокатит без паузы на ревью.** Между `git push` и применением — 0 секунд возможности
  остановиться, если ты понял, что закоммитил мусор.
- **Ломает прод между коммитами.** Если за минуту сделал 3 push'а (правил ошибку → ещё ошибка
  → починил), cron подхватит каждый — три рестарта подряд.
- **Нет pre-flight проверки.** Локальный `main` мог не быть синхронизирован с `origin/main`
  (ты забыл push) — cron всё равно сработает на старом состоянии репо на сервере.
- **Селективный up -d сложно реализовать.** Cron либо рестартит всё (опасно), либо нужна
  логика diff'а HEAD..origin/main → таблица сервисов (это уже сложность CI/CD).

## 3. CI/CD через GitHub Actions (отложено в проекте-носителе на v1.x — у вас может быть актуально раньше)

**Как это выглядит:**
```yaml
# .github/workflows/deploy.yml
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SSH_HOST }}
          key: ${{ secrets.SSH_KEY }}
          script: cd /opt/infra && git pull && docker compose up -d
```

**Почему обычно избыточно для соло-оператора в первой версии инфры:**

- **Ручной `./deploy-remote.sh` создаёт паузу для ревью.** Видишь что катишь, до того как
  нажмёшь скрипт. Для соло-оператора это безопаснее — снижается шанс «закоммитил → запушил →
  заметил ошибку через 2 минуты, а уже катится в проде».
- **Дополнительная сложность для одного человека.** Secrets в Actions, OIDC vs static keys,
  окна обслуживания, retry-логика, нотификации — много инфраструктуры для одного оператора.
- **GitHub Actions runtime платный** (после free tier) — для соло-оператора с десятком пушей
  в неделю это лишний расход без явной выгоды.
- **Ошибки в Actions-пайплайне не роняют прод автоматически** (если Action упал — деплой не
  применится), но ты потеряешь несколько минут на разбор «почему не задеплоилось» вместо
  того, чтобы видеть результат сразу при ручном запуске.

**Когда CI/CD действительно становится нужен:**

- Появится второй оператор → ручной запуск нелогичен («чей сейчас был push?»)
- Появится staging-сервер → нужны окружения, secrets per env
- Появится больше 2-3 push'ей в день → ручные триггеры превращаются в рутину

В этом случае `deploy-remote.sh` останется как ручной escape hatch (например, для отката
в выходной без запуска Actions), а основной поток уйдёт в Actions.

## 4. Push-to-pull manual deploy (выбрано в проекте-носителе — ADR 0015; для соло-оператора это разумный default)

**Как это выглядит:**

```bash
# на маке
cd <repo>
git add .
git commit -m "feat(my-app): bump to v1.5.2"
git push origin main
./scripts/deploy/deploy-remote.sh    # триггер
```

`deploy-remote.sh` делает:
1. **Pre-flight:** `git fetch && rev-parse main vs origin/main` — если local != origin, ошибка.
2. **SSH-триггер:** `ssh "$SERVER" "cd $INFRA_DIR && ./deploy.sh"`.

Серверный `deploy.sh` (тоже в репо) делает:
1. `git pull --ff-only` — никаких merge-конфликтов на сервере (если они возможны → кто-то
   правил вручную, аварийный режим).
2. **Селективный up -d** — diff между старым и новым HEAD по списку `services/*`. Только
   изменённые сервисы рестартятся.

**Что хорошо:**

- **Источник истины — git.** Текущее состояние репо = текущее состояние сервера.
  `git status` на сервере после деплоя должен быть чистым; не чистый → drift.
- **Откат коммитом.** `git revert <bad-sha> && git push && ./deploy-remote.sh` — за <5 мин.
- **Осмысленная история.** Каждый коммит — деплой. Можно `git log --since="1 week"` посмотреть
  что катили за неделю.
- **Пауза на ревью.** Между `git push` и `./deploy-remote.sh` — твой выбор. Передумал — `git revert`.
- **Унификация с приложенческими репо.** Тот же паттерн (`git push` → `deploy-remote.sh`)
  используется и для приложенческих сервисов в их собственных репозиториях — оператор учит
  одну схему, применяет везде.
- **DR-runbook ясный.** Шаг «клонировать `git@github.com:<your-org>/infra.git` в `$INFRA_DIR`,
  выполнить `./deploy.sh`» — работает в чистой среде из коробки.

**Что не идеально:**

- Серверный `.env` с секретами не в git (восстанавливается вручную при DR через
  менеджер паролей оператора). Закрывается в v1.x через SOPS / age — `.env.enc` в git,
  расшифровка в pipeline.
- Ручной шаг `./deploy-remote.sh` — не автомат. Компромисс осознанный.
- Компрометация рабочей машины → компрометация прода (через git-push-key + SSH-key
  к серверу). Mitigation: FDE (FileVault / LUKS), ключи в менеджере паролей,
  2FA на git-провайдере.

## Сводная таблица

| Парадигма | Drift-резистентность | Откат | Пауза на ревью | Сложность | Применимо |
|-----------|----------------------|-------|----------------|-----------|-----------|
| rsync + ручные правки | низкая | ручной | да | низкая | антипаттерн |
| cron автодеплой | средняя | через git revert | нет | средняя | отвергнуто |
| GitHub Actions CI/CD | высокая | через git revert | нет | высокая | команда / staging |
| **push-to-pull manual** | **высокая** | **через git revert** | **да** | **низкая** | **соло-оператор v1** |

Когда появится второй оператор или staging-сервер — пересмотр в пользу Actions
оправдан. До этого момента: одна модель деплоя для всех инфраструктурных стеков.
