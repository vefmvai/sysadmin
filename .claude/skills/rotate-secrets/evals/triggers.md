# Evals для /rotate-secrets

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «ротировать секрет»
- «сменить пароль БД»
- «rotate password»
- «обновить SSH-ключ»
- «secret expired»
- «плановая ротация»
- «токен утёк — ротировать срочно»
- «compromise — сменить все ключи»
- «обнови TLS-cert»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «настрой менеджер паролей» — это `/setup-secrets-vault` (первичная настройка), не ротация
- «куда хранить пароль» — это `/setup-secrets-vault`, не ротация
- «удали старый токен» — без замены = деструктив без процедуры → ответ персоны: «ротация — новый сначала, потом удаление старого. Просто удалить — отказ»
- «мигрировать с LastPass на Bitwarden» — миграция между менеджерами, не покрывается → ответ персоны: «это другой жанр, помогу ad-hoc»
- «забыл root-пароль» — это про восстановление, не ротация → ответ персоны: «ротация требует знать старый. Восстановление — отдельная процедура, опасная»
- «обнови все секреты разом» — ротация делается по одному секрету за раз для атомарной подмены → ответ персоны с предложением составить план ротаций по очереди

## Edge-cases (опционально)

- «сменить пароль и сразу удалить старый» — это нормальный rotate-flow (новый → подмена → старый удалить) → активировать с полной 4-шаговой процедурой
- «ротация без даунтайма» — для большинства секретов так и делается атомарно, но для DB-пароля может быть короткое окно при reload connection-pool → активировать, в брифинге явно показать ожидаемое окно
- «нужно сменить пароль БД, потому что прод-сервис о нём знает в трёх местах» — расширенный rotate → активировать, в pre-check собрать все места использования
