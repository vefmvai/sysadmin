# Плановое расписание ротации секретов

Базовое расписание для регулярной ротации секретов проекта-носителя. Конкретные
даты следующих ротаций живут в `inventory/shared/access.md`.

---

## Сводная таблица

| Тип секрета | Период плановой ротации | Триггер внеплановой |
|-------------|------------------------|---------------------|
| **DB password** (PostgreSQL/MySQL/Redis) | Ежеквартально (90 дней) | Подозрение на утечку, увольнение оператора с доступом |
| **API token** (Telegram, GitHub, Yandex OAuth, Cloudflare) | Раз в 6 месяцев (180 дней) | Утечка в публичный код, увольнение |
| **SSH-ключ** (deploy keys, операторские ключи) | Ежегодно (365 дней) | Утечка приватной части, увольнение |
| **TLS-сертификат** | Автоматически (Let's Encrypt 90 дней, обновление за 30 дней до истечения через acme.sh) | Compromise приватного ключа |
| **OAuth secret** (client secrets для Google/Yandex/etc) | По требованию провайдера, минимум ежегодно | Утечка, провайдер требует |
| **Passkey / WebAuthn credential** | Не ротируется планово (per-device) | Потеря устройства, увольнение |

---

## Обоснование периодов

### DB password — 90 дней

Компромисс между двумя крайностями:

- **Слишком часто (< 30 дней):** оператор устаёт, ротация становится ритуалом
  без внимания, риск ошибки растёт.
- **Слишком редко (> 365 дней):** при компрометации (которую могли не заметить)
  атакующий имеет долгое окно.

90 дней — индустриальный стандарт для сервисов с публичным интерфейсом, балансирует
оба риска. Для критичных систем (банкинг, платёжные шлюзы) — 30-60 дней.

### API token — 180 дней

API tokens обычно используются с ограниченными scope'ами (например, только read,
только конкретный bot). Утечка одного токена менее критична, чем утечка DB
password (root доступ к данным). Поэтому 180 дней — приемлемо.

Исключение: токены с scope `admin` или `write:org` — ротировать как DB password
(ежеквартально).

### SSH-ключ — 365 дней

SSH-ключи защищены passphrase (если используется) или хранятся в Keychain/pass.
Их компрометация требует доступа к рабочей машине оператора — что обычно означает
полную компрометацию, при которой плановая ротация ключа уже не помогает.

Ежегодная ротация — гигиеническая мера: если ключ утёк (например, оператор
случайно закоммитил приватную часть), окно эксплуатации ограничено.

### TLS — автоматически

Let's Encrypt выдаёт сертификаты на 90 дней. Через acme.sh + cron автоматическое
обновление за 30 дней до истечения — без участия оператора. Ручная ротация
требуется только при компрометации приватного ключа.

## Триггеры внеплановой ротации

### Утечка обнаружена

Не ждать планового срока. Действовать в течение часа:

1. Создать новый секрет, подменить во всех потребителях.
2. Revoke старого в провайдерской панели (если применимо).
3. Записать incident с разбором как утекло.
4. **Рассмотреть аудит других секретов** — не утекло ли что-то ещё тем же
   способом.

### Увольнение / отстранение оператора

Все секреты, к которым у уволенного был доступ — ротировать в день увольнения:

- DB passwords (если был root доступ к серверу — все).
- SSH-ключи операторские (удалить из authorized_keys, не оставить в архив).
- API tokens с правом записи.
- WebAuthn credentials уволенного — удалить из `passkeys.json`.

### Подозрение на компрометацию рабочей машины

Те же действия, что при утечке. Дополнительно — пересмотреть всё, что могло
храниться в Keychain/менеджере паролей на скомпрометированной машине.

---

## Что НЕ ротируется планово

- **Passkey / WebAuthn credentials** — привязаны к конкретному устройству
  (Touch ID / Face ID / YubiKey). При замене устройства — добавляется новый
  credential, старый удаляется. Плановая ротация не имеет смысла.
- **Secrets, защищённые HSM / Hardware tokens** — ротация требует физического
  доступа к устройству, не делается без необходимости.

## Интеграция с дашбордом

Дашборд PANEL-06 (план 07-03 проекта-носителя) подсвечивает секреты по дате
следующей ротации:

- **Зелёный:** > 30 дней до следующей ротации.
- **Жёлтый:** 14-30 дней.
- **Красный:** < 14 дней или просрочено.

Это позволяет видеть, когда подходит плановая ротация, без необходимости каждый
раз открывать `inventory/shared/access.md`.
