# Evals для /inventory-scan

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «инвентаризация сервера»
- «сними снимок сервера»
- «что у меня на сервере»
- «обнови inventory»
- «refresh inventory»
- «scan server»
- «покажи что запущено и в каких сетях»
- «drift inventory»
- «прошёл месяц — обнови карту сервера»
- «обнови схемы инфры»
- «отрисуй диаграмму сервера»
- «нарисуй mermaid топологию»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «проверь сервер» — это `/health-check` (быстро OK/FAIL), не inventory (полный снимок)
- «аудит безопасности» — это `/audit-security`, не inventory (хотя оба read-only)
- «удали ненужные контейнеры» — inventory только смотрит, не удаляет → ответ персоны: «сначала inventory покажет что есть, потом ad-hoc принимаешь решение по каждому»
- «инвентаризация по приложению» — узкая задача, inventory работает по серверу целиком → ответ персоны с уточнением «по серверу или по конкретному сервису?»
- «нужен список БД» — это часть inventory (databases.md), но фраза узкая → активировать с прицелом на databases-секцию
- «отчёт об изменениях за месяц» — inventory сравнивает с прошлым снимком (drift), но если прошлого нет — отчёта не будет → ответ персоны: «первый snapshot сделаю сейчас, drift появится со следующего раза»

## Edge-cases (опционально)

- «что-то у меня странное на сервере, разберись» — пограничная: inventory покажет полную картину, но не объяснит «странное» → активировать inventory, в финале предложить health-check или audit-security
- «обнови inventory после deploy» — типичный сценарий, корректно активировать (и это правило персоны — после deploy обновлять inventory)
- «inventory локально, не лезь на сервер» — невозможно: inventory нужен SSH-доступ → ответ персоны с объяснением и предложением посмотреть текущий inventory/ файлы (read-only)
