# Мониторинг-стек: компоненты и trade-offs

Справочник для выбора состава стека под конкретную ситуацию. Не каждому серверу нужны все
компоненты — RAM-бюджет ограничен, а конфигурация добавляет операционную сложность.

## RAM-бюджет (фактические замеры)

| Компонент | RAM (idle) | RAM (active) | Disk | Заметка |
|-----------|------------|--------------|------|---------|
| Uptime Kuma | 50-80 МБ | 100-150 МБ | 200 МБ | Больше при >100 monitors |
| Beszel hub | 30-50 МБ | 60-80 МБ | 50 МБ | + SQLite база растёт со временем |
| Beszel agent | 20-30 МБ | 40-60 МБ | минимально | Один на каждый сервер |
| Dozzle | 30-50 МБ | 80-150 МБ | минимально | Растёт с числом контейнеров |
| Dockge | 100-160 МБ | 150-200 МБ | минимально | Самый «толстый» компонент |
| Diun | 20-40 МБ | 40-80 МБ | 100 МБ | Растёт с числом отслеживаемых образов |
| **Итого** | **~250 МБ** | **~480 МБ** | **~350 МБ** | На сервере с 1 ГБ — критично |

**Правило:** на сервере с 1 ГБ RAM ставь только `kuma + beszel`; полный стек —
от 2 ГБ RAM свободно.

## Когда какой компонент брать

### Минимальная конфигурация: `COMPONENTS=kuma`

**Когда:** только что развёрнутый сервер, 1 пет-проект, оператор хочет «чтобы алертило, если
сайт упал». Самое простое — Uptime Kuma в одном контейнере, проверки HTTP/TCP/DNS,
Telegram-алерты. Без host metrics, без логов, без compose UI.

**Trade-off:** не видно «почему упало» (нет логов и метрик), только «что упало». Для
старта это нормально.

### Стандартная конфигурация: `COMPONENTS=kuma,beszel,telegram`

**Когда:** прод-сервер с 2+ сервисами, нужен и uptime, и метрики хоста (CPU/RAM/диск). Это
«дашборд для собственного спокойствия» — открыл, увидел, что всё зелёное.

**Trade-off:** ещё нет логов в браузере (для логов — SSH + `docker logs`). Но 99% случаев
проблем диагностируется по метрикам + uptime.

### Расширенная конфигурация: `COMPONENTS=kuma,beszel,dozzle,telegram`

**Когда:** сервер с >5 сервисами, регулярная диагностика логов в браузере. Dozzle даёт
live-логи всех контейнеров — удобно искать ошибки во время релиза.

**Trade-off:** docker.sock в Dozzle = доступ ко всему Docker. Защищай Basic Auth на nginx
+ Dozzle simple auth. Не пускать публично без TLS.

### Полная конфигурация: `COMPONENTS=kuma,beszel,dozzle,dockge,diun,telegram`

**Когда:** оператор хочет UI для compose-управления (Dockge) и уведомления о новых версиях
образов (Diun). Полный стек ~350 МБ RAM.

**Trade-off:** Dockge сам по себе функционал минимально полезен (compose-команды легко
выполнять через SSH), но даёт визуальный обзор stack'ов. Diun полезен — image updates
часто просматриваются.

## Альтернативы каждому компоненту

### Uptime Kuma vs Statping vs Healthchecks.io

| | Uptime Kuma | Statping-ng | Healthchecks.io (self-host) |
|---|-------------|-------------|---------------------------|
| Active probes | да | да | НЕТ — только «dead man's switch» |
| Push-based | нет | нет | да (cron alerts) |
| UI | хорошее | устарело | минималистичное |
| Notifications | 90+ типов | 12 типов | 70+ типов |

**Вердикт:** Kuma — лучший active-monitoring; Healthchecks — для cron jobs, не для uptime.

### Beszel vs Netdata vs Glances

| | Beszel | Netdata | Glances |
|---|--------|---------|---------|
| RAM (agent) | 30-60 МБ | 200-400 МБ | 40-80 МБ |
| UI | да (hub) | да | да |
| Multi-server | да | да (через cloud) | нет |
| Long-term storage | да (SQLite) | да (через cloud / S3) | нет |
| Учётка | bcrypt в SQLite | OAuth / cloud | нет |

**Вердикт:** Beszel — лучший баланс RAM/функций для self-hosted; Netdata — если нужны
тонкие метрики (Linux scheduler, eBPF), но платит RAM.

### Dozzle vs Loki+Grafana

Dozzle — для **live**-логов (читаешь сейчас). Loki — для **historical**-логов (поиск по
прошлому). Это разные задачи:

- Dozzle: «что происходит в этом контейнере прямо сейчас»
- Loki: «найди все ошибки auth за последние 7 дней»

Для self-hosted на одном сервере Dozzle обычно достаточно. Loki — если уже >3 серверов
или регулярный анализ исторических логов.

### Dockge vs Portainer vs Yacht

| | Dockge | Portainer CE | Yacht |
|---|--------|--------------|-------|
| Compose-only | да | нет (Docker все API) | да |
| RAM | 100-200 МБ | 200-400 МБ | 100-150 МБ |
| Учётки | да (multi-user) | да (multi-user) | да |
| Templates | нет | да | да |

**Вердикт:** Dockge простой и compose-focused; Portainer — мощный, но избыточный для одного
сервера. Если на сервере только compose-stacks, Dockge лучше.

### Diun vs Watchtower

| | Diun | Watchtower |
|---|------|-----------|
| Что делает | Уведомляет о новых версиях | Автоматически обновляет |
| Риск | Низкий (только notify) | Высокий (могут сломать сервис) |
| Конфиг | YAML / labels | Labels |

**Правило:** для прод-серверов **только** Diun (notify). Watchtower — для домашних
лабораторий, где простой не критичен.

## Особые случаи

### Сервер за CDN (Cloudflare flexible mode)

Let's Encrypt не нужен — TLS терминируется на CDN. Достаточно self-signed cert на nginx
+ flexible mode на CDN. Однако security-wise лучше strict mode + DNS-01 challenge.

### Сервер за корпоративным NAT (нет публичного IP)

HTTP-01 challenge не работает. Варианты:
1. DNS-01 через API DNS-провайдера (поддерживается acme.sh для большинства провайдеров)
2. Internal CA + internal DNS (требует инфраструктуру внутри компании)
3. mkcert + распространение root CA на клиенты

### Несколько серверов — один hub

Beszel hub поддерживает множество agent'ов. Архитектура:
- Hub на одном «контрольном» сервере (тот, где удобный UI-доступ)
- Agent на каждом целевом сервере, в `network_mode: host`
- Сопряжение через SSH-ключи (один ключ на пару hub↔agent)

Sync через API hub'а или вручную через UI.

### Сервер с <1 ГБ RAM

Ставь только `kuma`. Beszel host metrics можно получить через `node_exporter` + Grafana
Cloud (бесплатный tier — 10k metrics, для одного сервера хватит). Это вынос observability
из локального стека → меньше RAM на сервере.
