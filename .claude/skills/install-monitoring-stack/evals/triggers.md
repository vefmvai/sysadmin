# Evals для /install-monitoring-stack

> Источник правды для `description` в SKILL.md. Сначала пишутся triggers, потом
> из них формулируется description. При расхождении — описание обновляется под
> triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «поставь мониторинг»
- «хочу дашборд»
- «нужны глаза на сервер»
- «Uptime Kuma»
- «Beszel»
- «monitoring stack»
- «настрой дашборд по контейнерам»
- «поставь Dozzle и Dockge»
- «нужны алерты в Telegram если упадёт сайт»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «настрой VPN-панель» — это другой жанр (3X-UI / VPN-блок), не monitoring → отдельный VPN-скилл
- «настрой Grafana + Prometheus с метриками приложения» — тяжёлый стек, не покрывается этим скиллом → ответ персоны: «у меня лёгкий стек, для Grafana — отдельная задача, могу подсказать concept»
- «проверь сервер» — это `/health-check`, не установка стека
- «настрой алерты на бэкапы» — это часть `/setup-backups`, не monitoring-stack
- «обнови Uptime Kuma до новой версии» — это `/deploy-service` (update), не первичная установка
- «удали monitoring-стек» — обратная операция, скилл этого не делает → отдельный uninstall или ad-hoc

## Edge-cases (опционально)

- «поставь только Uptime Kuma» — скилл умеет, параметр `COMPONENTS=kuma` → активировать в частичном режиме
- «у меня сервер 1 ГБ RAM, можно мониторинг» — нет, скилл требует ≥ 2 ГБ → ответ персоны с предупреждением и ad-hoc минимальной конфигурацией (только Uptime Kuma локально)
- «хочу мониторинг, но без Telegram» — поддерживается, но без алертов теряется смысл → активировать, но в конце явно сказать «алерты не настроены, мониторинг будет нем»
