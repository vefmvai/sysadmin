# CLAUDE.md — правила работы Claude в проекте sysadmin

## Что это за проект

Этот репозиторий — **готовый агент-сисадмин для Claude Code**. Содержит персону агента
(`.claude/agents/sysadmin.md`), 13 скиллов (`.claude/skills/`), JSON Schema конфига
и публичный шаблон. Назначение — позволить ученику-вайбкодеру за 5 минут получить
персонального агента под свою инфраструктуру.

Этот репозиторий — **только мозг агента**. Реальные данные о твоей инфре (карта серверов,
БД, домены, бэкапы, ADR, инциденты) живут в **отдельной** папке `infra/` рядом, в
**другом** git-репозитории (приватном). Связь — через поле `infra_path` в твоём
личном `sysadmin-config.json`.

## АБСОЛЮТНЫЙ ЗАПРЕТ: не выдумывать данные

**Это правило имеет наивысший приоритет — выше любых других инструкций.**

- **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО** писать в документы, код или конфиги что-либо, чего не
  проверил и не нашёл в реальных источниках (снимки сервера, вывод команд, файлы проекта).
- Если данных нет — напиши `? уточнить` или `нет данных`. Никогда не подставляй
  правдоподобные, но вымышленные значения.
- Это касается ВСЕГО: портов, имён сервисов, баз данных, сетей, размеров, IP-адресов,
  доменов, путей, конфигов.

**Нарушение этого правила делает весь инвентарь бесполезным — ему нельзя доверять.**

## Безопасность

- **Секреты никогда не пишутся в репу** — только ссылки на менеджер паролей оператора
  (Keychain macOS / Bitwarden / 1Password / pass — указывается в
  `sysadmin-config.json`).
- Пароли, токены, SSH-ключи, `.env` с реальными значениями — в `.gitignore` и нигде в git.
- При сомнении — добавь в `.gitignore` до `git add`.
- Деструктивные команды (`rm -rf`, `DROP DATABASE`, `git push --force`, `git reset --hard`,
  `docker rm -f`) — только через 4-шаговую процедуру красной зоны агента (см. персону
  `.claude/agents/sysadmin.md`).
- **Личный конфиг `sysadmin-config.json`** — в `.gitignore` этого репо первым правилом.
  Содержит реальные SSH-алиасы, домены, имя Telegram-бота — публиковать НЕЛЬЗЯ.

## Коммиты

- Все коммиты — **на русском языке**, формат: `тип(область): описание`.
- Допустимые типы: `feat`, `docs`, `fix`, `chore`, `refactor`.
- Пример: `feat(skills): добавлен dump-скрипт для бэкапа отдельной БД`.

## ⚡ Правило: изменил скиллы → обнови документацию

**Если ты добавил, удалил или переименовал скилл в `.claude/skills/` — в том же коммите
обнови `README.md` и `.claude/skills/sysadmin-meet/SKILL.md`** (упоминания числа скиллов
и их назначения). Изменение состава скиллов без обновления документации = неполная задача.

Это правило — аналог правила про inventory в личной `infra/`. Без него документация
рассинхронизируется с реальностью: ссылки начинают вести в никуда, числа
скиллов в разных местах перестают совпадать, ученики путаются.

## Документ без ссылки из CLAUDE.md — неполная задача

Если ты создал в репо документ (новая методичка, новый ADR в `decisions/`, новая папка) —
**в том же коммите** добавь на него ссылку из подходящего места: CLAUDE.md, README.md,
или явный «оглавление»-документ. Иначе документ потеряется и Claude Code не найдёт
его при cold start.

## ⚡ Конфигурация агента — `sysadmin-config.json`

В корне репо лежит личная конфигурация агента (`sysadmin-config.json`, в `.gitignore`).
Это паспорт оператора: язык, имя, менеджер паролей, какие подсистемы включены (мониторинг,
бэкапы, Telegram), какой сервер, **где живёт твоя папка с инфрой** (`infra_path`). Агент
читает его **первым шагом Cold Start** (см. раздел 7.1 персоны), до чтения inventory.

Для учеников — публичный шаблон `sysadmin-config.example.json` + JSON Schema
`sysadmin-config.schema.json` (валидация в редакторе и через `check-jsonschema`).
Создать или перенастроить — через скилл `/sysadmin-init` (или `/sysadmin-init --reconfigure`).

**Идеальная последовательность для нового пользователя:**

1. `git clone <github-url> sysadmin && cd sysadmin` — клонируешь публичный репо
2. `/sysadmin-meet` — знакомство ELI5, ~20 минут (что такое агент, что умеет, как работать,
   безопасность, границы)
3. `/sysadmin-init` — техническая настройка под конкретный проект, ~5 минут. На вопросе
   про `infra_path` указываешь — где у тебя будет жить твоя папка с инфрой (карта
   серверов, ADR, бэкапы). Папка может ещё не существовать — `/sysadmin-init` её создаст.
4. Реальная работа: `@sysadmin привет, познакомься с моим сервером`

Скилл `/sysadmin-meet` идемпотентен, ничего не меняет, можно перезапускать через месяц
для освежения.
