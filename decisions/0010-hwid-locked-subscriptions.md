# 0010: Извлечение из закрытых (HWID-locked) подписок — отдельный скилл «под ключ»

**Статус:** Принято (переписан 2026-05-25 после боевого прогона — заменяет первую
редакцию, где механизм HWID был описан неверно)
**Дата:** 2026-05-25
**Автор:** —

## Контекст

Агент извлекает сервера из платной подписки провайдера, чтобы завести их в outbound
панели 3X-UI. Современные провайдеры (**Panterra VPN, NurVPN**, многие Happ-обёртки
на панели **Remnawave**) защищают список серверов **HWID-привязкой**: реальный
список отдаётся только зарегистрированному устройству, незнакомому приходит заглушка.

Повод — прогон ученика (Panterra) + последующий **боевой прогон на реальной
подписке** (провайдер на панели Remnawave). Первая редакция этого ADR (и фикс
v1.6.0) описывала механизм НЕВЕРНО — «свободный слот → предъявить HWID → провайдер
привяжет». Боевой прогон curl'ом вскрыл реальную картину и два бага.

### Реальный механизм (проверено curl, НЕ гипотеза)

Сервер сообщает состояние в HTTP-заголовках ответа:
- `x-hwid-active: true`, `x-hwid-limit: true` — HWID-механизм и лимит включены.
- `x-hwid-not-supported: true` — прислан НЕ-Happ клиент / без `X-Hwid` → заглушка
  «App not supported» (`vless://000...@0.0.0.0:1`).
- `x-hwid-max-devices-reached: true` — заголовки Happ есть, но HWID **не в списке
  устройств** И свободных слотов нет → заглушка «Достигнут лимит устройств».
- **HWID уже зарегистрированного устройства** (из plist установленного Happ
  оператора) → сервер отдаёт **полный конфиг**.

Это **два разных состояния, не одно**. И главное: рабочий путь — **HWID
устройства оператора, на котором подписка уже работает** (Happ зарегистрировал его
сам при первом запуске). Регистрация нового HWID = первый запрос с ним при
свободном слоте; отдельного HTTP-эндпоинта регистрации нет (`/register`, `/device`
→ 502). Освобождение слота делает пользователь в приложении/боте провайдера.

### Два бага, вскрытые прогоном

1. **Неверная трактовка `max_devices_reached` как «тупик / только через бот».**
   Агент (и я в диалоге) сделал вывод «curl не сработает». Опровергнуто: HWID из
   установленного Happ оператора отдал полный конфиг. Контрпример был в памяти
   (NurVPN извлекался этим же методом) — надо было перепроверить, а не теоретизировать.
2. **Формат ответа — Xray-JSON массив профилей**, не base64-список `vless://`.
   Panterra отдала 28 Xray-профилей (37 серверов). Старый `parse-subscription.sh`
   декодировал как base64, не нашёл `vless://`, дал ложный «0 серверов» = «тупик».

### Дополнительное требование оператора

Скилл должен работать «под ключ» для **полного новичка**: никаких технических
развилок перед пользователем, всё объяснять простым языком, спрашивать только то,
что человек физически может знать/сделать (дать ссылку, посмотреть/освободить
устройство в приложении).

## Решение

**Извлечение вынесено в отдельный скилл `extract-subscription-servers`** — это
отдельный жанр сложности (мультиформат + HWID-замок + слоты), его нельзя топить в
`configure-vpn-routing`. Скилл:

1. **Разведка заголовков** (`inspect-subscription.sh`) — читает `x-hwid-*`,
   `subscription-userinfo` (расход/срок), `profile-title`/`announce`, выдаёт
   человеко-читаемый вердикт `{kind, alive, hwid_state, …}`. По нему скилл говорит
   пользователю простым языком, что за подписка и что происходит.
2. **HWID — приоритет пути A** (`resolve-hwid.sh`): сначала HWID из установленного
   Happ оператора (слот не тратим, устройство уже в списке — это рабочий путь);
   путь B (стабильный сгенерированный HWID) — запасной, занимает слот при свободном.
3. **Распознавание формата** (`parse-subscription.sh` → `parse-xray-json.sh`):
   base64-vless / plain-vless / **Xray-JSON массив профилей** / sing-box JSON.
   Сначала `jq empty`, потом решение о декодировании (не наоборот).
4. **Диагностика состояний**: `not_supported` лечится правильными заголовками Happ
   (автоматически); `max_devices_reached` — действием пользователя (освободить слот
   в приложении), объяснённым простым языком. Не путать.
5. **Экспорт «под ключ»** (`export-servers.sh`): читаемый `.txt` (сводка по странам
   + параметры + готовые vless-ссылки) + JSON в `$INFRA/.../vpn-subscriptions/`.

`configure-vpn-routing` извлечением больше **не занимается** — читает готовый JSON
из infra (Шаг 5A.1), либо направляет оператора запустить `/extract-subscription-servers`.
Скрипты `parse-subscription.sh`, `resolve-hwid.sh`, `save-subscription-servers.sh`
удалены из configure-vpn-routing (логика переехала / стала не нужна).

Покрытие клиентов узкое (Happ основной + v2rayN фолбэк), не перебор — перебор
случайными HWID рискует жечь слоты.

## Последствия

- ✅ Инцидент «нет способа обойти» закрыт фактически: на боевой подписке скилл
  извлёк 37 серверов (проверено E2E).
- ✅ Реальный механизм задокументирован верно (заменена ошибочная первая редакция).
- ✅ Чистое разделение: извлечение (`extract-subscription-servers`) ↔ маршрутизация
  (`configure-vpn-routing`) ↔ клиентские конфиги (`generate-client-config`).
- ✅ «Под ключ» для новичка: технические термины только в логах, перед пользователем —
  бытовые действия и человеческая речь.
- ✅ Знание из `subscription-mirroring.md`/памяти стало исполняемым кодом.
- ⚠️ Класс файлов `.hwid-<provider>` в приватной infra/ (`600`) — не в публичный репо.
- ⚠️ Happ-профиль заголовков (DEVICE_OS=macOS и т.д.) — служебные значения запроса,
  имитирующие приложение; это не «выдуманный инвентарь» (правило №1 их не касается).
- ⚠️ Гео-блок поверх HWID (NurVPN на не-РФ IP → 502) — при правильном HWID и пустом
  ответе делать запрос с РФ-хоста.
