# 0001: Канон формы скилла

**Статус:** Принято
**Дата:** 2026-05-15
**Автор:** Claude (по результатам аудита `AUDIT-METHODOLOGY.md`) + согласовано с Василием

## Контекст

В репозитории `sysadmin/` к моменту аудита накопилось 13 скиллов. Аудит по 10 методичкам Anthropic (см. `AUDIT-METHODOLOGY.md`) выявил, что архитектурно скиллы зрелые, но без единого зафиксированного канона:

- Имена скиллов в императиве (`bootstrap-new-server`), методичка 03 рекомендует герундий — расхождение есть, но не критично; нужно зафиксировать выбор.
- Frontmatter содержит поле `when_to_use`, которого нет в стандарте методичек 02/03 — дублирует часть `description`.
- Префиксы непоследовательны: 11 скиллов без префикса (`bootstrap-new-server`, `deploy-service`), 2 с префиксом `sysadmin-` (`sysadmin-init`, `sysadmin-meet`).
- Размеры в основном укладываются в лимит 500 строк (методичка 03), но `sysadmin-meet` — 872 строки, `sysadmin-init` — 544.
- Ни одного evaluation-сценария ни в одном скилле.
- Структура папки скилла (`references/` + `scripts/` + `templates/`) уже соблюдается во всех 13 — это сильная сторона, надо зафиксировать.
- Описания скиллов используют XML-теги (`<role>`, `<context>`, `<goals>`) — это удачное расширение канона методички 08.

Без зафиксированного канона добавление новых скиллов (например, VPN-блок по внешнее ТЗ о добавлении VPN-блока) воспроизведёт текущий разнобой. Этот ADR закрывает все шесть точек роста среднего и низкого приоритета из аудита, касающиеся формы скилла.

Рассматривались альтернативы:
- **Не фиксировать канон, оставить как есть** — отвергнуто: добавление новых скиллов умножит дрифт.
- **Полностью переписать существующие скиллы под рекомендации методичек (герундий, без `when_to_use`)** — частично принято, но имена скиллов *не* меняем — см. секцию «Решение → 2. Имена».

## Решение

Любой скилл в репозитории `sysadmin/` соответствует канону ниже. Существующие 13 приводятся к канону отдельной задачей. Новые пишутся по канону сразу.

### 1. Структура папки

```
.claude/skills/<имя-скилла>/
├── SKILL.md                ← главный файл, ≤ 500 строк (hard cap)
├── references/             ← документы, которые читаются *по запросу* из SKILL.md
│   └── <тема>.md
├── scripts/                ← shell/python-скрипты, которые SKILL.md вызывает
│   └── <шаг>.sh
├── templates/              ← шаблоны файлов, которые скилл копирует/генерирует
│   └── <имя>.tmpl
└── evals/                  ← обязательно с момента принятия этого ADR
    └── triggers.md         ← positive/negative фразы для проверки распознавания
```

Папки `references/`, `scripts/`, `templates/` — опциональны. `SKILL.md` и `evals/triggers.md` — обязательны.

### 1.1. Имена скриптов внутри `scripts/`

Два сценария именования — выбор по типу процедуры скилла:

**Сценарий A: «Линейная процедура» — нумерованные имена `NN-<имя>.sh`.**

Используется когда:
- Скилл выполняет шаги **строго последовательно**.
- Каждый шаг зависит от предыдущего (например, нельзя сделать шаг 3 до шага 2).
- Порядок не описан только в SKILL.md, но и читается из `ls scripts/`.

Формат: двузначный десятичный префикс `00..99`, дефис, kebab-case имя, расширение.
Примеры: `01-ssh-hardening.sh`, `02-ufw-setup.sh`, `03-fail2ban.sh`.

Правила:
- **Без пропусков** в нумерации. Если есть `01..04`, то `05` — следующий,
  не `99`. Резервирование «магических» номеров (`99` для последнего шага)
  — анти-паттерн, унаследованный от init.d-стиля. Не используем.
- **`00`** разрешён для pre-check скриптов, которые запускаются **до**
  основной процедуры (детекция существующего состояния, валидация
  параметров, проверки read-only). Эта нумерация согласуется с конвенцией
  «Шаг 0: Pre-check» в SKILL.md.
- Если шагов больше 99 — это сигнал, что скилл слишком большой и его
  стоит разбить. На практике 5-10 шагов — типичный масштаб.

Скиллы с линейной процедурой (на момент принятия правила):
`bootstrap-new-server`, `setup-vpn-panel`, `setup-server-proxy`,
`migrate-server-to-server`.

**Сценарий B: «Набор инструментов» — имена без префикса.**

Используется когда:
- Скилл — это **коллекция операций**, которые SKILL.md вызывает в
  разном порядке по контексту.
- Скрипты могут запускаться независимо друг от друга или в разной
  комбинации.
- Порядок определяется логикой SKILL.md, не физическим расположением.

Формат: kebab-case описательное имя, без префикса.
Примеры: `parse-vless-link.sh`, `add-clients.sh`, `quick-health.sh`,
`backup-postgres.sh`.

Скиллы с набором инструментов (на момент принятия правила):
`audit-security`, `cleanup-existing-server`, `configure-vpn-routing`,
`deploy-service`, `generate-client-config`, `health-check`,
`install-monitoring-stack`, `inventory-scan`, `rotate-secrets`,
`setup-backups`, `setup-secrets-vault`, `sysadmin-init`.

**Критерий выбора:**

```
Если шаги скилла зависят друг от друга и порядок жёстко
определён процедурой → Сценарий A (нумерация).
Если шаги — независимые инструменты, вызываемые по
логике SKILL.md → Сценарий B (без префиксов).
```

В рамках одного скилла **смешивать стили нельзя** — либо все скрипты
с номерами, либо все без. Если в линейном скилле возникает скрипт-helper,
который вызывается из нескольких шагов — он либо именуется тоже с номером
(в той точке, где впервые вызывается), либо выносится в общую
библиотеку `scripts/lib-api/` корня репо (по образцу `scripts/lib-api/3xui.sh`).

### 2. Имена скиллов

Используем **императив или существительное** в kebab-case: `bootstrap-new-server`, `deploy-service`, `health-check`. **Не переходим на герундий** (`bootstrapping-server`), хотя методичка 03 его рекомендует.

Причины:
- Все 13 существующих скиллов уже в этом стиле.
- Переименование сейчас сломает обратную совместимость (внешние материалы проекта, документация пользователям, привычка пользователей).
- Когда репо превратится в плагин, имена закрепляются в `manifest.json` и `slash_prefix`-команды — переименование станет breaking change для всех установивших.
- Методичка 03 рекомендует, но не требует герундий.

**Префикс `sysadmin-`** — только для meta-скиллов (управляют самим агентом, а не инфраструктурой): `sysadmin-init` (настройка агента под оператора), `sysadmin-meet` (знакомство с агентом). Все остальные — без префикса.

### 3. Frontmatter

Минимальный обязательный набор полей:

```yaml
---
name: <имя-скилла-kebab-case>
description: |
  <2-4 строки в третьем лице: что делает + когда применять.>
  <Явные фразы-триггеры на русском и английском.>
allowed-tools: Bash, Read, Edit, Write   # минимально необходимые
---
```

Поля, которые **удаляются** из frontmatter всех существующих скиллов:
- `when_to_use` — дублирует `description`, нет в стандарте. Содержательную часть «когда не использовать» переносим в тело SKILL.md в раздел `<context>` → «НЕ предполагается».
- `disable-model-invocation: false` — равно дефолту, не нужно явно. Оставляем только когда `true`.

Поля, которые **могут быть** в frontmatter дополнительно:
- `disable-model-invocation: true` — для скиллов, которые **никогда** не должен автоматически подхватывать Claude (например, `sysadmin-meet` — он вызывается строго по `/sysadmin-meet`).
- `model: <модель>` — если скиллу нужна конкретная модель (редко).

### 4. Описание (`description`) — критерий правильности

`description` — единственное поле, которое Claude видит при выборе скилла. Если оно плохое — скилл либо не подхватывается, либо подхватывается не вовремя.

Правила:
- **В третьем лице** («Делает X», не «Я делаю X»).
- **Структура:** [что делает в 1-2 строки] + [явные триггерные фразы оператора] + [явные анти-триггеры или «когда НЕ использовать»].
- **Триггеры — двуязычные**, потому что операторы пишут вперемешку: «настрой VPN», «setup VPN», «подними прокси», «setup proxy».
- **Длина:** ≤ 8 строк (Claude читает description при каждом выборе — длинные тратят контекст).
- **Анти-триггеры — обязательны** хотя бы один: «НЕ для X». Это страховка от ложных срабатываний.

Пример (`bootstrap-new-server`):
```yaml
description: |
  Полная первичная настройка свежего VPS (Ubuntu 22.04+/Debian 12+):
  SSH-ключи, UFW, fail2ban, Docker + compose, /opt/, git с gitleaks pre-commit,
  шаблоны ADR/runbook/incident. Только свежие VPS.
  Триггеры: «настрой новый сервер», «получил VPS», «свежий ubuntu», «bootstrap server».
  НЕ для приведения хаоса в порядок (для этого — cleanup-existing-server).
```

### 5. Тело SKILL.md — структура

Используем XML-теги, как уже сложилось в существующих скиллах. Это расширение канона методички 08 (рекомендует XML для сложных промптов).

Обязательные секции:
- `<role>` — кто я и что делаю (один абзац).
- `<context>` — что предполагается / что НЕ предполагается перед запуском.
- `<goals>` — что должно стать TRUE после выполнения (чек-лист).
- `# Параметры` — таблица с required / default.
- `# Процедура` — шаги с явной классификацией зоны (🟢/🟡/🔴) и точками подтверждения.
- `# Финальный отчёт` — что показывает оператору в конце.

Опциональные секции:
- `# Подводные камни` — известные грабли, на которых спотыкались.
- `# Откат` — как откатить, если что-то пошло не так.
- `# Failed attempts` — что пробовали и не получилось, чтобы повторно не пытаться.

Что **не делаем** в теле:
- Не пишем длинных «лирических» вступлений. Тон существующих скиллов (`bootstrap-new-server`, `deploy-service`) — сухой, профессиональный, по делу.
- Не дублируем то, что есть в персоне или в общих knowledge-документах — ссылаемся.
- Не закладываем хардкод значений (IP, доменов, путей с конкретными именами) — только параметры. Хардкоды стандартизированных значений (порт 22, путь `/opt/`) допустимы.

### 6. Размер

**Hard cap: 500 строк** в `SKILL.md`. Это лимит из методички 03.

Если скилл получается длиннее — что-то делается неправильно:
- Большие справочные блоки выносятся в `references/`.
- Большие скрипты — в `scripts/`.
- Шаблоны файлов — в `templates/`.

Прогрессивное раскрытие — только **один уровень** ссылок вглубь. `SKILL.md` ссылается на `references/X.md`, но `references/X.md` *не* ссылается на `references/Y.md` дальше внутрь. Это правило методички 03 — не позволяет Claude уйти в кроличью нору.

### 7. Evaluations (обязательно)

В каждом скилле — `evals/triggers.md`. Структура:

```markdown
# Evals для /<имя-скилла>

## ДОЛЖНО триггерить (positive — минимум 3)
- «<фраза оператора, после которой Claude должен выбрать этот скилл>»
- ...

## НЕ ДОЛЖНО триггерить (negative — минимум 3)
- «<фраза, после которой Claude НЕ должен выбрать этот скилл>» — `<альтернатива: другой скилл / просто ответ / уточняющий вопрос>`
- ...

## Edge-cases (опционально)
- «<пограничная фраза>» — `<ожидание: с уточнением → активирует / без уточнения → не активирует>`
```

Минимум — 3 positive + 3 negative. На сложные скиллы (с пересечениями) — больше.

`triggers.md` — **источник правды для description**: сначала пишутся triggers (что скилл должен ловить), потом из них формулируется `description`. Не наоборот.

Проверка: на уровне репо лежит `scripts/run-evals.sh` (создаётся отдельно), который для каждого `triggers.md` запускает `claude -p` в headless-режиме и сверяет, какой скилл выбран. Прогон — вручную перед merge или релизом, не на каждый коммит.

### 8. Документация (правило проекта)

Из `CLAUDE.md`: «изменил скиллы → обнови документацию». При добавлении/удалении/переименовании скилла **в том же коммите** обновляется:
- `README.md` (число скиллов, таблица групп).
- `.claude/skills/sysadmin-meet/SKILL.md` (или его `parts/`, если уже распилен).

## Последствия

✅ **Плюсы:**
- Новые скиллы (включая VPN-блок) пишутся в единой форме без рассуждений «как делать».
- Поле `evals/triggers.md` — спецификация и тест в одном. Чище `description` и меньше ложных срабатываний.
- Существующие 13 скиллов получают чек-лист приведения к канону: удалить `when_to_use`, убедиться, что description в каноне, добавить `evals/triggers.md`.
- Канон документирован — следующий Claude или контрибьютор понимает форму без угадывания.

⚠️ **Минусы / компромиссы:**
- Сознательное отклонение от рекомендации методички 03 по герундию — фиксируется как осознанный выбор, чтобы будущий ревизор не сбивался.
- Hard cap 500 строк может потребовать рефакторинга крупных скиллов (`sysadmin-meet` 872, `sysadmin-init` 544) — это отдельная задача.
- Удаление `when_to_use` — breaking change для существующих 13 скиллов; требует прохода по всем.

## Связанное

- ADR-0002 (канон персоны) — комплементарный: персона определяет «кто я», скилл — «что делаю».
- ADR-0003 (структура знаний) — определяет, где живут общие references, на которые ссылаются несколько скиллов.
- ADR-0004 (формат evaluations) — детализирует тест-раннер для `evals/triggers.md`.
- `AUDIT-METHODOLOGY.md` — основание для этого ADR (точки роста №3, 5, 7, 8, 9).
- Методички Anthropic 02, 03, 04, 08 — теоретическая база.
