# 0005: Архитектура VPN-блока агента

**Статус:** Принято
**Дата:** 2026-05-15
**Автор:** Claude (на основе полного чтения 4 рисёрчей в `.planning/` —
2781 строка) + согласовано с Василием

## Контекст

К моменту принятия ADR в репозитории `sysadmin/` не было компетенции
по VPN и обходу блокировок. Появилось три типовых сценария, для которых
эта компетенция нужна:

1. **Семейный VPN.** Оператор, владеющий VPS, хочет дать себе и близким
   прокси через свою панель: телефоны, ноутбуки членов семьи получают
   QR-код и пользуются. Цель — обход блокировок зарубежного интернета
   (стриминговые сервисы, нейросети, мессенджеры) и приватность от
   локального провайдера.

2. **Серверный прокси.** Программы на VPS оператора (боты, скрипты, CI,
   `pip install`, `npm install`, `curl`, `git push`) не могут ходить
   в зарубежные API (Anthropic, OpenAI, GitHub) — либо из-за блокировок
   API по IP-геолокации, либо из-за TSPU. Нужен прозрачный прокси на
   localhost сервера, через который трафик уходит в VPN-outbound.

3. **Гибкая клиентская маршрутизация.** Оператор хочет на своих устройствах
   разное правило для разного трафика: РФ-сайты напрямую (быстрее),
   зарубежные через VPN, реклама — отрезана через DNS-фильтр. Это работа
   sing-box-клиента с rule_set, не серверная.

Технический ландшафт середины 2026 года:

- **TSPU** (DPI-фильтры на оборудовании ISP в РФ) блокирует stock-протоколы
  (OpenVPN, WireGuard, Shadowsocks, VMess, Trojan) по fingerprint и активному
  пробингу. Работают: VLESS+Reality, AmneziaWG 2.0, Hysteria 2 с Salamander
  (частично).
- **API-провайдеры (Anthropic, OpenAI)** блокируют РФ-IP на стороне ToS,
  независимо от наличия DPI у клиента.
- **Hetzner ASN, DigitalOcean, OVH, Cloudflare** — throttled с июня 2025
  («16-KB curtain»).
- **iOS App Store в РФ:** массовое удаление VPN-приложений 27-28 марта 2026
  (Streisand, Happ, v2Box, v2RayTun). На 2026-05-15 не удалены: Hiddify,
  Karing.
- **sing-box-vt** (де-факто продолжение SagerNet SFI после удаления 01.08.2024
  из App Store) застрял на версии 1.11.4 (24.02.2025), отстаёт от мейнлайна
  (1.13.12) на 4 минора.
- **Xray-core** активно развивается (v26.3.27 на 27.03.2026), Reality — его
  родная технология.
- **3X-UI от MHSanaei** — де-факто стандарт панели управления Xray (38.6k
  stars), с публичным REST API, документацией на Postman.

Рассматривались альтернативы:

- **Sing-box на сервере как первичная платформа** (вместо Xray через 3X-UI):
  отвергнуто. Sing-box на сервере оправдан только для Hysteria 2 inbound,
  TUIC, ShadowTLS, AnyTLS — это узкие случаи. 3X-UI с REST API даёт зрелую
  user-management обвязку, статистику, бэкап в Telegram. Sing-box на сервере
  требует прямой правки JSON или внешних обвязок.
- **Marzban вместо 3X-UI**: отвергнуто. Marzban на PostgreSQL — оверкилл
  для типового масштаба «оператор + семья» (1-20 клиентов). 3X-UI на SQLite
  подходит лучше.
- **WireGuard / OpenVPN как первичный протокол**: отвергнуто. Заблокированы
  TSPU в РФ (см. `vpn-protocols.md` §1.1-1.2). AmneziaWG 2.0 рассматривается
  как альтернатива, но узкая аудитория клиентов.
- **Только подписка-как-outbound, без поддержки своего загр.VPS**: отвергнуто
  Василием 2026-05-15: «то что в уроках у меня используется подписка не
  отменяет того что у агента должен быть навык создания впн соединений со
  своим зарубежным сервером напрямую». **Оба пути равноправны как компетенции.**
- **Все скиллы через клики в UI / скриншоты**: отвергнуто. ТЗ Василия и
  auto-memory `project-3xui-api-source` зафиксировали API-first. Прямая правка
  SQLite — fallback (см. `3x-ui-api.md` §13). Клики в UI — аварийный режим.
- **Три скилла (как в исходном ТЗ Василия)**: пересмотрено. Кейс «возьми
  мою подписку и сделай sing-box-конфиг для iPhone» — это не «настройка
  серверной маршрутизации» и не «серверный прокси». Это **генерация
  клиентских артефактов**. Выделено в четвёртый скилл `/generate-client-config`.
- **Reality на сервере оператора в РФ**: отвергнуто как избыточное. Трафик
  клиент→свой РФ-сервер идёт внутри РФ, TSPU не работает на внутри-РФ
  маршрутах. VLESS-TCP без Reality достаточно. Reality нужен только когда
  трафик пересекает DPI-границу: на загр.VPS как inbound (для multi-hop
  через свой загр.сервер).

## Решение

### 1. Серверное ядро — Xray через панель 3X-UI

Эталонная реализация — `MHSanaei/3x-ui`. **Форки не поддерживаем** —
каждый скилл проверяет, что установлен mhsanaei, и отказывает иначе
(см. `3x-ui-panel.md` §8).

Установка — через официальный одностроковый `install.sh` с фиксацией
версии (`VERSION=v2.5.5 bash install.sh v2.5.5`). Не «ставим из исходников
руками».

Сразу после установки — обязательная замена дефолтных кредов на сгенерированные
(логин ≠ `admin`, пароль 32 символа, случайный webBasePath, нестандартный
порт). Все секреты — в менеджер паролей оператора, не в git.

### 2. Серверный inbound — зависит от расположения сервера

Скилл `/setup-vpn-panel` принимает параметр `LOCATION`:

- **`LOCATION=ru-server`** — серверный inbound для клиентов = **VLESS-TCP
  без Reality**. Обоснование: трафик клиент→свой РФ-сервер идёт внутри РФ,
  TSPU не работает, маскировка не нужна. TLS — опционально для самой панели
  (через acme через 3X-UI или certbot снаружи).

- **`LOCATION=foreign-server`** — серверный inbound = **VLESS + Reality** на
  443/TCP (или 8443/10443 при компрометации 443). Обоснование: трафик
  клиент→свой загр.сервер пересекает TSPU, маскировка под легитимный HTTPS
  обязательна. `serverName` — из проверенного списка (`www.cloudflare.com`,
  `dl.google.com`, `www.microsoft.com`) с валидацией доступности перед
  выбором.

- **`LOCATION=ask`** (default) — сеньор-обёртка (4.3 персоны), объясняем оба
  варианта и зачем разный inbound.

### 3. Outbound — два равноправных пути

Скилл `/configure-vpn-routing` принимает параметр `OUTBOUND_KIND`:

- **`OUTBOUND_KIND=subscription`** (рекомендация по умолчанию для оператора
  в РФ) — outbound через подписную ссылку платного VPN-провайдера. Скилл
  разбирает `vless://`-link (или весь subscription URL → base64 → массив
  ссылок), создаёт outbound(-ы) в панели. Если ссылок несколько — собирает
  в balancer со стратегией `leastPing` + observatory.

  **Обоснование:** минимум операционных затрат, провайдер сам адаптируется
  к новым волнам DPI РКН, оператору достаточно периодически обновлять
  подписку.

- **`OUTBOUND_KIND=self-foreign`** (путь полного контроля) — outbound
  через свой заграничный VPS. Оператор отдельно запускает `/setup-vpn-panel
  --location=foreign-server` + `/configure-vpn-routing --inbound-only` на
  загр.VPS (= получает оттуда `vless://`-ссылку с Reality), потом подключает
  её как outbound в основной панели в РФ через тот же `/configure-vpn-routing`.

  **Обоснование:** независимость от провайдеров, возможность специфических
  rule-set на стороне загр.узла, полный контроль над выходным трафиком.

- **`OUTBOUND_KIND=ask`** (default) — сеньор-обёртка, объясняем плюсы/минусы
  обоих, рекомендуем `subscription` при сомнениях.

**Гибрид** (свой загр.VPS как основной + подписка как fallback через `balancer
fallbackTag`) — поддерживается, но не основной путь.

### 4. Серверный прокси — отдельный mixed inbound

Скилл `/setup-server-proxy` создаёт **второй inbound** на той же панели:

- Тип: `mixed` (SOCKS5 + HTTP на одном порту).
- `listen: 127.0.0.1` (только локальный доступ, не из интернета).
- `port: 1080`.
- Без аутентификации (доступ только локальный).
- `udp: false`.

Маршрутизация:
- Правило 1: `inboundTag: ["mixed-inbound"]` + `geosite:category-ru` +
  `geoip:ru` → `outbound: direct`.
- Правило 2: `inboundTag: ["mixed-inbound"]` → upstream balancer.

`/etc/environment`:
```
http_proxy=socks5h://127.0.0.1:1080
https_proxy=socks5h://127.0.0.1:1080
no_proxy=localhost,127.0.0.1
```

**Буква `h` в `socks5h` обязательна** — DNS-резолвинг на прокси, не на
клиенте. Без `h` — DNS-leak в локальный resolver, Anthropic API возвращает
403 (поведение характерно конкретно для anthropic SDK, см. issue #923).

**Защита от self-loop** (см. `3x-ui-panel.md` §7.7 — это самый критический
подводный камень VPN-блока): systemd drop-in override для `x-ui.service`:

```
[Service]
Environment="HTTP_PROXY="
Environment="HTTPS_PROXY="
Environment="NO_PROXY=*"
```

Без этого override панель и Xray уходят в петлю и падают. Скилл делает
override **первым шагом**, до записи `/etc/environment`.

### 5. Клиентское ядро — sing-box (через Hiddify / Karing / SFA)

Скилл `/generate-client-config` генерирует артефакты для клиентов:

- `vless://`-URI (универсальный, принимают все xray + sing-box клиенты).
- Sing-box JSON (для Hiddify/Karing/SFA/NekoBox).
- QR-код (через `qrencode`).

**Стратегия совместимости** (см. `client-apps.md` §9.3):

| Целевая платформа | Нижняя планка ядра sing-box | Запрещённые фичи |
|---|---|---|
| iOS | 1.11.x | AnyTLS, TLS fragment, новый DNS-формат, evaluate, package_name |
| Android | 1.12 | evaluate (1.14+) |
| Desktop | 1.13 | evaluate (1.14+ alpha) |
| Универсальный | 1.11.x | как iOS |

При сомнении — генерируется конфиг под нижнюю планку (iOS = 1.11.x), он
точно сработает везде.

**Рекомендации по клиентам** (на 2026-05-15):
- **iOS** — `Hiddify Proxy & VPN` или `Karing` (не удалены из RU App Store).
- **Android** — `Hiddify`, `NekoBox`, `SFA`.
- **macOS/Windows/Linux** — `Hiddify` (универсал) или sing-box CLI.

### 6. API-first — основной путь скиллов

Все четыре скилла работают через **REST API 3X-UI** (см. `3x-ui-api.md`):

- Общий helper `scripts/_lib-api.sh` для login / cookie / api_call /
  restart Xray / logout.
- Контроль успеха — функциональная проверка (после CRUD — проверка через
  list/get), не доверие `success: true` слепо.
- Пауза 100-200мс между массовыми запросами (защита от database lock).
- Retry с экспоненциальной задержкой при network errors.
- Откат при ошибке посередине сложной операции (не оставляем систему
  в полу-настроенном состоянии).

**Fallback к SQLite** — для сценариев, которые API не покрывает (`webBasePath`
изменение, кастомные sniffing-настройки глубже `destOverride`). Перед
правкой — обязательный бэкап.

**Аварийный fallback к UI** — оператор кликает в браузере по подробному
брифингу. Применяется **только если API и SQLite оба не работают**, каждый
такой случай — todo «обновить скилл когда узнаю как сделать X через API».

### 7. Конфигурация агента — блок `vpn` в `sysadmin-config.json`

Расширяем `sysadmin-config.schema.json` и `sysadmin-config.example.json`
секцией `vpn` (опциональной):

```json
"vpn": {
  "enabled": false,
  "panel_url": null,
  "panel_web_base_path": null,
  "server_proxy_enabled": false,
  "upstream_kind": "none",
  "default_reality_dest": "www.cloudflare.com"
}
```

Поля:
- `enabled` (boolean) — общий флаг «есть ли VPN-стек на сервере». Читается
  Cold Start, `/health-check`, `/audit-security`.
- `panel_url` (string|null) — `https://{domain}:{port}` без webBasePath.
  Записывается скиллом `/setup-vpn-panel` при первой установке.
- `panel_web_base_path` (string|null) — webBasePath (`/abc123/`). Нужен
  для API-вызовов из всех VPN-скиллов. Записывается тем же скиллом.
- `server_proxy_enabled` (boolean) — настроен ли mixed inbound + systemd
  override + /etc/environment. Читается `/health-check` (curl-тест), `/audit-security`.
- `upstream_kind` (enum: `subscription`/`self-foreign`/`mixed`/`none`) —
  как организован outbound. Используется в Cold Start для контекста и
  в подсказках персоны.
- `default_reality_dest` (string) — домен-донор по умолчанию для VLESS+Reality
  inbound на загр.VPS. Скилл может предложить другой при `LOCATION=foreign-server`,
  но это разумный дефолт.

`sysadmin-init` расширяется интерактивным вопросом про секцию `vpn` (включать
ли в этой сессии onboarding; конкретные поля заполняются при запуске
соответствующих скиллов).

### 8. Расширение персоны — точечно

В ядре `.claude/agents/sysadmin.md` (по ADR-0002):

- **§4.1 Capabilities** — активировать существующую заглушку «Сеть и обход
  блокировок» (убрать слова «развернётся при написании»), оставить одну
  плотную строку с актуальным составом.
- **§3 Character** — три новых якоря:
  - Запрос «настрой WireGuard/OpenVPN для обхода блокировок РФ» → объясняю
    про TSPU + предлагаю архитектуру через 3X-UI + подписку (или свой
    загр.VLESS+Reality).
  - Запрос «настрой прокси на сервере с панелью 3X-UI» → предупреждаю про
    self-loop, делаю systemd override **первым шагом**.
  - Запрос «дай VPN-конфиг для iPhone» → напоминаю про застрявший sing-box-vt
    1.11.4 + удаление из RU App Store, рекомендую Hiddify/Karing, генерирую
    конфиг под совместимость с 1.11.
- **§4.2 Презумпции** — добавляю презумпцию «Блокировок РФ-2026»: если
  оператор работает с РФ-инфрой и собирается обращаться к зарубежным API
  из неё — предполагать недоступность напрямую, поднимать вопрос настройки
  прокси.
- **§8.1 Доменные знания** — заменить заглушку про `networking/` на актуальный
  список 4 файлов.
- **§8.3 Skills index** — добавить новую группу «Сеть и обход блокировок»
  с 4 скиллами.

Контроль размера ядра: после правок ≤ 400 строк (hard cap по ADR-0002).
Если рост превысит — часть якорей или презумпция переносится в более
сжатую формулировку.

## Последствия

### Плюсы

- ✅ **Универсальный инструмент**: агент работает с любым VPS оператора,
  не привязан к конкретному провайдеру / конфигурации / геолокации.
- ✅ **Полнота компетенций**: оба пути outbound (подписка и свой загр.VPS)
  одинаково покрыты, оператор выбирает по контексту.
- ✅ **API-first**: повторяемость, отладка, журналирование. Скилл, упавший
  на N-м шаге, легко продолжается с N+1 при повторном запуске.
- ✅ **Защита от self-loop**: главный подводный камень VPN-блока (систему-петля
  при настройке прокси) автоматизирован, без шансов на человеческую ошибку.
- ✅ **Совместимость с iOS**: стратегия «нижняя планка sing-box 1.11» решает
  проблему отстающего sing-box-vt без знания, какое конкретно устройство
  у оператора.
- ✅ **Чистая структура knowledge**: 4 документа покрывают домен networking
  без дублирования, читаются скиллами и персоной (ADR-0003).
- ✅ **Frontmatter с last_verified**: проблема устаревания доменных знаний
  (особенно про блокировки) решена прозрачно — агент предупредит оператора,
  если документ старше 6 месяцев.
- ✅ **API-cheatsheet с шаблонами**: не нужно заново разбираться с Postman
  при каждой задаче.

### Минусы / компромиссы

- ⚠️ **Зависимость от эталонной 3X-UI**: форки не поддерживаются, оператор
  с форком получает явный отказ. Компенсация — явное сообщение с предложением
  установить эталонную версию.
- ⚠️ **Эволюция блокировок**: реальность РФ-2026 меняется на месяцы, не годы.
  `verification_interval: 6_months` решает планово, но между плановыми
  ревизиями возможны устаревания.
- ⚠️ **Сложность multi-hop с своим загр.VPS**: оператор должен сначала
  поднять загр.VPS отдельным запуском скилла, потом подключить его как
  outbound в основной. Это два разных запуска `/setup-vpn-panel` + два
  разных `/configure-vpn-routing`. Описано в брифинге сеньор-обёртки,
  но требует терпения оператора.
- ⚠️ **iOS-боль вне нашего контроля**: если Hiddify или Karing удалят из
  RU App Store — поломается весь iOS-сценарий. Mitigation — `client-apps.md`
  обновляется при каждой такой волне, рекомендации меняются.
- ⚠️ **Рост ядра персоны**: после расширения VPN-блоком ядро может выйти

### Что отвергнуто и НЕ войдёт в реализацию

- ❌ Sing-box на сервере как первичная серверная платформа.
- ❌ Поддержка форков 3X-UI.
- ❌ Marzban / X-UI / другие альтернативы 3X-UI.
- ❌ Reality на сервере оператора в РФ.
- ❌ Прямые `vless://` / `vmess://` URI как путь установки (нет такого: всё
  через панель с REST API).
- ❌ Бесплатные VPN-сервисы как outbound (security risk).
- ❌ Закрытые «только-приложение» VPN-сервисы как outbound (нет vless-link).
- ❌ Маршрутизация с фичами sing-box 1.12+ для iOS-конфигов.
