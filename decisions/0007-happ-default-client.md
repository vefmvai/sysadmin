# 0007: Happ — основной рекомендованный VPN-клиент

**Статус:** Принято
**Дата:** 2026-05-24
**Автор:** Claude (@sysadmin) на основе решения оператора + наблюдения оператора + прямой проверки happ.su

## Контекст

Оператор принял решение: основной рекомендованный VPN-клиент — **Happ** («Happ — Proxy
Utility», сайт happ.su). Happ — дефолт с первого подключения; Hiddify, Karing,
v2rayN, sing-box, v2rayNG — альтернативы. Мозг агента-сисадмина был
рассинхронизирован с этим решением.

Факты, зафиксированные на 2026-05-24:

- В прежнем knowledge основным рекомендованным iOS-клиентом был Hiddify, с
  аргументацией «Happ удалён из RU App Store ~28.03.2026, а Hiddify — нет».
  Эта аргументация присутствовала в `client-apps.md` (§1, §2.2), `vpn-reflexes.md`
  (§3.8.3), ядре персоны (§3.8.3), `vpn-consultation-flow.md`, `frontline-ru.md`,
  ADR-0005 §5, скилле `generate-client-config`.
- **Новый факт (наблюдение оператора, 2026-05-24):** в российском App Store
  сейчас **нет ни Happ, ни Hiddify**. Для установки любого VPN-клиента на iPhone
  (если его ещё нет на устройстве) в любом случае нужна **смена региона App Store**.
  На десктоп (Windows/macOS/Linux) Happ ставится напрямую с happ.su.
  Все ссылки на приложение: https://www.happ.su/main/ru
- **Проверка в той же сессии:** `https://www.happ.su/main/ru` отвечает HTTP 200,
  страница содержит «Happ — Proxy Utility» — сайт живой и официальный. Happ
  поддерживает VLESS и гибкую маршрутизацию (routing) на клиенте (Xray-ядро).
- Отдельным исследованием (2026-05-22) уже было подтверждено на устройстве, что
  Happ умеет on-device split-routing через импорт routing-профиля (JSON,
  deeplink `happ://routing/...`, конструктор routing.happ.su) — см. `client-apps.md`
  §3.6 и `routing-on-device-xray.md`. Агент заточен под Happ: умеет собирать
  routing-профили под его формат.
- Веб-инструмент (WebSearch/Tavily) в этой сессии агенту был недоступен; статус
  Hiddify в RU App Store как РКН-факт через инструмент не верифицирован — опора
  на наблюдение оператора (первоисточник) + curl-проверку happ.su. Проверка
  свежести knowledge показала 0 просроченных файлов (полный refresh не требовался —
  проблема была не в устаревании по TTL, а в рассинхроне аргументации с решением).

Рассматривалась альтернатива «оставить Hiddify основным» — отвергнута: её
единственная опора («Hiddify не удалён, а Happ удалён») несостоятельна, поскольку
удаление из RU App Store бьёт по обоим одинаково, и для iPhone в любом случае
нужна смена региона. «Не удалён» перестало быть преимуществом.

## Решение

Мы делаем **Happ основным рекомендованным VPN-клиентом** агента (по решению
оператора и потому что агент заточен под него — умеет собирать
routing-профили). Hiddify, Karing, v2rayN, sing-box (sing-box-vt/SFA), v2rayNG —
**альтернативы**.

Конкретно:

1. **Установка.** Десктоп (Win/macOS/Linux) — скачивание с happ.su (смена региона
   стора не нужна). iPhone — через смену региона App Store (как и для любого
   другого клиента сейчас, т.к. в RU App Store нет ни Happ, ни Hiddify).
2. **Аргументация про «удаление» исправлена** во всех документах: вместо «Happ
   удалён, Hiddify — нет» → «в RU App Store сейчас нет ни одного из них; для
   iPhone нужна смена региона App Store; Happ на десктоп ставится с happ.su».
3. **On-device routing.** Happ явно назван годным и основным для client-side
   маршрутизации (маршрутизация на устройстве): Xray-ядро, читает routing-правила.
   Hiddify для on-device split по-прежнему не годится (строит конфиг из подписки
   сам, не исполняет произвольный raw route) — это остаётся верным.
4. **Hiddify как технический факт сохранён** там, где он не «рекомендация по
   умолчанию», а пример sing-box-клиента, формат импорта, версия ядра — эти места
   не переписывались (вычищать Hiddify из домена не требуется).

Этот ADR **заменяет раздел §5 «Рекомендации по клиентам» ADR-0005** (где iOS =
Hiddify/Karing на 2026-05-15). Остальные разделы ADR-0005 (архитектура 3X-UI,
inbound/outbound, серверный прокси, API-first) остаются в силе.

## Последствия

- ✅ Мозг агента консистентен с решением оператора: Happ — дефолт,
  остальное — альтернативы.
- ✅ Аргументация честная и устойчивая: не зависит от волатильного «кто сейчас не
  удалён», опирается на проверяемые факты (happ.su жив; для iPhone нужна смена
  региона в любом случае).
- ✅ Снят внутренний рассинхрон `client-apps.md`: верхняя часть (рекомендация) и
  нижняя (§3.6, on-device Happ, проверено на устройстве) теперь согласованы.
- ✅ Happ как кроссплатформенный Xray-клиент с on-device routing хорошо ложится на
  компетенцию агента (сборка routing-профилей) и на дефолтную server-side модель.
- ⚠️ Статус приложений в RU App Store меняется волнами — конкретный факт «нет ни
  Happ, ни Hiddify» помечен датой (2026-05-24) и источником (наблюдение оператора),
  требует перепроверки через `/refresh-vpn-knowledge LAYER=live` при сомнении.
- ⚠️ Веб-верификация статуса Hiddify в RU App Store инструментом не сделана (нет
  доступа к WebSearch в сессии) — зафиксирована опора на первоисточник оператора.
  Если при следующем live-refresh обнаружится, что Hiddify вернулся в RU App Store,
  это не отменяет решения (Happ основной по решению оператора), но строку статуса нужно
  обновить.
- ⚠️ Для генерации именно sing-box JSON (`/generate-client-config`) Happ — не самый
  прямой адресат (он на Xray-ядре, sing-box JSON у него опционален). Это учтено:
  Happ принимает vless://, subscription, QR нативно; sing-box JSON остаётся для
  sing-box-клиентов.
