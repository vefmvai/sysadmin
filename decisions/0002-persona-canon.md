# 0002: Канон формы персоны агента

**Статус:** Принято
**Дата:** 2026-05-15
**Автор:** Claude (по результатам аудита `AUDIT-METHODOLOGY.md`) + согласовано с Василием

## Контекст

Персона агента `.claude/agents/sysadmin.md` к моменту аудита — 1244 строки (~12k токенов), которые уходят в контекст subagent **до** любой задачи. Аудит по 10 методичкам Anthropic (см. `AUDIT-METHODOLOGY.md`) определил это как точку роста №1 с высоким влиянием:

- Методичка 03: «SKILL.md ≤ 500 строк, детали в `references/`» — применима и к персоне.
- Методичка 06: «Если Claude игнорирует правило, файл вероятно слишком длинный».
- Методичка 07: «Just-in-time loading, прогрессивное раскрытие» — в текущей персоне отсутствует.

При этом содержательно персона — production-grade конструкция:
- Constitution C.1–C.8 — отдельный сильный артефакт (защита от prompt-injection, type-to-confirm для красной зоны, recovery from rule break).
- Trust Zones (раздел 6) с 4-шаговой процедурой красной зоны — точное соответствие методичке 08 §«Балансирование автономии и безопасности».
- Cold Start Protocol (раздел 7.1) — production-grade init с конкретными jq-командами.

Уничтожать эти конструкции при распиле нельзя — они работают. Задача: вынести в `agents/references/` всё, что не нужно в каждой задаче, оставив в `sysadmin.md` только то, что должно быть в контексте всегда.

Рассматривались альтернативы:
- **Оставить как есть** — отвергнуто (главная точка роста аудита).
- **Распилить на множество маленьких файлов** — отвергнуто, теряется когерентность; персона — это конституция, не модули.
- **Распилить на ядро (≤400 строк) + `references/` для редко используемых блоков** — принято.

## Решение

### 1. Структура папки персоны

```
.claude/agents/
├── sysadmin.md            ← ядро, ≤ 400 строк (hard cap)
└── references/            ← блоки, читаемые «по триггеру»
    ├── trust-zones.md     ← подробные процедуры зелёной/жёлтой/красной зоны
    ├── ritual-life.md     ← регламент жизни (раздел 7.3)
    ├── cold-start.md      ← Cold Start Protocol (раздел 7.1)
    ├── first-run.md       ← First-Run Protocol (раздел 7.2)
    ├── presumptions.md    ← презумпции (раздел 4.4) — полная версия
    └── memory-ritual.md   ← раздел 5 (ритуал памяти MUST/NEVER/MAY)
```

### 2. Что обязано остаться в `sysadmin.md` (ядре)

Ядро содержит **только** то, что должно быть в контексте при **любой** задаче. Всё остальное выносится.

Обязательные секции ядра:

1. **`# 1. Identity`** — кто я (один абзац: «сеньор-сисадмин, политика трёх зон, русский язык, конфиг `sysadmin-config.json`»).

2. **`# 2. Constitution`** — все C.1–C.9 **полностью**. Это абсолютные правила, нарушение которых разрушает агента. Они не выносятся.

3. **`# 3. Character` — сжатый**. Каждый якорь — 3-5 строк (триггер / действие / граница). Развёрнутые подробности — в ядре только если уникальны; если повторяются в скиллах — переносятся в скиллы.

4. **`# 4. Capabilities` — сжатый**. Список доменных компетенций (один пункт = одна область). Развёрнутые «как я думаю про X» — выносятся в скиллы по этой теме.

5. **`# 5. Memory ritual` — компактно**. Принцип `MUST/NEVER/MAY` + лимиты + ссылка на `references/memory-ritual.md` с подробностями.

6. **`# 6. Trust Zones` — компактно**. Краткое описание трёх зон + 4-шаговая процедура красной зоны (она критична и должна быть в ядре). Подробные процедуры объявления / возражений — в `references/trust-zones.md`.

7. **`# 7. Cold Start` — компактно**. Указатель «при холодном старте читаю `references/cold-start.md`». Сам протокол подробный — в `references/`.

8. **`# 8. Capabilities Index`** — что я могу делегировать (список скиллов с одной строкой описания + список knowledge-документов).

Hard cap ядра: **400 строк**.

### 3. Что выносится в `agents/references/`

Раздел персоны → файл `references/`:
- **Раздел 6.1-6.2 (детали Green/Yellow Zone, ~160 строк)** → `references/trust-zones.md`. В ядре остаётся: только определения зон и 4-шаговая процедура Red.
- **Раздел 7.1 Cold Start Protocol (~125 строк)** → `references/cold-start.md`. В ядре: «при холодном старте — читаю этот файл первым».
- **Раздел 7.2 First-Run Protocol (~110 строк)** → `references/first-run.md`.
- **Раздел 7.3 Регламент жизни (~190 строк)** → `references/ritual-life.md`. В ядре остаётся: список ритуалов одной строкой каждый.
- **Раздел 4.4 Презумпции (~55 строк)** → `references/presumptions.md`. В ядре остаётся: список (одна строка на презумпцию).
- **Раздел 5 Memory ritual (~80 строк)** → `references/memory-ritual.md`. В ядре остаётся: принципы MUST/NEVER/MAY и лимиты.

После выноса: ядро ~370-400 строк, `references/` — 6 файлов по 80-200 строк.

### 4. Прогрессивное раскрытие — правила

Ядро `sysadmin.md` ссылается на `references/` **явно** в тексте: «См. подробности в `references/trust-zones.md`». Claude читает файл только когда срабатывает триггер из ядра — это и есть just-in-time loading из методички 07.

**Один уровень вглубь**: `references/X.md` *не* ссылается на `references/Y.md` дальше. Если в X нужен Y — текст из Y дублируется в X или Y ссылается напрямую из ядра. Это правило методички 03 (защита от кроличьих нор контекста).

### 5. Стиль текста

Ядро пишется **в первом лице**, как сейчас («Я провожу...», «Я не терплю...»). Это работает — Claude хорошо подхватывает identity.

XML-теги в ядре **не используются**. Тело ядра — чистый markdown с заголовками. XML-теги — атрибут SKILL.md, не персоны (ADR-0001 §5).

Принцип методички 08 «говори что делать, а не что не делать» — соблюдается в Character и Capabilities; **исключение** — Constitution C.1-C.8, где `NEVER` оправдан (это безусловные запреты с эскалацией). Аудит зафиксировал 47+ упоминаний `NEVER` в персоне — после распила их станет меньше за счёт переноса в `references/`.

Few-shot примеры (методичка 08 §«3-5 разнообразных примеров в `<example>`») — добавляются в `references/`, не в ядре. Ядро остаётся компактным.

### 6. Размер и hard caps

| Файл | Hard cap | Soft target |
|---|---|---|
| `sysadmin.md` (ядро) | 410 строк | 350 строк |
| Файл в `references/` | 250 строк | 150 строк |
| Папка `references/` целиком | — | до 8 файлов |

Если ядро превысит 410 строк — что-то надо вынести. Если файл в `references/` превысит 250 строк — его надо разбить или часть оставить в коде скилла, не персоне.

> **Ревизия 2026-05-24:** hard cap ядра поднят 400 → 410. Причина: добавлено конституционное правило **C.9 «Запрет на суррогаты конфига и инвентаря»** по итогам инцидента с Windows-ученицей (агент при упавшей машинерии `/sysadmin-init` создал мёртвый суррогат `infra.md` вместо папки `infra/` с `sysadmin-config.json`). C.9 — безусловный NEVER-запрет, его место в Constitution §2 (дробить по `references/` нелогично — конституция читается целиком при каждой задаче). Конституция растёт только так: явный ADR + ревизия персоны (само правило C.0 «изменение конституции только через ADR»). Soft target 350 не меняется — это сигнал «пора выносить в references», а не запрет.

### 7. Якоря Character при добавлении новых компетенций

Когда у агента появляется новый домен компетенции (например, VPN/обход блокировок — внешнее ТЗ о добавлении VPN-блока):

1. В **Capabilities** (раздел 4 ядра) добавляется один пункт: «Сеть и обход блокировок: VPN-протоколы, маскировка, мульти-хоп, реальность РФ-2026».
2. В **Character** (раздел 3 ядра) добавляются якоря **только если** новая компетенция вызывает специфические рефлексы. Формат якоря — фиксирован:
   - **Триггер:** что вижу/слышу
   - **Действие:** что делаю (1-2 строки)
   - **Граница:** где останавливаюсь
3. Подробности компетенции — в **knowledge-документах** (см. ADR-0003) или внутри скиллов этой области, **не** в персоне.

### 8. Что **никогда** не делается в персоне

- Не дублируется содержимое скиллов. Если процедура есть в скилле — в персоне максимум ссылка.
- Не пишутся пошаговые инструкции. Шаги — в скиллах.
- Не хранятся конкретные данные оператора (IP, домены, имена) — это `sysadmin-config.json` и `infra/inventory/`.
- Не пишутся «лирические вступления» и многословные обоснования — каждый пункт максимально плотный.

## Последствия

✅ **Плюсы:**
- Контекст subagent уменьшается с ~12k до ~4k токенов в ядре + JIT-подгрузка `references/` по триггеру.
- Каждое правило в ядре видно и применимо; нет «потерянных в шуме» инструкций.
- Добавление новых компетенций (VPN) становится механическим: один пункт в Capabilities + якоря в Character + knowledge-документ.
- Constitution C.1-C.8 остаётся первоклассным артефактом — расширение методички 08 сохраняется.

⚠️ **Минусы / компромиссы:**
- Распил персоны — большая разовая работа (1244 → ~400 строк, 6 новых файлов в `references/`).
- При чтении персоны человеком (например, контрибьютор изучает агента) — придётся переходить по ссылкам. Компенсируется тем, что в ядре теперь всё, что нужно знать «в общем»; в `references/` — только когда нужны детали.
- Часть процедур из ядра попадёт в `references/`, и если Claude её не подгрузит — она не сработает. Защита: триггеры из ядра должны быть **явными и узнаваемыми** («при холодном старте → читаю `cold-start.md`»).

## Связанное

- ADR-0001 (канон скилла) — формирует «куда уходит часть содержимого старой персоны»: процедурные знания → скиллы.
- ADR-0003 (структура знаний) — определяет, что хранится в `agents/references/` (личное знание агента), а что — в `.claude/knowledge/` (доменные знания, общие для скиллов).
- `AUDIT-METHODOLOGY.md` — точка роста №1 (высокое влияние).
- Методички Anthropic 03, 06, 07, 08 — теоретическая база.
