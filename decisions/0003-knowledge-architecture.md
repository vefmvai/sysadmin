# 0003: Архитектура знаний агента

**Статус:** Принято
**Дата:** 2026-05-15
**Автор:** Claude (по результатам аудита `AUDIT-METHODOLOGY.md`) + согласовано с Василием

## Контекст

В sysadmin-агенте знания живут (или должны жить) в трёх потенциальных местах:

1. **Внутри персоны** (`.claude/agents/sysadmin.md`) — кто я, что я знаю в общих чертах, как я думаю про класс задач.
2. **Внутри скилла** (`.claude/skills/<имя>/references/*.md`) — детали, которые нужны конкретно этому скиллу при выполнении его процедуры.
3. **Где-то общем** — справочные материалы, которые нужны *нескольким* скиллам, или нужны вне рамок какого-либо скилла (например, оператор задал вопрос в свободной беседе).

Текущее состояние:
- (1) перегружено (см. ADR-0002).
- (2) — реализовано в каждом из 13 скиллов (5-7 KB на скилл, чисто).
- (3) — отсутствует как явная папка. Часть общих знаний размазана по `references/` отдельных скиллов и по разделу 8.1 персоны (`# 8.1 Knowledge база (читаемые при необходимости справочники)`), который ссылается на не существующие пока файлы.

внешнее ТЗ о добавлении VPN-блока (VPN-блок) подсветило структурную проблему: документ «VPN-протоколы 2026», «реальность блокировок РФ» нужен **трём новым скиллам сразу** и **персоне** (для брифингов в свободной беседе). Класть его в `references/` каждого из трёх скиллов = три копии и тройной дрифт. Класть только в одном скилле = два других скилла не знают про него. Класть в персону = раздувание ядра.

Нужна **четвёртая папка** — общая база доменных знаний, на которую могут ссылаться и персона, и скиллы.

Рассматривались альтернативы:
- **`.claude/references/`** — отвергнуто: путь `references/` уже занят внутри скиллов, конфликт смыслов («persona references», «skill references», «common references»).
- **Хранить общее в `references/` первого скилла, остальные ссылаются** — отвергнуто: создаёт скрытую зависимость между скиллами, при удалении первого скилла остальные ломаются молча.
- **`.claude/knowledge/`** — принято. Семантически чистое имя («знания агента, не процедуры»). Не пересекается с `references/` ни в скиллах, ни в персоне.

## Решение

Три места хранения знаний с чёткими критериями, куда что класть.

### 1. `agents/references/` — личное знание агента

**Что хранится:** части персоны, которые читаются по триггеру и не нужны в каждой задаче. Это не «знания о мире», а «знания агента о самом себе» — его процедуры, протоколы, презумпции.

**Примеры файлов** (определены в ADR-0002):
- `trust-zones.md` — детали процедур зелёной/жёлтой/красной зоны
- `cold-start.md` — Cold Start Protocol
- `first-run.md` — First-Run Protocol
- `ritual-life.md` — регламент жизни
- `presumptions.md` — презумпции
- `memory-ritual.md` — ритуал памяти

**Кто читает:** только сам агент (через ядро `sysadmin.md`).
**Кто пишет:** изменения в персоне → вручную через ADR (это конституционные изменения).

### 2. `.claude/skills/<имя>/references/` — знание скилла

**Что хранится:** детали, которые нужны конкретно этому скиллу при его процедуре. Если документ нужен только при выполнении этого скилла — он лежит здесь.

**Примеры** (уже существуют в 13 скиллах):
- `bootstrap-new-server/references/ssh-hardening.md` — настройка SSH под этот скилл
- `setup-backups/references/restic-retention.md` — стратегия retention для restic
- `audit-security/references/owasp-checklist.md` — чек-лист для аудита

**Кто читает:** скилл, когда выполняется.
**Кто пишет:** автор скилла при создании/обновлении.

**Критерий:** документ читается **только** этим скиллом. Если другой скилл начнёт его читать — это сигнал к переезду в `.claude/knowledge/` (см. ниже).

### 3. `.claude/knowledge/` — общая база доменных знаний

**Что хранится:** факты о мире (технологии, протоколы, стандарты, реальность блокировок) — знания, которые **не** про процедуру конкретного скилла, а про предметную область. Эти знания используются:
- Несколькими скиллами одновременно.
- Персоной в свободной беседе (брифинги, ответы на вопросы оператора).
- Будущими скиллами, которые мы ещё не написали.

**Структура:**
```
.claude/knowledge/
├── networking/                  ← домен: сеть и обход блокировок
│   ├── vpn-protocols.md         ← карта протоколов (OpenVPN, WireGuard, VLESS+Reality, ...)
│   ├── blocking-russia-2026.md  ← реальность блокировок РФ (с датой проверки)
│   ├── multi-hop-architecture.md
│   ├── 3x-ui-architecture.md
│   └── antipatterns.md
├── security/                    ← домен: безопасность (если понадобится в будущем)
│   └── ...
└── infrastructure/              ← домен: контейнеры, сети, БД (если понадобится)
    └── ...
```

**Кто читает:**
- Скиллы — через явную ссылку из своего `SKILL.md` («См. `.claude/knowledge/networking/vpn-protocols.md` для выбора протокола»).
- Персона — через якорь в Character или Capabilities («Вижу WireGuard в РФ-проде → читаю `.claude/knowledge/networking/blocking-russia-2026.md`, формулирую ответ»).

**Кто пишет:**
- Изначально — Claude при написании скиллов по результатам рисёрча (`AUDIT-METHODOLOGY.md` § «Что осталось непроверенным»).
- Обновляется — по плану, **каждые 6 месяцев** (см. секцию «Свежесть знаний» ниже).

### 4. Критерий: куда класть новый документ

Простой decision tree:

```
Документ нужен только одному скиллу?
├── Да → .claude/skills/<имя>/references/
└── Нет → Это про процедуру/правила/протоколы агента?
         ├── Да → .claude/agents/references/
         └── Нет (это про мир — технологии, протоколы, факты)
                  → .claude/knowledge/<домен>/
```

### 5. Свежесть знаний (критично для `.claude/knowledge/`)

Доменные знания устаревают. Особенно — про блокировки, версии ПО, списки доменов/IP.

**Правила:**

1. **Каждый файл в `.claude/knowledge/`** имеет в начале frontmatter:
   ```yaml
   ---
   last_verified: 2026-05-15
   verification_interval: 6_months
   sources:
     - https://...
     - https://...
   ---
   ```

2. **Когда агент читает knowledge-документ** при работе с оператором — проверяет `last_verified`:
   - Если возраст < 6 месяцев — использует как есть.
   - Если возраст 6-12 месяцев — использует с дисклеймером оператору: «по состоянию на дату X было так; рекомендую проверить актуальность».
   - Если возраст > 12 месяцев — отказывается использовать как факт, предлагает оператору обновить документ или отказывается от утверждений.

3. **Обновление** — отдельная задача, не «попутно». Триггеры обновления:
   - Истёк `verification_interval`.
   - Оператор сообщил, что документ устарел.
   - Агент в ходе работы обнаружил расхождение с реальностью.

### 6. Связь с персоной (раздел 8.1 «Knowledge база»)

В ядре `sysadmin.md` после распила (ADR-0002) раздел 8.1 трансформируется:

**Было** (текущее, ссылки на не существующие файлы):
```
## 8.1 Knowledge база
- knowledge/protocols/X.md
- knowledge/protocols/Y.md
- ...
```

**Стало:**
```
## 8.1 Доменные знания (.claude/knowledge/)
- networking/ — VPN, прокси, обход блокировок (см. оглавление в самой папке)
- security/ — TBA
- infrastructure/ — TBA
```

То есть в персоне — указатель на папку (что такое в принципе есть), а не пофайловый список. Это снимает дрифт «персона говорит про файл X, файла X нет».

### 7. Имена файлов и доменов

**Имена доменов** (подпапки в `.claude/knowledge/`):
- В единственном числе по существительному: `networking/`, `security/`, `infrastructure/`.
- Допустимо составное: `networking-proxy/` если нужно отделить от `networking/` в будущем. Сейчас — один домен на одну подпапку.

**Имена файлов** (kebab-case):
- Описательные: `vpn-protocols.md`, `blocking-russia-2026.md`, `multi-hop-architecture.md`.
- Без префиксов типа `00-`, `01-` — порядок чтения не важен, документы независимы.
- С датой в имени — **только** для документов, которые жёстко привязаны ко времени и переписываются полностью: `blocking-russia-2026.md` (если в 2027 будет `blocking-russia-2027.md` — старый архивируется).

### 8. Что **не** хранится в `.claude/knowledge/`

- Команды CLI и точный синтаксис конфигов — это процедурное, идёт в `scripts/` скилла.
- Шаблоны файлов — `templates/` скилла.
- Реальные данные конкретного оператора (IP, домены, инвентарь) — это `infra/inventory/` приватной папки оператора.
- Конституция, правила, протоколы агента — это `agents/references/`.
- Документы об архитектурных решениях — это `decisions/` (ADR).

## Последствия

✅ **Плюсы:**
- Чёткое разделение трёх типов знаний (личное агента / процедурное скилла / доменное общее).
- Один общий документ «VPN-протоколы 2026» вместо трёх копий в трёх скиллах.
- Персона может ссылаться на knowledge в свободных беседах (раздел Capabilities → конкретный файл).
- Frontmatter `last_verified` решает проблему устаревания доменных знаний — у реальных пользователей могут быть документы 2026 года, проверенные в 2028, и агент явно это скажет.
- При упаковке в плагин `.claude/knowledge/` — отдельная папка с понятным назначением, упаковывается так же, как `skills/` и `agents/`.

⚠️ **Минусы / компромиссы:**
- Новая папка верхнего уровня — `.claude/knowledge/`. Нужно отразить в `CLAUDE.md`, `README.md`, `sysadmin-meet`.
- Часть `references/` существующих скиллов может оказаться кандидатом на переезд в `.claude/knowledge/` — это разовая работа.
- Frontmatter с `last_verified` — новая ответственность автора документа. Если забыть — документ останется без даты, его актуальность не проверяется.

## Связанное

- ADR-0001 (канон скилла) — определяет `references/` внутри скилла.
- ADR-0002 (канон персоны) — определяет `agents/references/`.
- ADR-0004 (формат evaluations) — отдельный артефакт, не относится к знаниям.
- **ADR-0006 (слоистый VPN-knowledge)** — расширение этого ADR в части
  networking-домена. Вводит подпапки `_live/_reference/_meta/` с разными TTL
  (14/60/365 дней) и поле `layer` во frontmatter.
- внешнее ТЗ о добавлении VPN-блока проекта — первый реальный потребитель `.claude/knowledge/networking/`.
- `AUDIT-METHODOLOGY.md` — точка роста про прогрессивное раскрытие персоны.
- Методички Anthropic 03 (прогрессивное раскрытие), 07 (context engineering).
