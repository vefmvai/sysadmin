# 0004: Формат evaluations для скиллов

**Статус:** Принято
**Дата:** 2026-05-15
**Автор:** Claude (по результатам аудита `AUDIT-METHODOLOGY.md`) + согласовано с Василием

## Контекст

Аудит по 10 методичкам Anthropic зафиксировал точку роста №2 (высокое влияние): «нет ни одного evaluation в скиллах». Методички 02, 03, 04, 09 настаивают на 3+ eval-сценариях per skill — это и спецификация (что скилл должен ловить), и тест (правильно ли его подхватывает Claude).

Для агента-сисадмина с 13 (после ТЗ — 16) скиллами без evaluations есть конкретный риск: фразы операторов **пересекаются** между скиллами. Примеры:
- «настрой панель» — это `/install-monitoring-stack` (Grafana) или `/setup-vpn-panel` (3X-UI)?
- «обнови сервер» — это `/deploy-service`, `/audit-security` или просто `apt upgrade`?
- «защити сервер» — это `/audit-security`, `/setup-secrets-vault` или `/bootstrap-new-server`?

Без формализованных тестов триггеров невозможно гарантировать, что Claude разделяет эти случаи правильно.

Василий работает по подписке Claude Code, не через API напрямую. Headless-режим (`claude -p`) использует ту же подписку, что и интерактивная сессия — отдельный API-ключ не нужен. Это делает автотесты практически бесплатными.

Рассматривались альтернативы:
- **Только документ с triggers**, без автотестов — отвергнуто: при росте до 16+ скиллов ручная проверка перестаёт быть надёжной.
- **Сразу автотесты через Anthropic SDK на Python** — отвергнуто: дорого в разработке, требует отдельного API-ключа у будущих контрибьюторов.
- **Гибрид: документ обязателен, bash-раннер опционален, прогон через `claude -p`** — принято.

## Решение

### 1. Структура

Каждый скилл обязан иметь файл `evals/triggers.md` (см. ADR-0001 §1, §7).

В корне репо лежит `scripts/run-evals.sh` — bash-скрипт, который для каждого `triggers.md` запускает `claude -p` в headless-режиме и проверяет, какой скилл был активирован.

```
sysadmin/
├── .claude/skills/<имя>/
│   └── evals/
│       └── triggers.md          ← positive/negative фразы (читается человеком и скриптом)
├── scripts/
│   ├── run-evals.sh             ← главный раннер
│   └── eval-helpers/
│       ├── parse-triggers.sh    ← парсит triggers.md в JSON
│       └── invoke-claude.sh     ← вызывает claude -p и парсит результат
└── .eval-results/               ← результаты прогонов (в .gitignore)
    └── <timestamp>.md
```

### 2. Формат `triggers.md`

```markdown
# Evals для /<имя-скилла>

> Источник правды для description в SKILL.md. Сначала пишутся triggers, потом из них формулируется description. При расхождении — описание обновляется под triggers, не наоборот.

## ДОЛЖНО триггерить (positive — минимум 3)

- «настрой VPN-панель на сервере»
- «подними 3X-UI»
- «хочу обойти блокировки на своём VPS»

## НЕ ДОЛЖНО триггерить (negative — минимум 3)

- «расскажи как работает VLESS» — это вопрос про знания, не задача установки → ответ персоны со ссылкой на `.claude/knowledge/networking/_reference/vpn-protocols.md`
- «настрой VPN на ноутбуке» — клиент, не сервер → ответ персоны с уточнением
- «удали 3X-UI с сервера» — обратная операция → отдельный скилл `/uninstall-vpn-panel` или ручная процедура

## Edge-cases (опционально)

- «настрой панель» — неоднозначно (Grafana или 3X-UI) → должен уточнить у оператора, не активировать наугад
- «нужен VPN» (без контекста) — должен уточнить «на сервере или на устройстве» перед активацией
```

**Правила:**
- Positive — минимум 3, рекомендуется 5-7. Включать варианты на русском, английском, разговорные («подними мне» / «setup»), и ключевые слова из ТЗ.
- Negative — минимум 3. Каждый negative содержит **ожидаемое поведение** после стрелки `→`: что должен сделать Claude вместо активации этого скилла.
- Edge-cases — для скиллов с реальными пересечениями.

### 3. Тест-раннер: контракт

`scripts/run-evals.sh` работает так:

1. Принимает аргумент: имя скилла (`./run-evals.sh setup-vpn-panel`) или `--all` для прогона всех.
2. Для каждой фразы вызывает:
   ```bash
   claude -p "<фраза>" \
       --output-format json \
       --no-session-persistence \
       --append-system-prompt "Только выбери подходящий скилл из доступных. Не выполняй ничего. Ответ строго: SKILL: <имя> или SKILL: none. Если бы выбрал несколько — перечисли через запятую."
   ```
3. Парсит JSON-ответ, извлекает строку вида `SKILL: <имя>`.
4. Сверяет с ожиданием:
   - **Positive** → ожидание `<имя проверяемого скилла>`.
   - **Negative** → ожидание любое **кроме** `<имя проверяемого скилла>`.
   - **Edge-cases** — описаны явно в формате, парсятся отдельно.
5. Пишет результат в `.eval-results/<timestamp>.md` с разметкой ✅/❌.

**Что раннер НЕ делает:**
- Не запускает реальные скилли (никаких изменений на серверах).
- Не сохраняет историю сессий.
- Не пишет в `inventory/`, `knowledge/`, нигде кроме `.eval-results/`.

### 4. Когда прогонять evals

**Обязательно:**
- Перед merge в `main` любого изменения, которое добавляет/удаляет скилл или меняет `description`.
- Перед релизом плагина в marketplace (в будущем).

**Опционально:**
- При апдейте модели Claude (минорный сдвиг распознавания фраз — возможен).
- Когда оператор замечает «Claude позвал не тот скилл» — добавляем фразу в `triggers.md` соответствующего скилла как negative или positive (в зависимости от ожидания), прогоняем, чиним description.

**НЕ делается:**
- На каждый коммит (избыточно, скиллы редко меняются).
- Автоматически в CI до того, как репо станет плагином (нет необходимости).

### 5. Стоимость

Один прогон одной фразы — короткий вопрос к Claude (~200 токенов на запрос + ~50 на ответ). На 16 скиллов × 6 фраз = 96 вызовов = ~25k токенов суммарно. На подписке Claude Code — это считаные минуты активной работы из квоты. Полный прогон всех evals — не больше 10-15 минут wall-clock.

### 6. Минимум для существующих 13 скиллов

При приведении существующих скиллов к канону (см. ADR-0001) каждый получает `evals/triggers.md` со списком триггеров, выведенным из текущего `description` + анализа пересечений с соседними скиллами. Не «3 для галочки», а реальный список, который ловит:
- Типичные фразы оператора (русский + английский).
- Анти-триггеры (что **не** должно сюда попадать).
- Пересечения с соседями (если скилл рядом с другим — фраза второго должна быть в этом negative).

Прогон evals для всех 13 — отдельная задача после написания triggers, до merge VPN-блока. Если что-то не проходит — правится `description` (не triggers — triggers ближе к воле оператора).

### 7. Эволюция

**Сейчас:** bash-скрипт + `claude -p`, ручной прогон.

**При упаковке в плагин:** перевод на Python с `anthropic` SDK, интеграция в GitHub Actions CI. Это станет частью plugin marketplace contract (как у GSD и BookBench).

**Не сейчас:** избегаем over-engineering до того, как появилась реальная потребность.

## Последствия

✅ **Плюсы:**
- Каждый скилл имеет тест распознавания фразы оператора, не только описание.
- `triggers.md` — источник правды для `description`. Решает проблему «жадных» и «узких» описаний.
- Прогон через подписку Claude Code — без отдельного API-ключа и без дополнительных расходов.
- Готовая инфраструктура для плагина (тест-раннер просто меняется на Python + SDK).
- Решает риск пересечений между скиллами при росте до 16+.

⚠️ **Минусы / компромиссы:**
- `evals/` — новая обязательная папка в каждом скилле. Существующие 13 нужно пройти.
- Тест-раннер на bash — не самое масштабируемое решение (для 100+ скиллов нужно будет переписывать). Для текущих 16 — достаточно.
- Прогон evals — ручной. Если автор скилла забыл прогнать перед merge — поймает уже в продакшене на оператора. Защита: добавляется в чек-лист `CONTRIBUTING.md`.
- Headless-режим Claude Code зависит от того, что Василий и контрибьюторы запускают тесты в той же папке `sysadmin/` (иначе скиллы не видны). Это естественное ограничение.
