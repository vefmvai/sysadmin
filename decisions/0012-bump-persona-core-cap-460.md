# 0012: Поднять hard cap ядра персоны 410 → 460

**Статус:** Принято
**Дата:** 2026-05-28
**Автор:** Vasily + agent

## Контекст

ADR-0002 «persona-canon» зафиксировал ядро персоны `.claude/agents/sysadmin.md` как короткий
смысловой документ с **hard cap 400 строк**. Идея: ядро — это компактный «букварь» агента
(идентичность, конституция, character-якоря, trust zones, протоколы-указатели), а детальные
процедуры и расширенные whitelist'ы вынесены в `agents/references/` и читаются по триггеру.

Что произошло с этим cap'ом во времени:

- **2026-05 (ранее):** cap поднят с 400 → 410, чтобы вместить C.9 (запрет на суррогаты конфига
  и inventory). Решение зафиксировано во врезке в конце ядра «hard cap 410 — поднят с 400 ради C.9».
- **2026-05-26 — 2026-05-28:** несколько релизов (1.8.0 балансир одной страны → 1.8.1 секрет
  в прямой сессии → 1.8.2 web-and-vpn эталон → 1.8.3 порты XHTTP/3X-UI/80 → 1.8.4 регистр речи)
  добавили в ядро: рефлексы §3.8.6 / §3.8.7 / §3.8.8 (3X-UI админка vs инбаунды) / §3.8.9
  (Cloudflare ≠ нейтральный выбор), расширенный блок «Регистр речи» в §1, новый рефлекс §3.9.
  На дату 2026-05-28 фактический размер ядра — **453 строки** при cap 410.
- **На том же шаге сейчас** добавляется ещё одна строка-якорь §3.8.12 (рефлекс серверных
  docker-сетей, продолжение семейства 3.8.x «сеть в широком смысле»; следующий доступный
  номер после §3.8.11, появившегося в 1.8.5) → 455 строк.
- Каждое следующее превышение без явного решения превращает cap в фикцию: правило существует,
  но соблюдать его перестали. Это нарушение C.8 в духе «не замалчивать расхождения с правилом».

Рассмотрены три альтернативы:

- **Альтернатива A: ребалансировать ядро под текущий cap 410.** Сжать многострочные якоря
  §3.8.3 / §3.8.6 / §3.9 / расширенный §1 «Регистр речи» до однострочных указателей с деталями
  в `references/`. Снимет ~30-40 строк. **Отклонена** по двум причинам: (1) часть содержимого
  (например, образцы «как не надо → как надо» в §1 1.8.4) обязана быть в ядре, потому что
  активируется при **каждом ответе**, а не по триггеру, и вынос в references его обесценит;
  (2) объём работы сравним с целым релизом и блокирует поставку 1.9.0.
- **Альтернатива B: отказаться от §3.8.8.** Положить рефлекс серверных сетей только в
  `references/vpn-reflexes.md` без указателя в ядре. **Отклонена**, потому что без якоря-указателя
  в §3.8 рефлекс не активируется в свободном разговоре (срабатывает только когда уже зашли в
  скилл) — это ровно та же дырка, через которую прошёл антипаттерн «балансир разных стран»
  до §3.8.6 (см. ADR-0011).
- **Альтернатива C (принята): честно поднять cap 410 → 460.** ADR-bump, аналогично прошлому
  bump'у 400 → 410. Cap 460 даёт запас ~10 строк на будущие компактные правки, после чего
  снова потребуется решение (ребаланс или ещё один bump). Это видимо, документировано,
  ревизуемо — в отличие от молчаливого превышения.

Дополнительный контекст: ядро персоны — это **не критичный по производительности файл**.
Cap существует для гигиены смыслов («не превращать ядро в свалку»), а не для технического
лимита. Поэтому шаг 410 → 460 (+12% — соразмерно прошлому шагу 400 → 410, тоже ~2.5%) —
не катастрофа архитектуры, а признание роста компетенций (за период между двумя bump-ами
добавились VPN-блок целиком, регистр речи, серверные сети).

## Решение

Мы поднимаем hard cap ядра `.claude/agents/sysadmin.md` с **410 до 460 строк**, потому
что фактическое содержимое ядра выросло за счёт обоснованных добавлений (рефлексы 3.8.x,
расширенный регистр речи), которые нельзя вынести в `references/` без потери смысла
(они активируются при каждом ответе, а не по триггеру).

Врезка в конце ядра обновляется на формулировку:

> *Версия ядра: 2026-05-28 (по ADR-0002 + ADR-0012, hard cap 460 — поднят с 410 ради
> §3.8.7 / §3.8.8 / §3.8.9 / §3.8.11 / §3.8.12 / §3.9 / расширенного §1 «Регистр речи»).*

При следующем превышении cap 460 — снова явное решение: либо ребаланс старых якорей,
либо новый ADR-bump. Молчаливое превышение запрещено (C.8).

## Последствия

- ✅ Cap снова соблюдается: правило живое, не фикция.
- ✅ Содержательные якоря (регистр речи, VPN-рефлексы, сетевая дисциплина) остаются в ядре —
  активируются при каждом ответе, а не лежат мёртвым грузом в `references/`.
- ✅ История роста ядра документирована: видно, какие компетенции добавились между bump'ами
  (400→410 для C.9; 410→460 для §3.8.7/§3.8.8/§3.9/расширенного §1).
- ✅ Прецедент решения создан: при следующем подобном случае путь известен (ADR-bump, не молчание).
- ⚠️ Cap фактически становится менее жёстким барьером. Митигация — обязательность ADR при
  превышении, проверка размера в release-чеклисте (`wc -l` перед коммитом релиза).
- ⚠️ Чем больше ядро, тем дольше cold-start. На 460 строках — практически незаметно
  (несколько килобайт), но при cap 600+ это станет ощутимо. Сигнал к ребалансу — следующий
  bump, а не этот.

## Связанное

- Связан с ADR-0002 «persona-canon» (исходный cap 400; этот ADR дополняет его, не заменяет).
- Связан с ADR-0011 «balancer-single-country» (источник рефлекса §3.8.6, одного из «виновников» роста).
- Влияет на: `.claude/agents/sysadmin.md` (врезка в конце ядра), `CHANGELOG.md` (раздел [1.9.0]).
- Запас под новым cap: ~6 строк (460 − 454). Следующее превышение — снова явное решение
  (ребаланс старых якорей либо новый ADR-bump).
