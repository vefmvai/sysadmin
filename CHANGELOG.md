## [1.9.0] — 2026-05-28

Закрыта дырка в knowledge: у агента не было дефолтных правил по серверным сетям между
контейнерами. Поводом стал аудит мозга: VPN-блок проработан (4 файла knowledge + 7
рефлексов §3.8.1–§3.8.6 + расширения §3.8.7–§3.8.11), но **обычные docker-сети**
(сегментация frontend/backend, expose vs publish, host network, default bridge, права
на `/var/run/docker.sock`, UFW + Docker) были покрыты только одной врезкой в
`bootstrap-new-server` про UFW↔Docker. При этом боевой стек-носитель уже имеет
выкристаллизовавшийся 4-сеть-паттерн (ADR-0012 проекта-носителя, проверено 5 недель
эксплуатации) + 3 выстраданных урока (`internal:true` ломает DNAT, `external:true`
сети не удаляются при `compose down`, service key = DNS-алиас). Эталон извлечён из
работающего прода, не выдуман.

Параллельно: hard cap ядра персоны поднят с 410 до 460 строк (ADR-0012), потому что
рефлексы §3.8.7 / §3.8.8 / §3.8.9 / §3.8.11 / §3.8.12 / §3.9 и расширенный §1 «Регистр
речи» прибавили вес активному ядру, который нельзя вынести в `references/` без потери
смысла (они активируются при каждом ответе, а не по триггеру). Это сознательное
решение, аналог предыдущего bump'а 400→410 ради C.9. MINOR-бамп (1.8.5 → 1.9.0) —
новая компетенция, не патч.

## [1.8.5] — 2026-05-28

Восстановлена работоспособность всех четырёх VPN-скиллов на свежих установках
3X-UI. С релиза панели v3.0.0 (начало 05.2026) в неё добавлен CSRF middleware:
голый POST на `/login` с username/password — как делал наш `scripts/lib-api/3xui.sh`
со старта VPN-блока — отбивается HTTP 403 с пустым телом. Ученик 5-го модуля
(Полякова) на свежей панели v3.1.0 это вскрыл, исключив все ложные причины
(form vs JSON, заголовки User-Agent/Origin/Referer/X-Requested-With,
nginx vs loopback — везде 403, рубит именно CSRF). До этого фикса
`/setup-vpn-panel`, `/configure-vpn-routing`, `/setup-server-proxy`,
`/generate-client-config` были мертвы на любой свежей установке — авто-вход
по API не работал, оставались только UI-клики или прямая правка SQLite.

### Исправлено
- **`scripts/lib-api/3xui.sh` функция `api_login`** переведена на двухшаговый
  протокол с автодетектом версии панели. Шаг 1 — GET страницы логина для
  получения CSRF-токена из HTML-меты `<meta name="csrf-token">` или
  отдельного endpoint `GET /csrf-token` (появился в v3.0.x). Шаг 2 — POST
  `/login` с заголовком `x-csrf-token`, если токен есть; без заголовка,
  если на сервере v2.x (CSRF не введён). При HTTP 403 на основном POST —
  принудительное обновление токена через `/csrf-token` и повтор. Bash-API
  обратно совместимо: старые вызовы `api_login --domain X --port N
  --web-path P --admin U --password-ref REF` продолжают работать на v2.x.
- **`api_call`** научен пробрасывать `x-csrf-token` на POST/PUT/DELETE
  в режиме cookie на v3.0+ панели (CSRF middleware режет не только
  `/login`, но и `/xray/update`, `/logout` и др.). На GET-запросах
  заголовок не отправляется — не нужен. Добавлен автоматический retry
  с обновлением CSRF-токена на ответе 403 (один раз, без увеличения
  delay) — закрывает случай «токен просрочился между логином и операцией».

### Добавлено
- **`scripts/lib-api/3xui.sh` поддержка Bearer-токенов** — новый параметр
  `api_login --bearer-token-ref <ref>`. Токен создаётся оператором в UI
  панели (Settings → Security → API Tokens) и хранится в менеджере паролей
  (`keychain:3xui-panel-<server>-token` или эквивалент в pass/bw/op). При
  передаче `--bearer-token-ref` пропускается двухшаговый логин и cookie —
  все вызовы `api_call` идут с заголовком `Authorization: Bearer ...`.
  Bearer-режим устойчивее CSRF (не зависит от страницы логина), но требует
  ручного создания токена и поэтому не дефолт. Скиллы по-прежнему могут
  логиниться по username/password — Bearer применяется только если в
  `sysadmin-config.json` указан `panel_api_token_ref`.
- **§3.8.11 в персоне `sysadmin.md`** — компактный указатель и в полных
  формулировках `references/vpn-reflexes.md` (50 строк) — рефлекс «HTTP
  403 на программном входе в 3X-UI = CSRF, не сломанная панель». Триггер,
  действие (не объявлять «креды не подходят», не тыкаться вслепую,
  проверить версию, использовать обновлённый `api_login` или Bearer),
  границы, источники, инцидент.
- **Раздел §2 в `_reference/3x-ui-api.md`** переписан под новый протокол:
  предупреждение о breaking change v3.0.0, §2.1 «Логин с автодетектом CSRF»
  с минимальным bash-примером, §2.4 «Альтернатива: Bearer-token» с
  trade-off'ами, обновлённый §10.1 (таблица совместимости по версиям
  панели). `last_researched` поднят до 2026-05-28, источники включают
  release notes v3.0.0 и issue #4227.
- **Событие в `_live/timeline.md`** на 2026-05-28 «3X-UI v3.0.0+ ввёл
  CSRF, старые API-скрипты ловят 403 на /login» — категория «релиз панели
  (breaking change для скриптов)», HIGH-вес (release notes автора + issue
  #4227 + независимый прогон ученика), влияние и зафиксированность.

### Исследовано независимо
- Release notes 3x-ui v3.0.0 (MHSanaei): «feat(security): CSRF protection
  and security hardening across the application».
- Issue MHSanaei/3x-ui#4227 «API in v3.0.0» (10.05.2026, закрыт): первый
  публичный баг-репорт; в комментариях два рабочих обхода (Bearer
  Authorization и `x-csrf-token` заголовок), от автора подтверждение
  endpoint'а `GET /csrf-token`.
- v3.0.2 release notes: «sessions, SSRF, CSP nonce, CSRF logout, trusted
  proxies» — защита расширена; именованные Bearer-токены становятся
  поддерживаемой опцией.

### Совместимость
- На старых установках панели v2.x (до 05.2026) поведение не меняется —
  `api_login` пропускает CSRF-шаг (токен не находит ни в HTML, ни в
  `/csrf-token`), шлёт обычный POST. Ритуал работает «как раньше».
- На свежих v3.0+ — двухшаговый протокол активируется автоматически.
  Никаких ручных переключений в скиллах не требуется.
- Состав скиллов без изменений (20). Документ `README.md` и `sysadmin-meet`
  тоже не правились — структура скиллов та же, починена только internal
  API-механика.

## [1.8.4] — 2026-05-28

LLM подтягивает автоматически из IT-контекста. Повод — обратная связь от

обратное намерению. Состав скиллов без изменений (20).

## [1.8.3] — 2026-05-29

Достроена полная картина портов в эталоне сосуществования сайтов и VPN. Поводом
стали уточняющие вопросы оператора при чтении свежего `web-and-vpn-coexistence.md`
(1.8.2): «а где в этой схеме 3X-UI — за nginx или параллельно?» и «HTTP-трафик
тоже за nginx, или он только для https?». Оба вопроса показали, что для оператора
эталон умалчивал о двух важных вещах, очевидных автору, но не новичку: 3X-UI — не
один процесс на одном порту, а массив инбаундов в разных позициях относительно
nginx одновременно; nginx живёт не только на 443, но и на 80 — и без 80 ломается
автопродление Let's Encrypt. Состав скиллов без изменений (20).

### Добавлено
- **§2.3 «3X-UI — это не один процесс на одном порту, а массив инбаундов»** в
  `networking/_reference/web-and-vpn-coexistence.md`. Объясняет, что 3X-UI = панель
  + ядро Xray с массивом `inbounds`, каждый элемент — отдельный приёмник со своим
  портом и интерфейсом. Два независимых выбора на инбаунд (где слушать:
  `127.0.0.1` за nginx / `0.0.0.0` параллельно; какой протокол: XHTTP-WS-HTTPUpgrade
  за nginx / Reality-Vision параллельно / mixed-SOCKS внутренний / Hysteria2-TUIC
  по UDP мимо). В одной панели нормально жить инбаундам и за nginx, и параллельно,
  и внутренним одновременно.
- **§2.4 «Порт 80 — тоже nginx, и зачем»**. На 80 живут ровно две вещи: ACME-челлендж
  HTTP-01 для Let's Encrypt и редирект `http://` → `https://`. `NEVER` закрывать 80
  в UFW при HTTP-01 — сломает автопродление сертификатов.
- **§2.5 «Полная картина портов»** — сводная таблица «кто на каком порту в эталонном
  сервере» (80/443 nginx, опциональные 8443 Reality / 9443 UDP Hysteria2, всё остальное
  на `127.0.0.1`). Закрывает вопрос «nginx как главный привратник на сервере» одним
  взглядом.

### Изменено
- `last_researched` файла `web-and-vpn-coexistence.md` остался `2026-05-28` (источники
  не пересматривались), добавлено поле `last_updated: 2026-05-29` для отслеживания
  содержательных правок без перевыпуска источников.

## [1.8.2] — 2026-05-28

Закрыта дырка в knowledge: не было эталона «как сайты и VPN уживаются на одном
сервере, кто слушает порт 443». Поводом стал боевой прогон агента у ученицы

сломал VPN, вынес выбор оператору), но форму выбрал неудачную — технарский экзамен
из трёх вариантов с жаргоном («SNI-роутинг», «sqlite + рестарт ядра», «observatory»),
на котором оператор-новичок растерялся. Knowledge-заготовки под этот частый сценарий
не было — агент импровизировал, отсюда сухой технарский регистр вместо человеческого
выбора с рекомендацией. Состав скиллов без изменений (20).

### Добавлено

  «весь сервер», а не только VPN-кусок: железный закон «один TCP-порт = один

случай: агент ученицы оператора отказался работать после того, как она сама ввела

web), — это штатная рабочая передача, а не компрометация; ротация нужна, только когда
секрет покидает этот канал (git, публичный чат, скриншот, общий лог). Состав скиллов
без изменений (20).

  компрометация» и реакцией на каждый случай. `NEVER` требовать ротацию из-за того,

## [1.4.2] — 2026-05-25

Фикс резолва относительного `infrastructure.root_path`. Прогон у пользователя на

  **каталога конфига** (канон ADR-0008). Негативный кейс сохранён: если папки реально

  относительный путь от каталога конфига, tilde → `$HOME`, абсолютный (вкл. Windows
  `C:\`/`C:/`) как есть. Все потребители используют его, чтобы правило жило в одном месте.
- **ADR-0008** (`decisions/0008-infra-path-resolution.md`) — фиксирует канон резолва и
  почему база именно «каталог конфига» (`/sysadmin-init` кладёт конфиг внутрь папки инфры).

- **Параметры** `PROVIDER_SLUG`, `EXIT_COUNTRY`, `OUTBOUND_PRESET`, `PROBE_INTERVAL`.
- Триггеры: «вот моя подписка настрой выход», «хочу выходить из США», «чтобы IP
  не скакал», «настрой балансировщик» (configure-vpn-routing); «дай мне подключиться
  к своему серверу», «настрой Happ на мой сервер», «конфиг для Happ» (generate-client-config).

### Изменено
- **`setup-routing.sh`** — `PROBE_INTERVAL` по умолчанию `5m` (было `30s`): реже
  переоценка observatory → клиент дольше держится за один сервер → стабильнее IP.
  Добавлен `PROBE_SAMPLING` (для leastLoad). Комментарии-якоря про failover vs sticky.
- **`3x-ui-panel.md` §1.3–1.4** — зафиксировано (из исходника `BalancersTab.tsx`,
  проверено 2026-05-24): форма балансира 3X-UI = 4 поля, sticky-порога в мс нет;
  стабильный IP = одна страна + `probeInterval: 5m`; рекомендация 5m вместо 1m/30s.
- **Failed attempts** скилла — два новых граблекейса: «поиск sticky-порога в панели»
  (его там нет) и «заведение всех серверов подписки скопом» (= скачущий IP = бан).

## [1.2.0] — 2026-05-24

Автоматизации на карте инфраструктуры. По итогам реструктуризации Модуля 6 курса
«Архитектор ИИ-решений»: вместо «панели кнопок» ученик опирается на актуальную
Mermaid-карту от агента как на единый источник правды о том, что крутится само.
Значит, карта обязана отражать автоматизации — раньше они жили только в тексте.

### Добавлено
- **Сбор systemd-таймеров** (`dump-snapshot.sh` → `systemd-timers.txt`): на Ubuntu 24.04
  таймеры — основной способ расписания наравне с cron. Раньше собирались только
  enabled `*.service`, таймеры были полностью невидимы. `set +e` + honest-unknown
  при отсутствии systemctl.
- **Сбор скриптов-наблюдателей** (`watchers.txt`): долгоживущие процессы
  inotify/fswatch/watchdog — отличает «скрипт по расписанию» от «скрипт, слушающий
  событие». На карте это разные роли.
- **Новый inventory-документ `automations.md`** — сводная витрина всех автоматизаций
  независимо от механизма запуска (cron + systemd-timer + watcher + webhook + ручной).
  Колонки `name | trigger | schedule | runs | touches | log | status`. Колонка
  `touches` — источник связей для карты. Тип `automations` добавлен в шаблон
  `inventory-doc-template.md`.
- **Новая mermaid-диаграмма `automations.mmd`** (5-я): триггеры → автоматизации →
  что трогают (БД/сервисы/внешние API). Новый `classDef automation` (фиолетовый
  `#ede9fe/#7c3aed` — не конфликтует с app/db/mon/vpn/infra). Генерируется только
  при наличии автоматизаций (по образцу `vpn-architecture.mmd`).
- **Группа `automations` в `topology.mmd`** — факт наличия + ключевые связи на
  высоком уровне, без детализации триггеров (детали — в `automations.mmd`).
  Появляется только при непустом `automations.md`.

### Изменено
- **Правило синхронности §3.2 персоны** распространено на автоматизации: новая
  cron-задача / systemd-timer / скрипт-наблюдатель / pipeline или изменение того,
  что автоматизация трогает → обновить `automations.mmd` + группу в `topology.mmd`,
  иначе задача не закрыта.
- `inventory-scan/SKILL.md`: description, шаги 2/3/4/4.5/7, Verify-порог snapshot
  (≥14 → ≥16 файлов), 8 → 9 inventory-документов, 4 → 5 диаграмм.
- README + `diagrams/README.md`: состав диаграмм, условие генерации, цветовая семантика.

## [1.1.1] — 2026-05-24

Гибкость менеджеров паролей. По итогам диагностики пользователя на Windows: он
использует Kaspersky Password Manager, которого не было в закрытом enum схемы

Та же болезнь жёсткого списка, что лечили в 1.1.0 (хардкод вместо гибкости).

- Поля `secrets.manager_name` (имя нестандартного менеджера) и `secrets.cli_available`
  (умеет ли Claude доставать секреты автоматически).
- В `/sysadmin-init`: для `other` агент проводит ресёрч (WebSearch) — есть ли у менеджера
  CLI; честно объясняет расклад (нет CLI → пароли передаёшь руками) и предлагает
  альтернативу Bitwarden. Выбор за оператором. Запрет выдумывать наличие CLI.
- Агент учитывает `cli_available`: при `false` НЕ пытается доставать секреты сам, только
  указывает оператору, где взять руками (cold-start + setup-secrets-vault).

### Изменено
- `validate-config.sh`: jq-fallback принимает 6 менеджеров, требует `manager_name` при `other`.
  WARN про отсутствие check-jsonschema переформулирован в непугающий («необязательно»,
  не «ошибка») — jq-fallback достаточен, check-jsonschema остаётся опциональной надстройкой.

## [1.1.0] — 2026-05-24

Кроссплатформенность установки и принцип «не смог — скажи прямо». По итогам инцидента
с пользователем на Windows: настройка прошла криво, остался мёртвый `infra.md` вместо
папки `infra/` с `sysadmin-config.json` (причина — отсутствие `jq`, который не входит
в Git for Windows, и тихая импровизация агента вместо честной остановки).

### Добавлено
- **Адаптивный гейт окружения** (`_lib/ensure-local-env.sh`): динамический детект
  пакетного менеджера (brew/apt/dnf/yum/zypper/pacman/apk/nix/winget/choco/scoop/port)
  пробегом по PATH вместо привязки «ОС → менеджер». Авто-установка `jq` при отсутствии.
- **Универсальный fallback**: если пакетного менеджера нет — скачивание официального
  бинарника `jq` (github.com/jqlang/jq) под ОС+архитектуру. Работает на любой ОС без менеджера.
- **Финальная самопроверка** (`_lib/self-test-setup.sh`): после настройки агент тестирует,
  что всё реально работает (bash+jq, конфиг валиден, папка `infra/`, bridge-файл). При
  провале — честный вердикт «настройка не завершена, свяжись с разработчиком агента».
- **Конституционное правило C.9** в персоне: запрет на суррогаты конфига/инвентаря
  (никаких самодельных `infra.md`); «не смог → честный вердикт, не видимость работы».

### Исправлено
- Хардкод `/tmp/` заменён на `mktemp -d` (портируемо на Git Bash/WSL/macOS/Linux).
- FINAL CHECK после записи конфига: «Готово» печатается только если файл реально на
  диске и валиден — конец тихим «всё ок» при провалившейся записи.
- `find-config.sh`: `locate_sysadmin_root` понимает Windows-пути в bridge-файле
  (`C:\...`, `C:/...`, `/c/...`) + `cygpath -u`; единый резолвер без дублей grep-regex.
- Авто-добор `~/.sysadmin/bin` в PATH при каждом `source` helper'а — скачанный `jq`
  виден и в новом bash-процессе, и после перезапуска сессии.

### Изменено
- ADR-0002: hard cap ядра персоны поднят 400 → 410 (ради C.9).
- README + INSTALL: требования по окружению (bash обязателен, jq ставится автоматически),
  Windows-специфика. INSTALL клонирует последний тег динамически, без хардкода номера.
