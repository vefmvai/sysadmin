## [1.6.0] — 2026-05-25

HWID-locked подписки — `/configure-vpn-routing` научился доставать сервера
из подписок с привязкой к устройству (Panterra VPN, NurVPN, Happ-обёртки).
Поводом стал прогон ученика: агент получил заглушку от Panterra и трижды
ответил «нет способа обойти эту шифровку», хотя путь существовал — знание про
HWID-заголовки лежало в knowledge/памяти, но не было подключено к коду. Теперь
заглушка диагностируется, агент спрашивает про свободный слот и доводит дело
до списка серверов. Состав скиллов без изменений (19) — добавлен скрипт, не скилл.

### Добавлено
- **`configure-vpn-routing/scripts/resolve-hwid.sh`** — добыча HWID двумя путями
  (ADR-0010): путь A — из plist установленного Happ (слот не тратим), путь B —
  стабильный сгенерированный HWID в `infra/.../.hwid-<provider>` (переиспользуется
  между перезапусками, новые слоты не жжёт). `MODE=auto|happ|generate`.
- **ADR-0010** (`decisions/0010-hwid-locked-subscriptions.md`) — решение по
  HWID-замку: Happ основной / v2rayN фолбэк, диагностика заглушки (exit 3),
  явный вопрос про свободный слот, стабильный HWID. Перекликается с ADR-0007.
- `references/subscription-formats.md` — «Формат 5: HWID-locked» (Panterra/NurVPN/
  Happ), таблица провайдеров и дебаг-шаги обновлены.

### Изменено
- **`configure-vpn-routing/scripts/parse-subscription.sh`** — Happ-профиль
  заголовков (`User-Agent: Happ/4.9.0` + `X-Hwid` + `X-Device-*`) при заданном
  `HWID`; распознавание заглушки (`0.0.0.0` / «App not supported» / нулевые UUID)
  → **exit 3** с инструкцией вместо глухого «не найдено ссылок»; авто-ретрай
  Happ-профилем при заглушке от v2rayN с заданным HWID; нулевые UUID отсекаются
  при извлечении ссылок.
- **`configure-vpn-routing/SKILL.md`** — Шаг 5A.1 дополнен блоком «🔒 HWID-замок»
  (сеньор-диалог: объяснить замок → спросить свободный слот → добыть HWID →
  повторить); параметры `HWID`/`HWID_MODE`; граничный случай HWID-locked; ссылки
  на resolve-hwid.sh / ADR-0010 / subscription-mirroring.md.
- README — реестр ADR 9 → 10 решений.

## [1.5.0] — 2026-05-25

Скилл-дирижёр `/finalize-vpn-routing` — «доктор» для застрявшей VPN-настройки.
Для пользователя, у которого настройка маршрутизации 3X-UI зависла на полпути и
непонятно, что не так. Снимает READ-ONLY диагноз VPN-стека, показывает чеклист
«что есть / где дыры» простым языком, и доводит до идеала, дирижируя существующими
скиллами (`/configure-vpn-routing`, `/setup-server-proxy`) — НЕ дублирует их логику
и НЕ настраивает с нуля. Состав скиллов 18 → 19.

### Добавлено
- **Скилл `finalize-vpn-routing`** (`SKILL.md` + `scripts/diagnose-vpn-stack.sh`).
  Диагност+дирижёр: вход VLESS / выход upstream+балансир+observatory / гибкая
  маршрутизация «золотая середина» / локальный прокси на 127.0.0.1 — сверка с
  эталоном, чеклист ✅/❌, починка дыр прогоном подчинённых скиллов, финальный
  вердикт «доведено до идеала» + smoke-инструкция. Green-диагностика, Yellow-починка.
- `diagnose-vpn-stack.sh` — read-only снимок VPN-стека через lib-api
  (`api_get_xray_config`/`api_list_inbounds`), сверка с эталоном, JSON + сводка ✅/❌.

### Изменено
- **`setup-server-proxy/02-create-mixed-inbound.sh`** — remark mixed-inbound стал
  понятным новичку: «🔌 локальный прокси для программ на сервере (SOCKS5+HTTP)»
  (видно в панели). Закрывает запрос «пометка, что это локальный прокси».
- Число скиллов 18 → 19 в документации: `CLAUDE.md`, `README.md`,
  `agents/sysadmin.md`, `sysadmin-meet` (SKILL.md + parts/*); в `parts/03-skills.md`

## [1.4.3] — 2026-05-25

Фикс засорения снимка `/inventory-scan` stderr'ом ssh-клиента. Прогон у пользователя на
Windows/OpenSSH 9.x (отчёт 2026-05-25): в файлы снимка попадал транспортный шум ssh
(`mux_client_request_session`, post-quantum warning, `ControlSocket already exists`),
ломая идемпотентность — ручная чистка не помогала, следующий скан тащил мусор заново.

### Исправлено
- **stderr ssh-клиента в файлах снимка** (`dump-snapshot.sh`). Старое
  `run_cmd "$cmd" 2>&1 | redact_stream` сливало stderr удалённой команды и stderr
  локального ssh в один поток. Лечение двумя рычагами:
  - `run_cmd` вызывает `ssh -o LogLevel=ERROR -o ControlMaster=no` — глушит
    WARNING/INFO ssh-клиента у источника, пропуская реальные ERROR/FATAL.
  - `run_remote` разделяет потоки: stdout (данные) → файл снимка, stderr (диагностика
    удалённой команды) → `<label>.stderr.log` рядом, только если непустой. Оба потока
    проходят redaction — секрет в stderr тоже маскируется (приоритет №1).

### Добавлено
- **ADR-0009** (`decisions/0009-snapshot-stderr-separation.md`) — фиксирует разделение
  потоков и заглушение ssh-шума.
- Квирк в `dump-snapshot-quirks.md` — симптом, причина, лечение; класс файлов
  `*.stderr.log` как нормальный продукт скана.

### Изменено
- README: список ADR дополнен (0009), число решений 8 → 9.

## [1.4.2] — 2026-05-25

Фикс резолва относительного `infrastructure.root_path`. Прогон у пользователя на
Windows/Cygwin вскрыл ложный провал самопроверки `/sysadmin-init` (отчёт 2026-05-25):
агент рабочий, папка инфры на месте, но `self-test-setup.sh` печатал «Папка
инфраструктуры не создана» на любом относительном пути типа `../infra`.

### Исправлено
- **Ложный провал самопроверки на относительном `root_path`** (`self-test-setup.sh`).
  Причина: проверка `[ -d "../infra" ]` резолвилась от cwd процесса (у пользователя —
  `/c/Users/ARDOR`), а не от каталога конфига. Папка лежала рядом с конфигом и реально
  существовала, но проверка её не видела. Теперь относительный путь резолвится от
  **каталога конфига** (канон ADR-0008). Негативный кейс сохранён: если папки реально
  нет — проверка честно проваливается (`rc=1`).

### Добавлено
- **`resolve_infra_path`** — единый резолвер `root_path` в `_lib/find-config.sh`:
  относительный путь от каталога конфига, tilde → `$HOME`, абсолютный (вкл. Windows
  `C:\`/`C:/`) как есть. Все потребители используют его, чтобы правило жило в одном месте.
- **ADR-0008** (`decisions/0008-infra-path-resolution.md`) — фиксирует канон резолва и
  почему база именно «каталог конфига» (`/sysadmin-init` кладёт конфиг внутрь папки инфры).

### Изменено
- **Cold Start** (`cold-start.md`, Шаг 2) приведён к тому же канону — раньше резолвил
  относительный путь «от cwd на момент чтения», что могло разойтись с самопроверкой
  (зелёная галочка, за которой агент в работе не находит инфру). Теперь оба места
  резолвят одинаково.
- **Валидация визарда** (`wizard-flow.md`) — добавлен комментарий-предупреждение: резолв
  от cwd скилла согласован с каноном (`../infra/../infra == ../infra`), «упрощать» до
  `$PWD` нельзя.
- README: список ADR в дереве проекта дополнен (0007, 0008), число решений 6 → 8.

## [1.4.0] — 2026-05-24

UX VPN-маршрутизации для новичка: предсказуемый выход + стабильный IP. Скилл
`/configure-vpn-routing` научился вести непрофессионала за руку при настройке
outbound из подписки — извлекать сервера, сохранять их в `infra/`, размечать по
странам, спрашивать «с какого IP выходить» и собирать выход только из серверов
выбранной страны. Решает боль «прыжки по странам → бан аккаунта»: failover теперь
происходит в пределах одной страны. Зафиксирован факт из исходника 3X-UI: sticky-порог
переключения в миллисекундах в панели отсутствует (это клиентская настройка sing-box),
стабильность IP на сервере достигается «одна страна + редкий probeInterval».

### Добавлено
- **`scripts/save-subscription-servers.sh`** — сохраняет извлечённые из подписки
  сервера в `$INFRA/inventory/shared/vpn-subscriptions/<provider>.json` и размечает
  каждый по стране (эмодзи-флаг или текст в теге → ISO-код; неопознанный → `?`,
  страну НЕ выдумывает).
- **Шаг 5A (SKILL.md)** — диалог выбора страны выхода + выбор пресета:
  `country-failover` (несколько серверов одной страны + балансир, рекомендуемый) или
  `single` (один фиксированный сервер, абсолютно неизменный IP).
- **Параметры** `PROVIDER_SLUG`, `EXIT_COUNTRY`, `OUTBOUND_PRESET`, `PROBE_INTERVAL`.
- Триггеры: «вот моя подписка настрой выход», «хочу выходить из США», «чтобы IP
  не скакал», «настрой балансировщик» (configure-vpn-routing); «дай мне подключиться
  к своему серверу», «настрой Happ на мой сервер», «конфиг для Happ» (generate-client-config).

### Изменено
- **`setup-routing.sh`** — `PROBE_INTERVAL` по умолчанию `5m` (было `30s`): реже
  переоценка observatory → клиент дольше держится за один сервер → стабильнее IP.
  Добавлен `PROBE_SAMPLING` (для leastLoad). Комментарии-якоря про failover vs sticky.
- **`3x-ui-panel.md` §1.3–1.4** — зафиксировано (из исходника `BalancersTab.tsx`,
  проверено 2026-05-24): форма балансира 3X-UI = 4 поля, sticky-порога в мс нет;
  стабильный IP = одна страна + `probeInterval: 5m`; рекомендация 5m вместо 1m/30s.
- **Failed attempts** скилла — два новых граблекейса: «поиск sticky-порога в панели»
  (его там нет) и «заведение всех серверов подписки скопом» (= скачущий IP = бан).

## [1.2.0] — 2026-05-24

Автоматизации на карте инфраструктуры. По итогам реструктуризации Модуля 6 курса
«Архитектор ИИ-решений»: вместо «панели кнопок» ученик опирается на актуальную
Mermaid-карту от агента как на единый источник правды о том, что крутится само.
Значит, карта обязана отражать автоматизации — раньше они жили только в тексте.

### Добавлено
- **Сбор systemd-таймеров** (`dump-snapshot.sh` → `systemd-timers.txt`): на Ubuntu 24.04
  таймеры — основной способ расписания наравне с cron. Раньше собирались только
  enabled `*.service`, таймеры были полностью невидимы. `set +e` + honest-unknown
  при отсутствии systemctl.
- **Сбор скриптов-наблюдателей** (`watchers.txt`): долгоживущие процессы
  inotify/fswatch/watchdog — отличает «скрипт по расписанию» от «скрипт, слушающий
  событие». На карте это разные роли.
- **Новый inventory-документ `automations.md`** — сводная витрина всех автоматизаций
  независимо от механизма запуска (cron + systemd-timer + watcher + webhook + ручной).
  Колонки `name | trigger | schedule | runs | touches | log | status`. Колонка
  `touches` — источник связей для карты. Тип `automations` добавлен в шаблон
  `inventory-doc-template.md`.
- **Новая mermaid-диаграмма `automations.mmd`** (5-я): триггеры → автоматизации →
  что трогают (БД/сервисы/внешние API). Новый `classDef automation` (фиолетовый
  `#ede9fe/#7c3aed` — не конфликтует с app/db/mon/vpn/infra). Генерируется только
  при наличии автоматизаций (по образцу `vpn-architecture.mmd`).
- **Группа `automations` в `topology.mmd`** — факт наличия + ключевые связи на
  высоком уровне, без детализации триггеров (детали — в `automations.mmd`).
  Появляется только при непустом `automations.md`.

### Изменено
- **Правило синхронности §3.2 персоны** распространено на автоматизации: новая
  cron-задача / systemd-timer / скрипт-наблюдатель / pipeline или изменение того,
  что автоматизация трогает → обновить `automations.mmd` + группу в `topology.mmd`,
  иначе задача не закрыта.
- `inventory-scan/SKILL.md`: description, шаги 2/3/4/4.5/7, Verify-порог snapshot
  (≥14 → ≥16 файлов), 8 → 9 inventory-документов, 4 → 5 диаграмм.
- README + `diagrams/README.md`: состав диаграмм, условие генерации, цветовая семантика.

  и `other` — для любого нестандартного менеджера (Kaspersky, Dashlane, NordPass, браузерный).
- Поля `secrets.manager_name` (имя нестандартного менеджера) и `secrets.cli_available`
  (умеет ли Claude доставать секреты автоматически).
- В `/sysadmin-init`: для `other` агент проводит ресёрч (WebSearch) — есть ли у менеджера
  CLI; честно объясняет расклад (нет CLI → пароли передаёшь руками) и предлагает
  альтернативу Bitwarden. Выбор за оператором. Запрет выдумывать наличие CLI.
- Агент учитывает `cli_available`: при `false` НЕ пытается доставать секреты сам, только
  указывает оператору, где взять руками (cold-start + setup-secrets-vault).

### Изменено
- `validate-config.sh`: jq-fallback принимает 6 менеджеров, требует `manager_name` при `other`.
  WARN про отсутствие check-jsonschema переформулирован в непугающий («необязательно»,
  не «ошибка») — jq-fallback достаточен, check-jsonschema остаётся опциональной надстройкой.

## [1.1.0] — 2026-05-24

Кроссплатформенность установки и принцип «не смог — скажи прямо». По итогам инцидента
с пользователем на Windows: настройка прошла криво, остался мёртвый `infra.md` вместо
папки `infra/` с `sysadmin-config.json` (причина — отсутствие `jq`, который не входит
в Git for Windows, и тихая импровизация агента вместо честной остановки).

### Добавлено
- **Адаптивный гейт окружения** (`_lib/ensure-local-env.sh`): динамический детект
  пакетного менеджера (brew/apt/dnf/yum/zypper/pacman/apk/nix/winget/choco/scoop/port)
  пробегом по PATH вместо привязки «ОС → менеджер». Авто-установка `jq` при отсутствии.
- **Универсальный fallback**: если пакетного менеджера нет — скачивание официального
  бинарника `jq` (github.com/jqlang/jq) под ОС+архитектуру. Работает на любой ОС без менеджера.
- **Финальная самопроверка** (`_lib/self-test-setup.sh`): после настройки агент тестирует,
  что всё реально работает (bash+jq, конфиг валиден, папка `infra/`, bridge-файл). При
  провале — честный вердикт «настройка не завершена, свяжись с разработчиком агента».
- **Конституционное правило C.9** в персоне: запрет на суррогаты конфига/инвентаря
  (никаких самодельных `infra.md`); «не смог → честный вердикт, не видимость работы».

### Исправлено
- Хардкод `/tmp/` заменён на `mktemp -d` (портируемо на Git Bash/WSL/macOS/Linux).
- FINAL CHECK после записи конфига: «Готово» печатается только если файл реально на
  диске и валиден — конец тихим «всё ок» при провалившейся записи.
- `find-config.sh`: `locate_sysadmin_root` понимает Windows-пути в bridge-файле
  (`C:\...`, `C:/...`, `/c/...`) + `cygpath -u`; единый резолвер без дублей grep-regex.
- Авто-добор `~/.sysadmin/bin` в PATH при каждом `source` helper'а — скачанный `jq`
  виден и в новом bash-процессе, и после перезапуска сессии.

### Изменено
- ADR-0002: hard cap ядра персоны поднят 400 → 410 (ради C.9).
- README + INSTALL: требования по окружению (bash обязателен, jq ставится автоматически),
  Windows-специфика. INSTALL клонирует последний тег динамически, без хардкода номера.
